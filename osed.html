1. WinDbg and x86 Architecture

1.1 x86 Register Set

The x86 architecture provides eight 32-bit general-purpose registers, a 32-bit instruction pointer (EIP), and a 32-bit flags register (EFLAGS). Each general-purpose register can be accessed as a full 32-bit value (EAX), a 16-bit lower half (AX), or as individual 8-bit high/low pairs (AH/AL). Understanding register roles is essential for following exploit payloads, ROP chains, and shellcode execution flow.

EAX typically holds return values from function calls and is the accumulator for arithmetic. ECX serves as a loop counter. EDX extends EAX for 64-bit multiplication and division results. ESI and EDI are source and destination pointers for string operations (MOVS, CMPS, SCAS). EBP anchors the current stack frame, and ESP always points to the top of the stack. EIP cannot be directly modified — it advances automatically as instructions execute, but overwriting a saved return address on the stack redirects it after a RET instruction.

1.2 x86 Memory Layout

A Windows process is laid out in a 4 GB virtual address space (with default 2 GB user-mode / 2 GB kernel split). The lower addresses hold the PE image sections (.text for executable code, .data for initialized globals, .bss for uninitialized globals, .rdata for read-only data and import tables). The heap grows upward from lower addresses and is used for dynamic allocations via HeapAlloc/malloc. The stack occupies high user-mode addresses and grows downward — each new PUSH decrements ESP by 4 bytes, and each POP increments it.

The Thread Environment Block (TEB) at fs:[0] holds the pointer to the SEH chain, the stack base and limit, and a pointer to the Process Environment Block (PEB). The PEB contains the image base address, the loader data structures (InMemoryOrderModuleList) used to walk loaded DLLs, and process parameters. Shellcode commonly traverses the PEB to find kernel32.dll's base address at runtime.

1.3 x86 Instruction Set

Exploit development requires fluency with a core set of x86 instructions. Data movement instructions (MOV, PUSH, POP, LEA, XCHG) transfer values between registers, memory, and the stack. Arithmetic instructions (ADD, SUB, INC, DEC, MUL, DIV) perform calculations and set EFLAGS accordingly. Logic instructions (AND, OR, XOR, NOT, SHL, SHR) are used for bitwise operations — XOR EAX, EAX is the standard idiom for zeroing a register without introducing null bytes.

Control flow instructions (JMP, Jcc, CALL, RET, LOOP) redirect execution. CALL pushes the return address (address of the next instruction) onto the stack and jumps to the target. RET pops the top of the stack into EIP, which is the fundamental mechanism exploits abuse — overwriting the saved return address on the stack causes RET to redirect execution to attacker-controlled code. NOP (0x90) is a single-byte instruction that does nothing and advances EIP by one, forming NOP sleds in exploits.

1.4 Calling Conventions

Calling conventions define how functions receive arguments, return values, and clean up the stack. In cdecl (the C default), arguments are pushed right-to-left and the caller cleans the stack after the call with an ADD ESP, N instruction. In stdcall (used by most Win32 APIs), arguments are also pushed right-to-left but the callee cleans the stack using RET N where N is the total argument byte count. fastcall passes the first two arguments in ECX and EDX with remaining arguments on the stack, and the callee cleans up.

The distinction between cdecl and stdcall matters for exploit development because it determines whether the return address is at a fixed offset from the buffer or shifts depending on argument count. When analyzing a vulnerable function in IDA or WinDbg, check the calling convention to correctly calculate buffer-to-EIP offsets.

1.5 WinDbg Setup and Configuration

WinDbg is Microsoft's primary user-mode and kernel-mode debugger. For OSED, it is the tool of choice for all exploit development tasks — crash analysis, breakpoint-based flow tracing, and memory inspection during payload development. Install WinDbg Preview from the Microsoft Store or download the classic version from the Windows SDK installer. The debugger must be configured with a symbol path so that it can resolve function names for system DLLs.

The Microsoft symbol server provides PDB files for all Windows system libraries. Configuring the symbol path with srv* syntax tells WinDbg to download symbols on demand and cache them locally. 1.7 Debugging a Running Process

When developing an exploit, the typical workflow involves attaching WinDbg to the target application, setting breakpoints on the vulnerable function or the point where user input is processed, sending the payload, and examining the resulting state. Breakpoints can be set on addresses, function names, or memory access patterns. Hardware breakpoints (ba) fire on read, write, or execute access to a specific address — useful for catching when a buffer is accessed without modifying the code.

After a crash, WinDbg displays the faulting instruction and register state. The !analyze -v command performs automated crash analysis, identifying the exception type, faulting address, and a preliminary stack trace. For access violations during exploit development, the key information is the value in EIP (to confirm control), the state of ESP (to find the shellcode landing zone), and whether the exception was a read or write violation.

2. Stack Buffer Overflows

Modern Windows binaries may be compiled with Data Execution Prevention (DEP), Address Space Layout Randomization (ASLR), and Control Flow Guard (CFG). DEP marks the stack and heap as non-executable, ASLR randomizes module and stack base addresses, and CFG validates indirect call targets at runtime. The course starts with applications that have these protections disabled, then progressively enables DEP (Module 8) and ASLR (Module 9) to teach bypass techniques.

2.1 Understanding the Stack Frame

Every function call creates a stack frame that holds the function's local variables, saved register state, and the return address. When a function is called with CALL, the processor pushes the address of the next instruction (the return address) onto the stack. The called function's prologue then pushes the caller's EBP (saved frame pointer) and sets EBP to the current ESP, establishing a reference point for accessing both parameters (at positive offsets from EBP) and local variables (at negative offsets).

A stack buffer overflow occurs when a function writes more data to a stack-allocated buffer than the buffer can hold. Because local variables sit between ESP and the saved EBP/return address, an overflow that writes past the buffer's boundary first corrupts adjacent local variables, then overwrites the saved EBP, and finally overwrites the return address. When the function executes its epilogue (LEAVE; RET), the corrupted return address is popped into EIP, redirecting execution to an attacker-controlled address.

[EBP + 0]Saved EBP (caller's frame pointer)3rd — overwritten

Fuzzing is the process of sending progressively larger or malformed inputs to the target application to trigger a crash. For network services, a simple fuzzer sends increasing-length strings over a TCP connection and records when the service stops responding. For file-format targets, the fuzzer generates files with oversized fields. After identifying the approximate crash length, reduce the payload to find the exact size that overwrites EIP.

2.5 Identifying Bad Characters

Certain byte values are transformed, truncated, or stripped by the target application during input processing. A null byte (0x00) terminates C strings and will truncate the payload. Carriage return (0x0D) and line feed (0x0A) signal end-of-line in text protocols. These characters cannot appear in the shellcode or anywhere in the exploit buffer after the last critical control point. The bad character analysis involves sending every byte value (0x01 through 0xFF, excluding 0x00 which is always bad) and comparing what the application writes to memory against what was sent.

2.6 Finding a Return Address

After confirming EIP control and identifying bad characters, the exploit needs a reliable return address. For a vanilla stack overflow without DEP, the standard technique uses a JMP ESP gadget — an instruction at a fixed address in a loaded module that jumps to whatever ESP points to. After the RET instruction pops the overwritten return address into EIP and executes the JMP ESP, execution continues at ESP which points directly into the attacker's buffer (the C's section in the verification step). The JMP ESP address must not contain any bad characters and must reside in a module without ASLR, SafeSEH, or rebase protections.

3. Exploiting SEH Overflows

3.1 Structured Exception Handling Internals

Windows x86 applications use Structured Exception Handling (SEH) to manage runtime errors such as access violations, divide-by-zero, and illegal instructions. Each thread maintains its own exception chain rooted at offset 0x00 in the Thread Environment Block (TEB), accessible via the FS segment register at fs:[0]. The chain is a singly linked list of _EXCEPTION_REGISTRATION_RECORD structures stored on the stack.

Each record consists of two DWORD fields: a pointer to the next record in the chain (nSEH) and a pointer to the exception handler function (SEH). The final record in the chain has its nSEH field set to 0xFFFFFFFF, indicating the end of the list. When an exception occurs, the OS dispatcher walks this chain from the first record forward, invoking each handler until one claims the exception or the default handler terminates the process.

3.2 SEH Chain and the Default Handler

When an exception fires, the OS invokes ntdll!KiUserExceptionDispatcher, which calls ntdll!RtlDispatchException. This function iterates through the SEH chain, calling each handler with the exception record, the registration record, the context, and the dispatcher context. A handler returns ExceptionContinueExecution (0) to resume execution at the faulting instruction, or ExceptionContinueSearch (1) to pass the exception to the next handler in the chain.

If no handler claims the exception, the default handler kernel32!UnhandledExceptionFilter is invoked. In a debugger-attached scenario this transfers control to the debugger. Without a debugger, the process terminates with the familiar crash dialog. The critical insight for exploitation is that the nSEH and SEH fields live on the stack — a buffer overflow that reaches beyond the saved return address will overwrite these fields, redirecting exception handling to attacker-controlled code.

3.4 Finding POP POP RET Gadgets

When the OS dispatches an SEH exception, it pushes several arguments onto the stack before calling the handler. The critical detail is that a pointer to the _EXCEPTION_REGISTRATION_RECORD (which contains the nSEH field the attacker controls) is the second argument on the stack — at ESP+8 at the point the handler is called. A POP POP RET gadget removes two DWORDs from the stack, then returns into the address at ESP+8, which is the nSEH field. This redirects execution to the four bytes at the nSEH location, where the attacker places a short jump over the SEH handler address and into the shellcode that follows.

The POP instructions can use any general-purpose register — POP EAX; POP EBX; RET, POP ECX; POP ECX; RET, or any combination. The gadget must come from a module that does not have SafeSEH enabled (or does not participate in the SafeSEH table). Mona automates this search and filters results by SafeSEH, ASLR, and DEP status.

3.5 Short Jump and nSEH Payload Layout

After the POP POP RET gadget executes, EIP lands at the four bytes of the nSEH field on the stack. These four bytes must contain a short jump instruction that leaps over the four-byte SEH handler address into the shellcode that immediately follows. The standard short jump for this purpose is \xEB\x06\x90\x90 — a two-byte JMP SHORT +6 followed by two NOPs for alignment. This jumps 6 bytes forward from the end of the JMP instruction: 2 bytes remaining in nSEH + 4 bytes of SEH = 6 bytes, landing directly at the start of the shellcode.

If space after the SEH handler is too small for the full shellcode, a reverse jump can be used instead. Place the shellcode before the nSEH field in the padding area, and use a negative short jump (\xEB\x80 for -128 bytes) or a near jump to reach it. The available jump range for a short jump is -128 to +127 bytes from the end of the instruction.

The EXP-301 course calls this technique "Island-Hopping" — using a chain of short jumps, conditional jumps, and near jumps to navigate through constrained or fragmented buffer space to reach the shellcode. When the overflow splits the payload across non-contiguous regions, island-hopping connects the gaps: a short JMP in nSEH leaps to a small island of controlled space, which contains another JMP (forward or backward) to the next island, eventually reaching a region large enough for the full shellcode. Conditional jumps (JZ, JNZ) can also be used by setting flags with a preceding CMP or TEST instruction.

3.6 SafeSEH and SEHOP Bypass

SafeSEH is a compile-time protection that embeds a table of valid exception handler addresses into the PE image. When an exception occurs, ntdll!RtlIsValidHandler checks whether the handler address exists in this table. If the handler points to an address inside a SafeSEH-enabled module but is not in the table, the OS refuses to call it and terminates the process. The bypass is to use a POP POP RET gadget from a module that was compiled without SafeSEH — such a module has no handler table, so the validation check is skipped entirely.

Structured Exception Handler Overwrite Protection (SEHOP) is a runtime mitigation that validates the integrity of the SEH chain before dispatching an exception. SEHOP walks the chain and verifies that the last record's handler points to ntdll!FinalExceptionHandler. If the chain is corrupted (as it would be after an overflow), SEHOP detects the break and terminates the process. SEHOP is enabled by default on Windows Server 2008+ and Windows 10+ but can be disabled on client SKUs. Bypassing SEHOP requires reconstructing a valid chain that passes the walk check, which is significantly harder than SafeSEH bypass.

5.1 The Space Restriction Problem

After overwriting EIP in a stack buffer overflow, the remaining space between the return address and the end of the buffer may be too small to hold a full reverse-shell payload. A typical Meterpreter or custom reverse shell requires 350–500 bytes, but the controllable region after EIP may only offer 30–80 bytes. This space restriction makes direct shellcode placement impossible.

Before giving up on a target, examine the entire process memory at the time of the crash. The full input may be stored in another location — an earlier field in the protocol, a heap allocation, or a separate buffer on the stack. The goal is to identify any region that holds attacker-controlled data and has enough room for the final payload.

5.2 Egg Concept and Staged Payloads

An egg is a unique 4-byte marker that is repeated twice (8 bytes total) and prepended to the second-stage shellcode. The egghunter — a small first-stage payload (~32 bytes) — scans the entire process address space looking for this 8-byte sequence. When it finds the marker, it jumps to the code immediately following it, executing the full shellcode.

The egg must be unique enough that it does not appear naturally in process memory. Common choices include w00t (0x74303077) or c0d3 (0x33643063). The tag is repeated twice because the egghunter itself contains one copy of the tag — if it only searched for a single instance, it could find itself instead of the actual payload.

5.3 NtAccessCheckAndAuditAlarm Egghunter

The most reliable egghunter technique for Windows uses the NtAccessCheckAndAuditAlarm system call (syscall number 0x02 on 32-bit Windows) to validate whether a memory address is readable without causing an access violation. The egghunter iterates through memory one page at a time (4096 bytes), calling this syscall for each page. If the syscall returns STATUS_ACCESS_VIOLATION (0xC0000005), the entire page is skipped. If the page is accessible, the egghunter scans it byte by byte looking for the 8-byte egg marker.

This approach is more efficient and reliable than using SEH-based egghunters because it does not require registering a custom exception handler. The syscall returns an error code instead of crashing the process, allowing the egghunter to safely traverse unmapped and protected memory regions.

5.4 Custom Egghunter Assembly

The NtAccessCheckAndAuditAlarm egghunter compiles to approximately 32 bytes. Writing it in NASM and assembling it into raw bytes produces a compact first-stage payload that fits in almost any buffer space constraint. Every instruction must avoid null bytes since the egghunter is typically delivered through string-based input.

; egghunter.asm — NtAccessCheckAndAuditAlarm egghunter (32 bytes)
; Egg tag: w00t (0x74303077)

    or dx, 0x0fff          ; align to end of page
next_byte:

    push edx               ; address argument for syscall

    pop edx                ; restore address
    je next_page           ; skip page if inaccessible

    mov edi, edx           ; point EDI at current address
    scasd                  ; compare [EDI] with EAX, EDI += 4
    jnz next_byte          ; first tag not found, next byte
    scasd                  ; compare second tag
    jnz next_byte          ; second tag not found, next byte
    jmp edi                ; egg found — jump past both tags into shellcode

5.5 Egghunter in Stack Overflow Exploits

In a standard stack overflow exploit with limited post-EIP space, the egghunter is placed in the small buffer immediately after the return address. EIP is overwritten with a JMP ESP gadget address so execution lands on the egghunter. The second-stage payload (egg marker + shellcode) is placed in a different part of the input — for example, in an earlier field, a separate protocol command, or the beginning of the same buffer (before the EIP overwrite offset).

The exploit must ensure that the second-stage buffer is still in memory and accessible when the egghunter runs. If the application processes input in stages and discards earlier data, the egg may no longer be present. Testing the full exploit flow end-to-end in WinDbg is essential to confirm both the egghunter and the egg are in memory simultaneously.

5.6 Egghunter in SEH Exploits

SEH exploits offer even less space for shellcode because the layout requires the nSEH field (4 bytes) and SE handler (4 bytes) to be at specific offsets. After the POP POP RET gadget executes and control returns to nSEH, only the space between nSEH and the end of the overwritten buffer is available. A short jump in nSEH (EB 06 or similar) skips over the SE handler bytes and lands in this limited area — exactly where the egghunter goes.

The second-stage payload (egg + shellcode) is placed in an earlier part of the buffer or a different input field. The nSEH short jump lands on the egghunter, which scans memory until it finds the egg, then redirects execution to the full payload.

5.7 Optimizing Egghunter Speed

The default egghunter scans from virtual address 0x00000000, which means it must traverse all unmapped low-address space before reaching user-mode allocations. On systems with large address spaces or memory-intensive applications, this can take several seconds. Optimizing the starting address reduces scan time significantly.

If the approximate location of the egg is known (e.g., it is always placed on the stack or heap), the egghunter can be modified to start scanning from a nearby address. Another optimization is to align the scan to page boundaries and increase the step size within readable pages if the egg is known to be page-aligned.

Heap-biased startStart from a typical heap base (e.g., 0x00500000)Skips low-address kernel regions
Direction hintScan backward from a high address if the egg is in upper memoryVariable

6. Creating Custom Shellcode

6.1 Position-Independent Code

Shellcode must execute correctly regardless of where in memory it is loaded. This means it cannot contain absolute addresses — every memory reference must be relative to the current instruction pointer. The fundamental technique for obtaining the current EIP in x86 is the CALL/POP pattern: a CALL instruction pushes the address of the next instruction onto the stack, and a subsequent POP retrieves it into a register.

The JMP-CALL-POP technique is the standard idiom for position-independent data references. A JMP instruction at the start of the shellcode jumps forward past the data, landing on a CALL instruction that calls backward to the code right after the initial JMP. This CALL pushes the address of the data (which sits between the CALL target and the CALL itself) onto the stack, where a POP instruction retrieves it.

FSTENVfldz; fstenv [esp-0x0c]; pop regGet EIP via FPU environment save (alternative to CALL/POP)

6.2 Locating kernel32.dll Base Address

Every Windows process loads kernel32.dll, which exports essential functions like GetProcAddress, LoadLibraryA, and VirtualAlloc. To call these functions from shellcode, the base address of kernel32 must be resolved at runtime. The standard technique walks the Process Environment Block (PEB) structures maintained by the Windows loader.

On 32-bit Windows, the Thread Environment Block (TEB) is accessible at fs:[0x00], and the PEB pointer is at fs:[0x30]. The PEB contains a pointer to the PEB_LDR_DATA structure at offset 0x0C, which maintains three doubly-linked lists of loaded modules. The InMemoryOrderModuleList (at offset 0x14 within PEB_LDR_DATA) lists modules in the order they were loaded into memory. The first entry is typically the executable itself, the second is ntdll.dll, and the third is kernel32.dll.

6.3 Parsing the PE Export Table

Once the base address of kernel32.dll is known, specific exported functions are located by parsing the Portable Executable (PE) export table. The PE header starts at the base address with an IMAGE_DOS_HEADER, whose e_lfanew field (offset 0x3C) points to the IMAGE_NT_HEADERS. The OptionalHeader.DataDirectory[0] (the export directory) provides an RVA to the IMAGE_EXPORT_DIRECTORY structure.

The export directory contains three parallel arrays: AddressOfNames (array of RVAs to function name strings), AddressOfNameOrdinals (array of 16-bit ordinal values), and AddressOfFunctions (array of RVAs to function entry points). To resolve a function by name: iterate through AddressOfNames comparing each string, note the index when a match is found, use that index into AddressOfNameOrdinals to get the ordinal, and use the ordinal as an index into AddressOfFunctions to get the function RVA.

6.4 Hash-Based API Resolution

Embedding full function name strings in shellcode wastes space and makes the code larger. Hash-based resolution computes a short numeric hash of each function name and compares it against pre-computed target hashes. The most common algorithm is ROR13 (rotate right by 13 bits), which produces a 4-byte hash for each function name. The shellcode only stores the 4-byte hash instead of the full string, saving significant space when resolving multiple functions.

The resolution strategy is to first resolve GetProcAddress and LoadLibraryA from kernel32.dll's export table using hash comparison. Once these two functions are available, every other function in any DLL can be resolved by loading the DLL with LoadLibraryA and retrieving function pointers with GetProcAddress.

6.6 Null-Byte Elimination

Many input functions (e.g., strcpy, sprintf) treat the null byte (0x00) as a string terminator and stop copying data. If shellcode contains null bytes, it will be truncated when placed in a buffer through these functions. The most common sources of null bytes are MOV instructions with small immediate values (e.g., MOV EAX, 0x00000001 encodes as B8 01 00 00 00), PUSH 0, and instructions that reference absolute addresses. Each of these can be replaced with equivalent instructions that avoid null bytes.

6.7 Shellcode Encoding and Decoding

When the shellcode must avoid specific bad characters beyond just null bytes (e.g., \x0a, \x0d, \x20), an encoder transforms the entire payload into bytes that pass through the input filter. A small decoder stub is prepended to the encoded shellcode. At runtime, the decoder reverses the transformation in place, then execution falls through to the decoded shellcode.

The simplest encoder is single-byte XOR: each shellcode byte is XORed with a chosen key. The decoder stub XORs each byte back with the same key. The XOR key itself must not be a bad character, and the XOR of any shellcode byte with the key must not produce a bad character either. If a single key cannot satisfy these constraints, a multi-byte key or a different encoding scheme (e.g., additive, substitution) is required.

7. Reverse Engineering for Bugs

7.1 Static Analysis Methodology

Static analysis is the process of examining a compiled binary without executing it. In IDA Pro the first step is to load the target executable or DLL and let the auto-analysis engine complete. IDA identifies functions, cross-references, strings, and imported API calls. The goal is to map the attack surface — every path where external input enters the program — and follow it toward memory-manipulation operations that lack bounds checking.

Attack surface identification starts with imports. Network-facing applications import recv, WSARecv, or ReadFile; file parsers import CreateFileA/ReadFile; command handlers import string-processing functions. Each import is a potential entry point for attacker-controlled data. Cross-referencing these imports reveals which internal functions consume external input and where processing occurs.

Synchronizing WinDbg and IDA Pro is essential for efficient reverse engineering. The target module's base address in WinDbg will differ from IDA's default image base. Use IDA's Edit → Segments → Rebase program to rebase the database to the actual load address from lm m [module] in WinDbg. Once rebased, every address in IDA matches WinDbg — breakpoints found during static analysis can be set directly. Hook the recv API with bp ws2_32!recv to intercept all incoming network data, then inspect the buffer on each break to trace exactly which function processes each field of the input.

7.2 Identifying Dangerous Function Calls

Certain C library functions are inherently unsafe because they perform no bounds checking on destination buffers. When attacker-controlled data reaches one of these functions, and the destination buffer is smaller than the input, a buffer overflow occurs. IDA makes it straightforward to search for these functions in the Imports window or by searching disassembly for call instructions referencing them.

The distinction between dangerous and safe alternatives matters for determining exploitability. Functions like strncpy, snprintf, and memcpy_s accept a length parameter — but only if the developer supplies the correct size. A common mistake is passing the source length instead of the destination buffer size, which defeats the bounds check entirely.

7.3 Tracing Input to Vulnerable Sinks

A vulnerability is only exploitable if attacker-controlled data can reach it. Input tracing starts at a source — typically recv or ReadFile — and follows the data through every function call, buffer copy, and transformation until it arrives at a dangerous sink. This is data-flow analysis. Each intermediate function must be examined to determine whether it truncates, filters, or transforms the input in ways that limit exploitation.

Data often passes through multiple layers before reaching a vulnerable function. A network server might call recv into a raw buffer, then parse the buffer into a protocol structure, then extract a field and pass it to a processing function that calls strcpy. Each layer is an opportunity for bounds checking — and each layer without a check is a link in the exploit chain.

7.4 Stack Frame Analysis in IDA

IDA reconstructs the stack frame of each function by analyzing sub esp, push, and mov [ebp-X] instructions. Double-clicking a function's stack frame (or pressing Ctrl+K) opens the Stack Variables view, which lists every local variable, its offset from EBP, and its size. The offset tells you how far each variable is from the saved return address — the critical distance for calculating overflow padding.

IDA names stack variables automatically. A variable at [ebp-208h] appears as var_208. The size of a buffer is inferred from the gap between consecutive variables: if var_208 is followed by var_108, the buffer is 0x208 - 0x108 = 0x100 (256) bytes. The saved return address is at [ebp+4], so the total overflow distance from var_208 to EIP is 0x208 + 4 = 524 bytes.

7.6 Protocol Analysis and Fuzzing

Many EXP-301 targets implement custom network protocols. Protocol analysis means capturing traffic (with Wireshark or by reading the binary's parsing code in IDA) and identifying the message format: header fields, length fields, command identifiers, and data payloads. Once the protocol structure is understood, a fuzzer can systematically send malformed messages that violate expected constraints — oversized fields, missing delimiters, negative lengths, and unexpected command codes.

Targeted fuzzing is more effective than blind fuzzing. Instead of randomly mutating bytes, the fuzzer should exercise specific code paths identified during static analysis. For each input field that reaches a dangerous function, the fuzzer generates inputs designed to overflow that specific buffer while keeping the rest of the protocol structure valid so the parser does not reject the message before it reaches the vulnerable code.

Some protocols include a checksum or hash field that the server validates before processing the payload. If the checksum is wrong, the server discards the packet and the overflow is never reached. During reverse engineering, identify the checksum algorithm in IDA — common implementations include CRC32, simple byte sums, XOR folding, or custom hash functions. Once identified, implement the same algorithm in the exploit script so that every crafted packet passes validation and reaches the vulnerable code path.

7.7 Vulnerability Classification

After identifying a crash, classify the vulnerability to determine the appropriate exploitation technique. A stack buffer overflow with EIP control is the most straightforward — it leads directly to Module 2 (basic overflow) or Module 8 (DEP bypass). An SEH overwrite requires the Module 3 technique. 8. Stack Overflows and DEP Bypass

8.1 Data Execution Prevention Theory

Data Execution Prevention (DEP) marks memory pages as either executable or writable, but not both simultaneously. The hardware implementation uses the NX (No eXecute) bit in the page table entry — when the CPU attempts to execute code on a page marked NX, it raises an access violation. This defeats traditional shellcode execution on the stack because the stack is mapped as read/write but not executable.

Windows implements DEP in four modes. OptIn (default on client OS) only protects processes that explicitly opt in. OptOut (default on server OS) protects all processes except those explicitly exempted. AlwaysOn and AlwaysOff are self-explanatory. Process-level DEP can also be set via the /NXCOMPAT linker flag, which sets a flag in the PE header requesting DEP protection regardless of the system policy.

Windows Defender Exploit Guard (WDEG) adds an additional enforcement layer on top of the system DEP policy. Through Exploit Protection settings, administrators can force DEP on specific processes even if they opt out, override system-wide policy, or enable additional mitigations like bottom-up ASLR and CFG. Check the current WDEG configuration for a target process with Get-ProcessMitigation -Name [process-name].exe in PowerShell, or view system-wide defaults with Get-ProcessMitigation -System. If WDEG enforces DEP on a process that would otherwise have it disabled, the exploit must include a ROP-based DEP bypass regardless of the system policy.

DEP ModeValueBehavior

OptOut3DEP enabled for all processes except explicitly opted out (server default)

8.2 Return-Oriented Programming Concepts

Return-Oriented Programming (ROP) bypasses DEP by chaining small sequences of existing executable instructions — called gadgets — that each end with a RET instruction. Instead of injecting new code, the exploit arranges the stack so that each RET pops the next gadget address, effectively building a program from fragments of legitimate code already in memory.

The stack controls execution entirely. Each gadget performs a small operation (load a register, add values, push a value) and then executes RET, which pops the next address from the stack and jumps to it. By carefully sequencing gadgets and their arguments on the stack, the chain can call a Windows API function like VirtualProtect to mark the stack as executable, then return to shellcode that now sits in executable memory.

8.4 Building a VirtualProtect ROP Chain

VirtualProtect changes the memory protection on existing pages. By calling VirtualProtect(lpAddress, dwSize, PAGE_EXECUTE_READWRITE, &lpflOldProtect) on the stack region containing shellcode, DEP is effectively disabled for that memory range. The shellcode can then execute normally. This is the most common DEP bypass because it does not require allocating new memory or writing to arbitrary addresses.

The ROP chain must place four arguments on the stack in the correct order, then transfer execution to VirtualProtect. The challenge is that several arguments are dynamic — lpAddress should point to the shellcode, and lpflOldProtect needs a writable address for the output parameter. The PUSHAD technique simplifies this by using register-loading gadgets followed by a single PUSHAD instruction that pushes all eight general-purpose registers onto the stack in the correct order for the API call.

8.5 Building a VirtualAlloc ROP Chain

VirtualAlloc allocates a new block of memory with the specified protection flags. By calling VirtualAlloc(lpAddress, dwSize, flAllocationType, flProtect) with flProtect=PAGE_EXECUTE_READWRITE, the chain creates a fresh RWX memory region. The chain is more complex than VirtualProtect because after allocation, the shellcode must be copied from the stack into the new memory. This requires an additional memcpy-equivalent ROP sequence or a REP MOVS gadget chain. Alternatively, combine VirtualAlloc with a return-to-shellcode by making lpAddress a predictable value and placing shellcode at a known stack offset.

9. Stack Overflows and ASLR Bypass

9.1 ASLR Theory and Implementation

Address Space Layout Randomization (ASLR) randomizes the base addresses of executables, DLLs, the stack, and the heap each time a process starts (or each boot on older Windows versions). On Windows, ASLR is enabled per-module via the /DYNAMICBASE linker flag and enforced by the kernel's memory manager. The entropy varies by region — image bases receive 8 bits of randomization (256 possible positions) in 32-bit processes, while the stack and heap receive higher entropy. On 64-bit systems, ASLR entropy increases dramatically to 17+ bits for images and 24+ bits for the stack.

Windows Vista and later enable ASLR by default for all system DLLs and executables compiled with /DYNAMICBASE. Older third-party modules or applications compiled without this flag retain fixed base addresses across reboots, which creates a critical weakness — any single non-ASLR module in the process address space provides a stable foothold for gadgets and return addresses.

Windows Defender Exploit Guard (WDEG) can enforce mandatory ASLR — forcing randomization on modules that were not compiled with /DYNAMICBASE. When enabled, WDEG overrides the module's PE header and applies ASLR regardless. Check the system-wide WDEG ASLR policy with Get-ProcessMitigation -System and per-process overrides with Get-ProcessMitigation -Name [process].exe. If mandatory ASLR is active, previously "non-ASLR" modules are no longer reliable gadget sources, and the exploit must use an information leak instead.

9.3 Partial EIP Overwrite

When ASLR randomizes a module's base address, the lower bytes of code addresses within that module remain relatively stable — only the higher-order bytes change. A partial EIP overwrite exploits this by overwriting only the least significant 1 or 2 bytes of the saved return address on the stack, leaving the upper bytes intact. This constrains the redirect to stay within the same module where the return address already points, avoiding the need to know the full randomized base address.

The technique requires precise control of the overflow length. Instead of overwriting all 4 bytes of the saved EIP, the attacker writes exactly 1 or 2 bytes, redirecting execution to a useful gadget whose low-order address bytes are known. The effectiveness depends on how many useful gadgets exist within the reachable address range (256 locations for a 1-byte overwrite, 65536 for 2 bytes).

9.7 ROP-Based Shellcode Encoder

When bad characters prevent placing raw shellcode in the exploit buffer, build a ROP chain that decodes the shellcode at runtime. Instead of sending the final shellcode directly, encode each DWORD of the shellcode into values that avoid all bad characters, then use ROP gadgets to decode and write them to an executable memory region one DWORD at a time. This technique is essential when the buffer filter removes bytes like 0x00, 0x0a, 0x0d, or 0x20 that appear in the shellcode.

The encoding strategy uses SUB or ADD complements. For each 4-byte DWORD of shellcode, find two or three values that, when subtracted from each other (or added), produce the target DWORD — and ensure none of the operand bytes are bad characters. The ROP chain then executes a sequence: zero a register, subtract the complement values, and write the result to the target address. By repeating this for every DWORD and incrementing the target pointer, the entire shellcode is reconstructed in clean memory.

3. SUB complement 1sub eax, [value1]; retFirst subtraction (negative accumulation)
4. SUB complement 2sub eax, [value2]; retSecond subtraction produces the target DWORD in EAX (via underflow)
5. Write DWORDmov [edi], eax; retWrite decoded DWORD to the target address

def encode_dword(target, bad_chars):
    """Find two values v1, v2 where (0 - v1 - v2) & 0xFFFFFFFF == target
    and no byte in v1 or v2 is a bad character."""
    neg = (0 - target) & 0xFFFFFFFF
    for v1_b0 in range(1, 256):
        if v1_b0 in bad_chars:
            continue
        for v1_b1 in range(1, 256):
            if v1_b1 in bad_chars:
                continue
            v1_hi = (v1_b1 << 8) | v1_b0

                v2 = (neg - v1) & 0xFFFFFFFF

    return None

10. Format String Specifier Attack Part I

10.1 Format String Vulnerability Theory

A format string vulnerability occurs when user-controlled input is passed directly as the format string argument to a printf-family function (printf, sprintf, fprintf, snprintf, wprintf). Normally, these functions expect the programmer to supply a fixed format string with specifiers (%x, %s, %d, %n, %p) and corresponding arguments on the stack. When the format string itself comes from the attacker, they control which specifiers are processed, and the function reads (or writes) stack memory that was never intended to be accessible.

The root cause is passing user input as the first argument: printf(user_input) instead of printf("%s", user_input). In the vulnerable case, each format specifier in the user's input causes printf to consume the next value from the stack as though it were a corresponding argument. Since no actual arguments exist beyond the format string pointer, the function reads whatever happens to be on the stack — saved registers, return addresses, local variables, and frame pointers.

10.4 Arbitrary Write with %n

The %n specifier writes the number of characters printed so far to the address pointed to by the corresponding stack argument. By controlling the address on the stack (via the format string offset) and the number of characters printed (via padding specifiers like %Nx), the attacker achieves an arbitrary write primitive — writing a controlled value to a controlled address. This is the most powerful format string primitive and enables full code execution.

A 4-byte write with %n requires printing exactly the desired value's worth of characters before the %n specifier. 11.1 Precise Value Writes with %hn

Writing an arbitrary 4-byte value to an address requires two %hn writes — one for the lower 2 bytes and one for the upper 2 bytes. The format string must contain both target addresses (the base address and base+2), and two %hn specifiers with calculated padding to produce the desired byte values. The writes must be ordered from smallest to largest value to ensure the padding calculations produce positive numbers.

The padding for each write is computed as (desired_value - chars_already_printed) mod 65536. Since the character count is cumulative across the entire format string, the second write's padding must account for all characters already output by the first write. Sorting the two half-word values from smallest to largest simplifies the calculation and avoids negative padding.

11.2 Overwriting the Global Offset Table

On Linux, dynamically linked functions are resolved through the Procedure Linkage Table (PLT) and Global Offset Table (GOT). The first time a function like printf is called, the PLT stub redirects to the dynamic linker, which resolves the function's real address and writes it into the GOT entry. Subsequent calls read the resolved address directly from the GOT. Overwriting a GOT entry with a format string replaces the resolved function address with an attacker-controlled address, redirecting execution whenever the function is next called.

The attacker finds the GOT address of a frequently called function (such as printf, puts, or exit) using objdump -R or readelf -r, then uses the %hn write primitive to overwrite the GOT entry with the address of shellcode, a system() call, or a stack pivot gadget. The overwrite takes effect immediately — the next call to the target function jumps to the attacker's address.

11.5 Full Format String Exploitation Chain

A complete format string exploit combines reconnaissance, leaking, and writing into a single automated flow. The exploit first identifies the format string offset, then leaks runtime addresses to defeat ASLR, calculates target addresses, performs the overwrite (GOT entry, SEH handler, or function pointer), and finally triggers the hijacked code path. On Windows, the chain typically overwrites the SEH handler or an IAT entry and triggers an exception or API call. On Linux, GOT overwrites followed by triggering the replaced function are standard.

Combining ROP with format strings under ASLR introduces additional constraints. A standard VirtualAlloc or VirtualProtect ROP chain assumes fixed gadget addresses, but ASLR randomizes the module base on each boot. The format string leak from Part I provides the base address, but the shellcode location on the stack is also unknown — it shifts with each run. The exploit must leak the module base via the read primitive, calculate all gadget addresses at runtime, determine where on the stack the shellcode will land (using a second leaked stack address), and build the format string write payload to place the ROP chain and shellcode pointer at the correct location. When the format string buffer is too small to hold both the ROP chain and shellcode, the exploit must store shellcode in a larger buffer elsewhere in the process — a global command buffer in the .data section, or a protocol input buffer that persists across requests. A prior protocol command places the shellcode there, and the ROP chain pivots execution to that known address after the format string write redirects EIP.
